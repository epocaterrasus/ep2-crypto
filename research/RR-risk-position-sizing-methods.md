KELLY CRITERION — DEEP DIVE\n\n### The Continuous Kelly Formula\n\nThe standard binary Kelly formula (`f* = (bp - q) / b`) is wrong for trading because it assumes binary outcomes. If you just give each signal 1/3 of the budget, you're ignoring that some signals are more volatile than others.\n\n### The Approach\n\n**Equal Risk Contribution (ERC):** Each signal/strategy contributes equally to total portfolio risk.\n\n```\nσ_portfolio² = Σ_i Σ_j w_i × w_j × σ_i × σ_j × ρ_ij\n```\n\nFor equal risk contribution:\n```\nw_i × (Σ_j w_j × σ_i × σ_j × ρ_ij) = w_k × (Σ_j w_j × σ_k × σ_j × ρ_kj)  for all i,k\n```\n\nIn practice, for uncorrelated signals (which you should aim for):\n```\nw_i ∝ 1 / σ_i\n```\n\nHigher volatility strategies get smaller allocations.\n\n### For Your Three Signal Types\n\nAssume:\n- **Microstructure signals**: High frequency, low per-trade variance, many trades\n- **Macro signals**: Lower frequency, higher per-trade variance, fewer trades\n- **Cascade signals**: Event-driven, high variance, rare\n\nThe risk budget allocation should inversely weight by signal variance:\n\n```python\nimport numpy as np\nfrom scipy.optimize import minimize\n\ndef risk_parity_weights(\n    strategy_volatilities: np.ndarray,\n    correlation_matrix: np.ndarray,\n    total_risk_budget: float = 1.0\n) -> np.ndarray:\n    \"\"\"\n    Compute risk parity weights for multiple strategies.\n\n    Args:\n        strategy_volatilities: Annualized volatility of each strategy's returns.\n        correlation_matrix: Correlation matrix between strategy returns.\n        total_risk_budget: Total risk budget (1.0 = 100%).\n\n    Returns:\n        Array of weights (risk allocations) for each strategy.\n    \"\"\"\n    n = len(strategy_volatilities)\n    cov_matrix = np.outer(strategy_volatilities, strategy_volatilities) * correlation_matrix\n\n    def risk_contribution_error(weights: np.ndarray) -> float:\n        port_var = weights @ cov_matrix @ weights\n        if port_var <= 0:\n            return 1e10\n\n        # Initial guess: inverse volatility\n    w0 = (1.0 / strategy_volatilities)\n    w0 = w0 / np.sum(w0) * total_risk_budget\n\n    constraints = [\n        {\"type\": \"eq\", \"fun\": lambda w: np.sum(w) - total_risk_budget}\n    ]\n    bounds = [(0.01, total_risk_budget)] * n\n\n    result = minimize(\n        risk_contribution_error,\n        w0,\n        method=\"SLSQP\",\n        bounds=bounds,\n        constraints=constraints,\n    )\n\n    return result.x\n\ndef allocate_risk_budget(\n    total_equity: float,\n    total_risk_fraction: float,\n    strategy_vols: dict[str, float],\n    correlations: dict[tuple[str, str], float] | None = None,\n) -> dict[str, float]:\n    \"\"\"\n    Allocate risk budget across strategies using risk parity.\n\n    Args:\n        total_equity: Account equity.\n        total_risk_fraction: Total risk fraction per trade cycle\n            (e.g., 0.005 = 0.5%).\n        strategy_vols: Dict of strategy name -> recent volatility.\n        correlations: Optional pairwise correlations.\n\n    Returns:\n        Dict of strategy name -> dollar risk budget.\n    \"\"\"\n    names = list(strategy_vols.keys())\n    n = len(names)\n    vols = np.array([strategy_vols[name] for name in names])\n\n    if correlations is None:\n        # Assume uncorrelated — simplifies to inverse-vol weighting\n        corr_matrix = np.eye(n)\n    else:\n        corr_matrix = np.eye(n)\n        for (s1, s2), rho in correlations.items():\n            i, j = names.index(s1), names.index(s2)\n            corr_matrix[i, j] = rho\n            corr_matrix[j, i] = rho\n\n    weights = risk_parity_weights(vols, corr_matrix)\n    total_risk_dollars = total_equity * total_risk_fraction\n\n    return {\n        name: weights[i] * total_risk_dollars\n        for i, name in enumerate(names)\n    }\n```\n\n### Practical Example\n\n```python\n# Three strategies with different volatility profiles\nstrategy_vols = {\n    \"microstructure\": 0.15,  # Lower vol, more consistent\n    \"macro_momentum\": 0.35,  # Higher vol, bigger swings\n    \"cascade_detector\": 0.50, # Highest vol, event-driven\n}\n\n# Result with $100k equity, 0.5% total risk:\n# microstructure: ~$260 risk budget (52% of total)\n# macro_momentum: ~$150 risk budget (30% of total)\n# cascade_detector: ~$90 risk budget (18% of total)\n```\n\n---\n\n## 6. ANTI-MARTINGALE VS FIXED\n\n### Anti-Martingale (Reverse Martingale)\n\nIncrease position size after wins, decrease after losses. The logic: winning streaks indicate the system is \"in sync\" with the market.\n\n**Fixed Ratio Method (Ryan Jones):**\n```\nNew Level = Previous Level + Delta\n```\nWhere delta is a fixed dollar amount of profit required before increasing size by 1 unit.\n\n**Percentage-based anti-martingale:**\n```\nIf last N trades profitable: size = base_size × (1 + increment)\nIf last N trades losing: size = base_size × (1 - decrement)\n```\n\n### Does It Work for Momentum Crypto?\n\n**Theoretically yes, with caveats.** The argument for anti-martingale in momentum systems:\n\n1. **Serial correlation in returns:** If your momentum signals have autocorrelated returns (which they should, by definition), then a winning streak increases the posterior probability that the next trade will also win.\n\n2. **Regime alignment:** Winning streaks suggest the system is aligned with the current market regime.\n\n3. **Capital preservation:** Reducing size during drawdowns ensures you survive to trade another day.\n\n**The evidence is mixed:**\n\n- Academic studies (Vince, Tharp) show anti-martingale outperforms fixed sizing when returns have positive autocorrelation.\n- For IID (independent) returns, anti-martingale performs identically to fixed sizing in expectation.\n- For mean-reverting returns (negative autocorrelation), anti-martingale is WORSE than fixed.\n\n**For 5-minute BTC momentum:** Returns likely have short-term positive autocorrelation (momentum effects last minutes to hours), so mild anti-martingale may help. But the autocorrelation is weak and noisy.\n\n### Implementation\n\n```python\nimport numpy as np\nfrom collections import deque\n\nclass AntiMartingalePosition:\n    \"\"\"\n    Anti-martingale position sizing with configurable parameters.\n    \"\"\"\n\n    def __init__(\n        self,\n        base_risk_fraction: float = 0.0025,\n        lookback: int = 20,\n        win_increment: float = 0.3,\n        loss_decrement: float = 0.5,\n        max_multiplier: float = 2.0,\n        min_multiplier: float = 0.25,\n        win_threshold: float = 0.6,\n    ):\n        \"\"\"\n        Args:\n            base_risk_fraction: Base risk per trade (e.g., 0.25%).\n            lookback: Number of recent trades to evaluate.\n            win_increment: Fraction to increase when on a streak (0.3 = +30%).\n            loss_decrement: Fraction to decrease when losing (0.5 = -50%).\n            max_multiplier: Maximum sizing multiplier.\n            min_multiplier: Minimum sizing multiplier.\n            win_threshold: Win rate threshold to trigger increase.\n        \"\"\"\n        self.base_risk = base_risk_fraction\n        self.lookback = lookback\n        self.win_increment = win_increment\n        self.loss_decrement = loss_decrement\n        self.max_mult = max_multiplier\n        self.min_mult = min_multiplier\n        self.win_threshold = win_threshold\n        self.recent_trades: deque[float] = deque(maxlen=lookback)\n        self.current_multiplier = 1.0\n\n    def update(self, trade_return: float) -> None:\n        \"\"\"Record a completed trade.\"\"\"\n        self.recent_trades.append(trade_return)\n        self._recalculate_multiplier()\n\n    def _recalculate_multiplier(self) -> None:\n        if len(self.recent_trades) < self.lookback // 2:\n            self.current_multiplier = self.min_mult  # Conservative until enough data\n            return\n\n        wins = sum(1 for t in self.recent_trades if t > 0)\n        recent_win_rate = wins / len(self.recent_trades)\n\n        if recent_win_rate > self.win_threshold:\n            # Scale up proportionally to how far above threshold\n            excess = recent_win_rate - self.win_threshold\n            self.current_multiplier = min(\n                1.0 + excess * self.win_increment / 0.1,\n                self.max_mult,\n            )\n        elif recent_win_rate < (1 - self.win_threshold):\n            # Scale down when losing\n            deficit = (1 - self.win_threshold) - recent_win_rate\n            self.current_multiplier = max(\n                1.0 - deficit * self.loss_decrement / 0.1,\n                self.min_mult,\n            )\n        else:\n            # In neutral zone, drift toward 1.0\n            self.current_multiplier = 0.9 * self.current_multiplier + 0.1 * 1.0\n\n    def get_risk_fraction(self) -> float:\n        return self.base_risk * self.current_multiplier\n```\n\n### Recommendation\n\n**Use mild anti-martingale** — reduce more aggressively during drawdowns than you increase during win streaks. The asymmetry is key:\n\n- After 5+ consecutive wins: increase by 10-20%\n- After 5+ consecutive losses: decrease by 30-50%\n\nThis captures some regime-alignment benefit while prioritizing capital preservation.\n\n---\n\n## 7. MAXIMUM DRAWDOWN-CONSTRAINED SIZING\n\n### The Grossman-Zhou Framework\n\nGrossman and Zhou (1993) solved the optimal portfolio problem subject to a maximum drawdown constraint. The key result:\n\n**For a risky asset with Sharpe ratio `S` and a max drawdown constraint `D`:**\n\n```\nf* = (S² / (2γ)) × (1 / (1 + λ × (W/M)^(-2γ/S²)))\n```\n\nWhere:\n- `W` = current wealth\n- `M` = maximum wealth achieved so far (high water mark)\n- `γ` = risk aversion parameter calibrated to the drawdown constraint\n- `λ` = Lagrange multiplier for the drawdown constraint\n\n**Simplified practical version:**\n\nThe fraction of capital to invest is approximately:\n\n```\nf* ≈ min(f_kelly, D / (σ × z_α × √T))\n```\n\nWhere:\n- `D` = maximum acceptable drawdown (e.g., 0.15 for 15%)\n- `σ` = portfolio volatility\n- `z_α` = z-score for confidence level (2.33 for 99%)\n- `T` = time horizon in periods\n\n### For 15% Maximum Drawdown\n\n```python\nimport numpy as np\nfrom scipy.stats import norm\n\ndef max_drawdown_constrained_size(\n    equity: float,\n    max_drawdown: float,\n    trade_volatility: float,\n    confidence_level: float = 0.99,\n    trades_per_day: int = 288,\n    horizon_days: int = 30,\n    kelly_fraction: float | None = None,\n) -> dict:\n    \"\"\"\n    Position size constrained by maximum acceptable drawdown.\n\n    Args:\n        equity: Current account equity.\n        max_drawdown: Maximum acceptable drawdown as fraction (0.15 = 15%).\n        trade_volatility: Standard deviation of per-trade returns.\n        confidence_level: Confidence level for drawdown bound (0.99 = 99%).\n        trades_per_day: Expected number of trades per day.\n        horizon_days: Planning horizon in days.\n        kelly_fraction: Optional Kelly fraction (used as upper bound).\n\n    Returns:\n        Constrained position sizing parameters.\n    \"\"\"\n    total_trades = trades_per_day * horizon_days\n    z_alpha = norm.ppf(confidence_level)\n\n    # Expected maximum drawdown formula (approximate):\n    # E[MDD] ≈ σ × √(2 × ln(N)) for N trades (from Brownian bridge theory)\n    # For a confidence bound:\n    # MDD_bound ≈ f × σ × z_α × √(T)\n    # where T is related to the expected drawdown duration\n\n    # Solving for f:\n    # f = max_drawdown / (σ × z_α × √(2 × ln(N)))\n    drawdown_factor = trade_volatility * z_alpha * np.sqrt(2 * np.log(total_trades))\n\n    if drawdown_factor > 0:\n        dd_constrained_f = max_drawdown / drawdown_factor\n    else:\n        dd_constrained_f = 0.01\n\n    # Apply Kelly as upper bound if provided\n    if kelly_fraction is not None:\n        final_f = min(dd_constrained_f, kelly_fraction)\n    else:\n        final_f = dd_constrained_f\n\n    # Ensure minimum viable size\n    final_f = max(final_f, 0.0001)\n\n    return {\n        \"risk_fraction\": final_f,\n        \"dollar_risk_per_trade\": equity * final_f,\n        \"max_drawdown_target\": max_drawdown,\n        \"confidence_level\": confidence_level,\n        \"dd_constrained_f\": dd_constrained_f,\n        \"kelly_fraction\": kelly_fraction,\n        \"binding_constraint\": \"drawdown\" if dd_constrained_f < (kelly_fraction or float('inf')) else \"kelly\",\n    }\n\ndef dynamic_drawdown_sizing(\n    equity: float,\n    high_water_mark: float,\n    max_drawdown: float,\n    base_risk_fraction: float,\n) -> float:\n    \"\"\"\n    Dynamically reduce position size as drawdown approaches limit.\n\n    Uses linear scaling: as you approach max DD, size approaches zero.\n\n    Args:\n        equity: Current equity.\n        high_water_mark: Peak equity achieved.\n        max_drawdown: Maximum acceptable drawdown fraction.\n        base_risk_fraction: Risk fraction at no drawdown.\n\n    Returns:\n        Adjusted risk fraction.\n    \"\"\"\n    current_dd = 1.0 - (equity / high_water_mark)\n\n    if current_dd <= 0:\n        return base_risk_fraction\n\n    # Linear scaling: full size at 0% DD, zero size at max DD\n    # With a buffer — start reducing at 50% of max DD\n    reduction_start = max_drawdown * 0.5\n    if current_dd >= max_drawdown:\n        return 0.0  # Stop trading\n\n    if current_dd <= reduction_start:\n        return base_risk_fraction\n\n    # Linear interpolation\n    progress = (current_dd - reduction_start) / (max_drawdown - reduction_start)\n    return base_risk_fraction * (1.0 - progress)\n```\n\n### For Your System (15% Max DD, 288 trades/day)\n\nAssuming per-trade volatility of ~0.3% (typical for 5-min BTC):\n- At 99% confidence: risk fraction ≈ 0.10-0.20% per trade\n- At 95% confidence: risk fraction ≈ 0.15-0.30% per trade\n\nThis aligns with the quarter-Kelly / 0.25% fixed fractional numbers.\n\n---\n\n## 8. LEVERAGE OPTIMIZATION\n\n### Optimal Leverage Formula\n\nFor a system with:\n- Expected return per trade: `μ`\n- Variance per trade: `σ²`\n- Transaction costs per trade: `c`\n\n**Optimal leverage:**\n```\nL* = (μ - c) / σ²\n```\n\nThis is just the continuous Kelly criterion. Leverage IS position sizing for futures.\n\n### For a 55% Accuracy System\n\nAssuming 1:1 risk/reward with 0.3% average trade size:\n- `μ` = 0.55 × 0.003 - 0.45 × 0.003 = 0.0003 (0.03% per trade)\n- `σ²` ≈ 0.003² = 0.000009\n- Full Kelly leverage: 0.0003 / 0.000009 ≈ 33x\n\n**But this is full Kelly — suicidal.** Quarter Kelly gives ~8x, which is more reasonable.\n\n### Leverage and Funding Rate Interaction\n\nPerpetual futures have funding rates, typically charged every 8 hours. On BTC:\n- Average funding rate: ±0.01% per 8 hours (±0.03% per day)\n- Can spike to ±0.1% per 8 hours during extreme sentiment\n- At 10x leverage: funding cost = 0.1% × 10 = 1.0% of equity per day during spikes\n\n**This eats your edge.** With a 0.03% expected return per trade and 288 trades/day:\n- Daily expected return: 288 × 0.03% = 8.64% (unleveraged on position)\n- Daily funding cost at 10x during normal times: 10 × 0.03% = 0.3%\n- Net: 8.34% (funding is small)\n\nBut during extreme funding:\n- Daily funding cost at 10x: 10 × 0.3% = 3.0%\n- This is significant and must be factored into sizing.\n\n### Implementation\n\n```python\nimport numpy as np\n\ndef optimal_leverage(\n    expected_return_per_trade: float,\n    variance_per_trade: float,\n    transaction_cost_per_trade: float,\n    funding_rate_per_trade: float,\n    kelly_fraction: float = 0.25,\n    max_leverage: float = 20.0,\n) -> dict:\n    \"\"\"\n    Compute optimal leverage for perpetual futures.\n\n    Args:\n        expected_return_per_trade: Mean return per trade (e.g., 0.0003).\n        variance_per_trade: Variance of returns per trade.\n        transaction_cost_per_trade: Taker/maker fee per trade.\n        funding_rate_per_trade: Pro-rated funding cost per trade.\n        kelly_fraction: Fraction of Kelly to use.\n        max_leverage: Exchange maximum leverage cap.\n\n    Returns:\n        Optimal leverage and related metrics.\n    \"\"\"\n    net_edge = expected_return_per_trade - transaction_cost_per_trade - funding_rate_per_trade\n\n    if net_edge <= 0:\n        return {\n            \"optimal_leverage\": 0.0,\n            \"reason\": \"No edge after costs\",\n            \"net_edge\": net_edge,\n        }\n\n    if variance_per_trade <= 0:\n        return {\n            \"optimal_leverage\": 0.0,\n            \"reason\": \"Invalid variance\",\n        }\n\n    full_kelly_leverage = net_edge / variance_per_trade\n    fractional_leverage = full_kelly_leverage * kelly_fraction\n    capped_leverage = min(fractional_leverage, max_leverage)\n\n    # Expected growth rate at this leverage\n    growth_rate = capped_leverage * net_edge - 0.5 * (capped_leverage ** 2) * variance_per_trade\n\n    return {\n        \"optimal_leverage\": capped_leverage,\n        \"full_kelly_leverage\": full_kelly_leverage,\n        \"fractional_kelly_leverage\": fractional_leverage,\n        \"net_edge_per_trade\": net_edge,\n        \"expected_growth_per_trade\": growth_rate,\n        \"expected_daily_growth\": growth_rate * 288,\n        \"max_leverage_cap\": max_leverage,\n    }\n\ndef funding_aware_sizing(\n    base_leverage: float,\n    current_funding_rate: float,\n    funding_threshold: float = 0.0005,\n    reduction_factor: float = 0.5,\n) -> float:\n    \"\"\"\n    Reduce leverage when funding rates are extreme.\n\n    Args:\n        base_leverage: Normal operating leverage.\n        current_funding_rate: Current 8-hour funding rate.\n        funding_threshold: Rate above which to reduce size.\n        reduction_factor: How much to reduce (0.5 = halve leverage).\n\n    Returns:\n        Adjusted leverage.\n    \"\"\"\n    abs_funding = abs(current_funding_rate)\n\n    if abs_funding <= funding_threshold:\n        return base_leverage\n\n    # Linear reduction above threshold\n    excess = abs_funding - funding_threshold\n    reduction = min(excess / funding_threshold, 1.0) * (1.0 - reduction_factor)\n\n    return base_leverage * (1.0 - reduction)\n```\n\n### Practical Leverage Recommendations\n\n| System Edge (after costs) | Quarter Kelly Leverage | Max DD (expected) |\n|--------------------------|----------------------|-------------------|\n| 0.01% per trade | 2-3x | 10-15% |\n| 0.03% per trade | 5-8x | 15-25% |\n| 0.05% per trade | 8-12x | 20-30% |\n| 0.10% per trade | 15-20x | 30-45% |\n\n**For a realistic 55% win rate system with costs: 3-8x leverage is the sweet spot.**\n\n---\n\n## 9. POSITION SIZING WITH UNCERTAIN EDGE — BAYESIAN KELLY\n\n### The Problem\n\nYour estimated win rate is not a point estimate — it has uncertainty. If you think your win rate is 55% but it could be anywhere from 52% to 58%, full Kelly at 55% will lead to over-betting when the true rate is 52% and under-betting when it's 58%.\n\n### Bayesian Kelly Framework\n\nInstead of computing Kelly from a point estimate, compute the expected Kelly fraction by integrating over the posterior distribution of the edge:\n\n```\nf_bayesian = E_θ[f_kelly(θ)]    — but this is wrong!\n```\n\n**The correct formulation** (due to the concavity of log utility):\n\n```\nf_bayesian = argmax_f  E_θ[E[ln(1 + f × R) | θ]]\n```\n\nThis is NOT the same as computing Kelly for each possible θ and averaging. Due to Jensen's inequality, the Bayesian optimal f is LESS than the average Kelly.\n\n### Implementation\n\n```python\nimport numpy as np\nfrom scipy.optimize import minimize_scalar\nfrom scipy.stats import beta as beta_dist\n\ndef bayesian_kelly(\n    wins: int,\n    losses: int,\n    avg_win: float,\n    avg_loss: float,\n    prior_alpha: float = 1.0,\n    prior_beta: float = 1.0,\n    kelly_fraction: float = 0.25,\n    n_samples: int = 5000,\n) -> dict:\n    \"\"\"\n    Bayesian Kelly criterion with Beta posterior over win rate.\n\n    Args:\n        wins: Number of winning trades observed.\n        losses: Number of losing trades observed.\n        avg_win: Average winning trade return.\n        avg_loss: Average losing trade return (positive number).\n        prior_alpha: Beta prior alpha (1.0 = uniform).\n        prior_beta: Beta prior beta (1.0 = uniform).\n        kelly_fraction: Fraction of Bayesian Kelly to use.\n        n_samples: Monte Carlo samples for integration.\n\n    Returns:\n        Bayesian Kelly fraction and diagnostics.\n    \"\"\"\n    # Posterior distribution: Beta(alpha + wins, beta + losses)\n    post_alpha = prior_alpha + wins\n    post_beta = prior_beta + losses\n\n    # Sample win rates from posterior\n    p_samples = beta_dist.rvs(post_alpha, post_beta, size=n_samples)\n\n    # For each sampled win rate, compute the expected log growth\n    def expected_log_growth(f: float) -> float:\n        total_growth = 0.0\n        for p in p_samples:\n            # Expected log growth for this win rate\n            if f * avg_win > 0 and f * avg_loss < 1:\n                growth = p * np.log(1 + f * avg_win) + (1 - p) * np.log(1 - f * avg_loss)\n                total_growth += growth\n            else:\n                total_growth += -1e10\n        return -total_growth / n_samples\n\n    result = minimize_scalar(expected_log_growth, bounds=(0.001, 0.99), method='bounded')\n    bayesian_f = result.x\n\n    # Point estimate Kelly for comparison\n    p_hat = post_alpha / (post_alpha + post_beta)\n    point_kelly = p_hat / avg_loss - (1 - p_hat) / avg_win\n\n    # Posterior statistics\n    p_mean = post_alpha / (post_alpha + post_beta)\n    p_std = np.sqrt(\n        (post_alpha * post_beta)\n        / ((post_alpha + post_beta) ** 2 * (post_alpha + post_beta + 1))\n    )\n\n    return {\n        \"bayesian_kelly_f\": bayesian_f * kelly_fraction,\n        \"full_bayesian_kelly\": bayesian_f,\n        \"point_estimate_kelly\": max(0, point_kelly),\n        \"win_rate_mean\": p_mean,\n        \"win_rate_std\": p_std,\n        \"win_rate_95ci\": (\n            beta_dist.ppf(0.025, post_alpha, post_beta),\n            beta_dist.ppf(0.975, post_alpha, post_beta),\n        ),\n        \"n_trades_observed\": wins + losses,\n        \"reduction_from_point\": 1 - (bayesian_f / max(point_kelly, 0.001)),\n    }\n\ndef minimum_trades_for_confidence(\n    target_precision: float = 0.02,\n    confidence: float = 0.95,\n) -> int:\n    \"\"\"\n    How many trades needed to estimate win rate within target_precision?\n\n    For a binomial proportion, the CI width is approximately:\n    width = 2 × z × √(p(1-p)/n)\n\n    For worst case (p=0.5):\n    n = (z / target_precision)² × 0.25\n\n    Args:\n        target_precision: Desired half-width of CI (0.02 = ±2%).\n        confidence: Confidence level.\n\n    Returns:\n        Minimum number of trades needed.\n    \"\"\"\n    from scipy.stats import norm\n    z = norm.ppf(0.5 + confidence / 2)\n    n = (z / target_precision) ** 2 * 0.25\n    return int(np.ceil(n))\n\n# Example:\n# To know win rate within ±2% with 95% confidence: need ~2,401 trades\n# At 288 trades/day, that's ~8.3 days of trading\n# To know within ±1%: need ~9,604 trades (~33 days)\n```\n\n### Key Insight\n\nWith 500 trades and 55% observed win rate, your 95% CI is roughly [50.6%, 59.4%]. Kelly at 50.6% win rate with 1:1 payoff is only 1.2%, while Kelly at 59.4% is 18.8%. The Bayesian approach accounts for this uncertainty and sizes conservatively — typically 30-50% less than point-estimate Kelly.\n\n**This is one of the strongest arguments for fractional Kelly.** Even if you don't explicitly do the Bayesian calculation, using quarter Kelly implicitly accounts for parameter uncertainty.\n\n---\n\n## 10. COMPOUNDING EFFECTS AT HIGH FREQUENCY\n\n### The Problem\n\nAt 288 trades/day, small sizing errors compound rapidly:\n\n- A 0.1% over-allocation per trade compounds to: (1.001)^288 - 1 = 33.4% over-exposure per day\n- Over a week: (1.001)^(288×7) - 1 = 676% over-exposure\n\n### Fixed vs Proportional Sizing\n\n**Fixed dollar sizing:** Risk $X per trade regardless of equity changes.\n- Pro: No compounding drift.\n- Con: Doesn't grow with account; doesn't shrink during drawdowns.\n\n**Proportional sizing:** Risk X% of current equity per trade.\n- Pro: Grows with account (geometric growth); naturally deleverages during drawdowns.\n- Con: Requires recalculation every trade; rounding errors accumulate.\n\n**For 5-minute crypto: Use proportional sizing but recalculate from actual equity, not estimated equity.**\n\n### Preventing Position Size Drift\n\n```python\nimport numpy as np\nimport logging\n\nlogger = logging.getLogger(__name__)\n\nclass PositionSizeManager:\n    \"\"\"\n    Manages position sizing with drift prevention for high-frequency trading.\n    \"\"\"\n\n    def __init__(\n        self,\n        initial_equity: float,\n        base_risk_fraction: float = 0.0025,\n        recalc_interval: int = 50,\n        max_drift_pct: float = 0.10,\n        equity_fetch_fn=None,\n    ):\n        \"\"\"\n        Args:\n            initial_equity: Starting account equity.\n            base_risk_fraction: Risk per trade as fraction of equity.\n            recalc_interval: Recalculate from true equity every N trades.\n            max_drift_pct: Maximum allowed drift before forced recalc.\n            equity_fetch_fn: Callable that returns actual exchange equity.\n        \"\"\"\n        self.base_risk = base_risk_fraction\n        self.recalc_interval =
