Target Variable Construction\n\n### Options\n\n**Simple Return:**\n$$y_t = \\frac{P_{t+h} - P_t}{P_t}$$\n\n**Log Return (preferred continuous target):**\n$$y_t = \\ln\\left(\\frac{P_{t+h}}{P_t}\\right)$$\n\n**Binary Classification (up/down):**\n$$y_t = \\begin{cases} 1 & \\text{if } r_{t+h} > 0 \\\\ 0 & \\text{if } r_{t+h} \\leq 0 \\end{cases}$$\n\n**Ternary Classification (up/flat/down):**\n$$y_t = \\begin{cases} +1 & \\text{if } r_{t+h} > \\tau \\\\ 0 & \\text{if } |r_{t+h}| \\leq \\tau \\\\ -1 & \\text{if } r_{t+h} < -\\tau \\end{cases}$$\n\n**Quantile-Based (adaptive threshold):**\n$$y_t = \\text{quantile\\_bin}(r_{t+h}, \\text{rolling quantiles})$$\n\n### Threshold Calibration\n\n```python\ndef compute_adaptive_threshold(\n    returns: pd.Series,\n    window: int = 2016,  # 1 week\n    flat_quantile: float = 0.33,  # middle 33% is \"flat\"\n) -> pd.Series:\n    \"\"\"\n    Adaptive threshold based on recent volatility.\n    Fixed thresholds fail because BTC volatility changes 10x across regimes.\n    \n    For 5-min BTC returns:\n    - Low vol periods: threshold ~ 0.05% (5 bps)\n    - High vol periods: threshold ~ 0.30% (30 bps)\n    \"\"\"\n    abs_returns = returns.abs()\n    threshold = abs_returns.rolling(window=window, min_periods=100).quantile(\n        flat_quantile\n    )\n    return threshold\n\n\ndef construct_ternary_target(\n    close: pd.Series,\n    horizon: int = 1,  # 1 bar = 5 min ahead\n    flat_quantile: float = 0.33,\n    window: int = 2016,\n) -> pd.Series:\n    \"\"\"\n    Ternary target: -1, 0, +1 with adaptive threshold.\n    \"\"\"\n    # Forward return (what we're predicting)\n    fwd_return = np.log(close.shift(-horizon) / close)\n    \n    # Adaptive threshold from PAST returns only (no leakage)\n    past_returns = np.log(close / close.shift(1))\n    threshold = compute_adaptive_threshold(past_returns, window, flat_quantile)\n    \n    target = pd.Series(0, index=close.index, dtype=int)\n    target[fwd_return > threshold] = 1\n    target[fwd_return < -threshold] = -1\n    \n    return target\n\n\ndef construct_quantile_target(\n    close: pd.Series,\n    horizon: int = 1,\n    n_bins: int = 5,\n    window: int = 2016,\n) -> pd.Series:\n    \"\"\"\n    Quantile-based target. Maps forward returns into n_bins based on\n    rolling distribution. Automatically handles changing volatility.\n    \n    Bin 0 = most negative returns, Bin n_bins-1 = most positive.\n    \"\"\"\n    fwd_return = np.log(close.shift(-horizon) / close)\n    \n    def rolling_quantile_bin(arr: np.ndarray) -> float:\n        if len(arr) < 100:\n            return np.nan\n        current = arr[-1]\n        past = arr[:-1]\n        quantiles = np.nanpercentile(\n            past, np.linspace(0, 100, n_bins + 1)[1:-1]\n        )\n        return np.searchsorted(quantiles, current)\n    \n    return fwd_return.rolling(window=window, min_periods=100).apply(\n        rolling_quantile_bin, raw=True\n    )\n```\n\n**Recommendation for 5-min BTC prediction:**\n\nUse **ternary classification** with adaptive threshold as the primary target:\n- It acknowledges that \"no meaningful move\" is a valid prediction\n- Adaptive threshold handles regime changes\n- The `flat_quantile=0.33` means roughly 33% of bars are \"flat\" — adjust based on your strategy's minimum required move to be profitable (must exceed trading costs)\n- Specifically: if round-trip cost is 6 bps (3 bps taker each way), set threshold floor at 6 bps\n\nFor ensemble diversity, also train a regression head on log returns — the continuous target provides richer gradient signal during training.\n\n---\n\n## 7. Class Imbalance\n\n### Typical Distribution for 5-Min BTC Ternary Target\n\nWith adaptive threshold at the 33rd percentile of absolute returns:\n- Up: ~33%\n- Flat: ~33%  \n- Down: ~33%\n\nThis is balanced by construction. But if you use a fixed threshold, you might see:\n- Up: ~20%\n- Flat: ~60%\n- Down: ~20%\n\n### Handling Approaches\n\n```python\nimport logging\nfrom collections import Counter\n\nimport numpy as np\nimport pandas as pd\n\nlogger = logging.getLogger(__name__)\n\n\ndef compute_class_weights(y: pd.Series) -> dict[int, float]:\n    \"\"\"\n    Inverse frequency class weights.\n    Equivalent to sklearn's class_weight='balanced'.\n    \"\"\"\n    counts = Counter(y.dropna().astype(int))\n    total = sum(counts.values())\n    n_classes = len(counts)\n    \n    weights = {}\n    for cls, count in counts.items():\n        weights[cls] = total / (n_classes * count)\n    \n    logger.info(f\"Class distribution: {dict(counts)}\")\n    logger.info(f\"Class weights: {weights}\")\n    return weights\n\n\ndef temporal_class_weights(\n    y: pd.Series,\n    window: int = 8640,  # 30 days of 5-min bars\n    min_periods: int = 1000,\n) -> pd.DataFrame:\n    \"\"\"\n    Time-varying class weights. Important for crypto because\n    class distribution shifts across market regimes.\n    \n    Bull market: more +1, fewer -1\n    Bear market: more -1, fewer +1\n    Range-bound: more 0\n    \"\"\"\n    weights_df = pd.DataFrame(index=y.index, columns=[-1, 0, 1], dtype=float)\n    \n    for cls in [-1, 0, 1]:\n        cls_indicator = (y == cls).astype(float)\n        rolling_freq = cls_indicator.rolling(\n            window=window, min_periods=min_periods\n        ).mean()\n        # Weight = 1/3 / frequency (balanced target = 1/3 each)\n        weights_df[cls] = (1.0 / 3.0) / rolling_freq.clip(lower=0.01)\n    \n    return weights_df\n```\n\n### What Works and What Doesn't\n\n| Method | Recommendation | Reason |\n|---|---|---|\n| Adaptive threshold | **Best** | Prevents imbalance by construction |\n| Class weights | **Good** | Simple, no data synthesis, works with any model |\n| Temporal class weights | **Good** | Handles regime shifts |\n| SMOTE | **Avoid** | Synthesizes fake financial data — breaks temporal structure, creates leakage between train/val |\n| Random oversampling | **Avoid** | Duplicates exact samples, leads to overfitting on specific market conditions |\n| Random undersampling | **OK with care** | Loses data, but acceptable if you have millions of bars |\n\n**Recommendation:** Use adaptive threshold to achieve roughly equal classes, then use class weights to fine-tune. Never use SMOTE or synthetic oversampling for time series — it fundamentally violates the temporal structure.\n\nFor XGBoost: `scale_pos_weight` parameter or `sample_weight` in `.fit()`\nFor neural nets: weighted cross-entropy loss\n\n```python\nimport torch\nimport torch.nn as nn\n\nclass WeightedCrossEntropyLoss(nn.Module):\n    \"\"\"Weighted CE loss for ternary classification.\"\"\"\n    \n    def __init__(self, class_weights: dict[int, float]):\n        super().__init__()\n        # Map from {-1: w, 0: w, 1: w} to tensor indexed [0, 1, 2]\n        weight_tensor = torch.tensor([\n            class_weights.get(-1, 1.0),\n            class_weights.get(0, 1.0),\n            class_weights.get(1, 1.0),\n        ], dtype=torch.float32)\n        self.loss_fn = nn.CrossEntropyLoss(weight=weight_tensor)\n    \n    def forward(\n        self, predictions: torch.Tensor, targets: torch.Tensor\n    ) -> torch.Tensor:\n        # Targets must be remapped: {-1, 0, 1} -> {0, 1, 2}\n        targets_remapped = targets + 1\n        return self.loss_fn(predictions, targets_remapped.long())\n```\n\n---\n\n## 8. Feature Decorrelation\n\n### The Problem\n\nTypical crypto feature sets are highly correlated:\n- RSI_14 and RSI_21: ~0.95 correlation\n- SMA_20 and EMA_20: ~0.99 correlation\n- Volume and dollar_volume: ~0.98 correlation\n- Various momentum indicators: 0.7-0.9 correlations\n\n### PCA for Decorrelation\n\n```python\nimport logging\n\nimport numpy as np\nimport pandas as pd\nfrom sklearn.decomposition import PCA, IncrementalPCA\n\nlogger = logging.getLogger(__name__)\n\n\nclass RollingPCA:\n    \"\"\"\n    Rolling PCA for feature decorrelation without look-ahead bias.\n    \n    Fits PCA on a rolling window of past data, transforms current data.\n    \"\"\"\n    \n    def __init__(\n        self,\n        n_components: float | int = 0.95,  # float = explained variance ratio\n        window: int = 8640,  # 30 days\n        refit_interval: int = 288,  # refit every 24 hours\n    ):\n        self.n_components = n_components\n        self.window = window\n        self.refit_interval = refit_interval\n    \n    def transform(self, df: pd.DataFrame) -> pd.DataFrame:\n        \"\"\"\n        Apply rolling PCA. Returns DataFrame with PC_0, PC_1, ... columns.\n        \"\"\"\n        result_frames = []\n        last_fit_idx = -self.refit_interval  # force fit on first iteration\n        pca = None\n        \n        for i in range(self.window, len(df)):\n            # Refit PCA periodically\n            if i - last_fit_idx >= self.refit_interval:\n                window_data = df.iloc[i - self.window:i].dropna()\n                if len(window_data) < 100:\n                    continue\n                \n                pca = PCA(n_components=self.n_components)\n                pca.fit(window_data.values)\n                last_fit_idx = i\n                \n                n_kept = pca.n_components_\n                logger.info(\n                    f\"PCA refit at index {i}: keeping {n_kept} components \"\n                    f\"({pca.explained_variance_ratio_.sum():.2%} variance)\"\n                )\n            \n            if pca is not None:\n                row = df.iloc[i:i+1].values\n                if not np.any(np.isnan(row)):\n                    transformed = pca.transform(row)\n                    row_df = pd.DataFrame(\n                        transformed,\n                        index=[df.index[i]],\n                        columns=[f\"PC_{j}\" for j in range(transformed.shape[1])],\n                    )\n                    result_frames.append(row_df)\n        \n        if not result_frames:\n            return pd.DataFrame()\n        return pd.concat(result_frames)\n```\n\n### Does Decorrelation Help Tree Models?\n\n**Short answer: No, and it can hurt.**\n\nTree models (XGBoost, LightGBM) handle correlated features naturally:\n- They split on one feature at a time — correlated features are redundant but not harmful\n- Feature importance is shared among correlated features (which can be misleading for interpretation but doesn't hurt prediction)\n- PCA destroys interpretability — PC_3 means nothing\n- PCA is a linear transformation — trees can learn non-linear interactions from raw features that PCA destroys\n\n**For neural networks:** PCA can help by:\n- Reducing input dimensionality (fewer parameters, faster training)\n- Breaking multicollinearity that can create saddle points in optimization\n- But it adds complexity and another source of potential bugs\n\n**Recommendation:** Skip PCA for the tree component of the ensemble. For neural nets, prefer **manual feature selection** (drop one of each highly correlated pair) over PCA. Use PCA only if you have 100+ features and need dimensionality reduction.\n\n### Better Alternative: Correlation-Based Feature Pruning\n\n```python\ndef prune_correlated_features(\n    df: pd.DataFrame,\n    threshold: float = 0.90,\n    method: str = \"spearman\",\n) -> list[str]:\n    \"\"\"\n    Remove features with correlation above threshold.\n    Keeps the feature with higher variance (more informative).\n    Returns list of features to KEEP.\n    \"\"\"\n    corr_matrix = df.corr(method=method).abs()\n    upper_tri = corr_matrix.where(\n        np.triu(np.ones(corr_matrix.shape, dtype=bool), k=1)\n    )\n    \n    to_drop = set()\n    for col in upper_tri.columns:\n        high_corr = upper_tri.index[upper_tri[col] > threshold].tolist()\n        for correlated_col in high_corr:\n            # Drop the one with lower variance\n            if df[col].var() >= df[correlated_col].var():\n                to_drop.add(correlated_col)\n            else:\n                to_drop.add(col)\n    \n    kept = [c for c in df.columns if c not in to_drop]\n    logger.info(\n        f\"Pruned {len(to_drop)} correlated features \"\n        f\"(threshold={threshold}): {sorted(to_drop)}\"\n    )\n    return kept\n```\n\n---\n\n## 9. Cointegration\n\n### Theory\n\nTwo series $X_t$ and $Y_t$ are cointegrated if both are I(1) (non-stationary, integrated of order 1) but a linear combination $Y_t - \\beta X_t$ is I(0) (stationary). This means they share a common stochastic trend — they may diverge temporarily but revert to their equilibrium relationship.\n\n### Testing for Cointegration\n\n**Engle-Granger Two-Step Test:**\n\n```python\nimport logging\n\nimport numpy as np\nimport pandas as pd\nfrom statsmodels.tsa.stattools import adfuller, coint\n\nlogger = logging.getLogger(__name__)\n\n\ndef test_cointegration(\n    series_a: pd.Series,\n    series_b: pd.Series,\n    significance: float = 0.05,\n) -> dict:\n    \"\"\"\n    Engle-Granger cointegration test.\n    \n    Common pairs to test with BTC:\n    - BTC vs ETH (strongest cointegration in crypto)\n    - BTC vs total crypto market cap\n    - BTC vs BTC perpetual futures (basis trade)\n    - BTC vs inverse dollar index (DXY)\n    \"\"\"\n    # Align indices\n    aligned = pd.concat(\n        [series_a.rename(\"a\"), series_b.rename(\"b\")], axis=1\n    ).dropna()\n    \n    # Engle-Granger test\n    coint_stat, pvalue, crit_values = coint(aligned[\"a\"], aligned[\"b\"])\n    \n    is_cointegrated = pvalue < significance\n    \n    # If cointegrated, compute the cointegrating vector\n    hedge_ratio = np.nan\n    spread_half_life = np.nan\n    \n    if is_cointegrated:\n        # OLS regression: b = alpha + beta * a + epsilon\n        from numpy.polynomial.polynomial import polyfit\n        beta, alpha = np.polyfit(aligned[\"a\"], aligned[\"b\"], 1)\n        hedge_ratio = beta\n        \n        # Compute spread\n        spread = aligned[\"b\"] - hedge_ratio * aligned[\"a\"]\n        \n        # Half-life of mean reversion (Ornstein-Uhlenbeck)\n        spread_lag = spread.shift(1).dropna()\n        spread_diff = spread.diff().dropna()\n        common_idx = spread_lag.index.intersection(spread_diff.index)\n        \n        # Regress delta_spread on spread_lag\n        slope = np.polyfit(spread_lag.loc[common_idx], spread_diff.loc[common_idx], 1)[0]\n        if slope < 0:\n            spread_half_life = -np.log(2) / slope\n    \n    result = {\n        \"coint_statistic\": coint_stat,\n        \"pvalue\": pvalue,\n        \"is_cointegrated\": is_cointegrated,\n        \"hedge_ratio\": hedge_ratio,\n        \"spread_half_life_bars\": spread_half_life,\n    }\n    \n    logger.info(\n        f\"Cointegration test: p={pvalue:.4f}, \"\n        f\"cointegrated={is_cointegrated}, \"\n        f\"hedge_ratio={hedge_ratio:.4f}, \"\n        f\"half_life={spread_half_life:.1f} bars\"\n    )\n    return result\n```\n\n**Johansen Test (for multiple series):**\n\n```python\nfrom statsmodels.tsa.vector_ar.vecm import coint_johansen\n\n\ndef test_johansen_cointegration(\n    df: pd.DataFrame,\n    det_order: int = 0,  # 0 = no deterministic terms, -1 = no constant\n    k_ar_diff: int = 5,\n) -> dict:\n    \"\"\"\n    Johansen test for cointegration among multiple series.\n    Use when testing BTC, ETH, SOL, etc. simultaneously.\n    \n    Returns number of cointegrating relationships and eigenvectors.\n    \"\"\"\n    result = coint_johansen(df.dropna(), det_order=det_order, k_ar_diff=k_ar_diff)\n    \n    # Trace statistic test\n    trace_stats = result.lr1\n    trace_crit_95 = result.cvt[:, 1]  # 95% critical values\n    \n    n_coint = sum(trace_stats > trace_crit_95)\n    \n    return {\n        \"n_cointegrating_relations\": n_coint,\n        \"trace_statistics\": trace_stats.tolist(),\n        \"trace_critical_95\": trace_crit_95.tolist(),\n        \"eigenvectors\": result.evec.tolist(),\n    }\n```\n\n### Using Cointegration for Feature Construction\n\n```python\ndef cointegration_features(\n    btc_price: pd.Series,\n    other_price: pd.Series,\n    hedge_ratio: float,\n    spread_half_life: float,\n    window: int = 2016,  # 1 week\n) -> pd.DataFrame:\n    \"\"\"\n    Construct features from cointegrated pair spread.\n    \n    Key features:\n    1. Normalized spread (z-score) — mean-reverting by construction\n    2. Spread velocity — how fast spread is changing\n    3. Spread distance from equilibrium in half-life units\n    \"\"\"\n    spread = other_price - hedge_ratio * btc_price\n    \n    features = pd.DataFrame(index=btc_price.index)\n    \n    # Z-score of spread\n    roll_mean = spread.rolling(window=window, min_periods=100).mean()\n    roll_std = spread.rolling(window=window, min_periods=100).std()\n    features[\"spread_zscore\"] = (spread - roll_mean) / roll_std.replace(0, np.nan)\n    \n    # Spread velocity (change per bar)\n    features[\"spread_velocity\"] = spread.diff()\n    \n    # Spread acceleration\n    features[\"spread_accel\"] = features[\"spread_velocity\"].diff()\n    \n    # Bars since last equilibrium crossing\n    zero_cross = (spread - roll_mean).apply(np.sign).diff().abs() > 0\n    features[\"bars_since_equilibrium\"] = (\n        zero_cross.cumsum()\n        .groupby(zero_cross.cumsum())\n        .cumcount()\n    )\n    \n    # Half-life ratio (how many half-lives from equilibrium)\n    if spread_half_life > 0:\n        features[\"half_life_distance\"] = (\n            features[\"spread_zscore\"].abs() * spread_half_life\n        )\n    \n    return features\n```\n\n**BTC-specific cointegration findings:**\n- BTC-ETH: Strong cointegration, hedge ratio ~0.05 (but shifts over time), half-life ~50-200 bars at 5-min\n- BTC-BTC_PERP: Very strong cointegration (it's the same asset), spread = funding rate basis, half-life ~24-72 bars\n- BTC-DXY: Weak/unstable cointegration, breaks during crypto-specific events\n- Recommendation: Use BTC-ETH spread z-score and BTC perpetual basis as features. Recompute hedge ratio monthly with rolling Engle-Granger.\n\n---\n\n## 10. Non-Linear Transformations\n\n### Box-Cox Transform\n\n$$y = \\begin{cases} \\frac{x^\\lambda - 1}{\\lambda} & \\text{if } \\lambda \\neq 0 \\\\ \\ln(x) & \\text{if } \\lambda = 0 \\end{cases}$$\n\n**Requires strictly positive values.** Works for: volume, volatility, bid-ask spread.\n\n### Yeo-Johnson Transform\n\nGeneralization of Box-Cox that works with negative values:\n\n$$\\psi(x, \\lambda) = \\begin{cases} \\frac{(x+1)^\\lambda - 1}{\\lambda} & \\text{if } \\lambda \\neq 0, x \\geq 0 \\\\ \\ln(x+1) & \\text{if } \\lambda = 0, x \\geq 0 \\\\ -\\frac{(-x+1)^{2-\\lambda} - 1}{2-\\lambda} & \\text{if } \\lambda \\neq 2, x < 0 \\\\ -\\ln(-x+1) & \\text{if } \\lambda = 2, x < 0 \\end{cases}$$\n\n```python\nfrom scipy.stats import yeojohnson, boxcox, yeojohnson_normmax\n\nimport logging\nimport numpy as np\nimport pandas as pd\n\nlogger = logging.getLogger(__name__)\n\n\ndef rolling_yeo_johnson(\n    series: pd.Series,\n    window: int = 8640,  # 30 days\n    refit_interval: int = 2016,  # refit weekly\n    min_periods: int = 1000,\n) -> pd.Series:\n    \"\"\"\n    Rolling Yeo-Johnson transform. Fits lambda on past data,\n    applies to current data. No look-ahead bias.\n    \"\"\"\n    result = pd.Series(np.nan, index=series.index)\n    last_lambda = None\n    last_fit_idx = -refit_interval\n    \n    for i in range(window, len(series)):\n        if i - last_fit_idx >= refit_interval:\n            window_data = series.iloc[i - window:i].dropna()\n            if len(window_data) >= min_periods:\n                try:\n                    last_lambda = yeojohnson_normmax(window_data.values)\n                    last_fit_idx = i\n                except Exception:\n                    pass  # keep previous lambda\n        \n        if last_lambda is not None:\n            val = series.iloc[i]\n            if not np.isnan(val):\n                transformed, _ = yeojohnson([val], lmbda=last_lambda)\n                result.iloc[i] = transformed[0]\n    \n    return result\n```\n\n**Are these useful for financial features?**\n\n| Feature Type | Transform | Useful? |\n|---|---|---|\n| Volume | log(1+x) or Box-Cox | Yes — volume is extremely right-skewed |\n| Volatility (ATR, std) | log or Box-Cox | Yes — makes it more Gaussian |\n| Returns | Yeo-Johnson | Marginal — returns are already ~symmetric |\n| RSI | None needed | No — already bounded [0, 100] |\n| Bid-ask spread | log | Yes — heavily right-skewed |\n| Order book imbalance | Yeo-Johnson | Marginal |\n\n**Recommendation:** For 5-min BTC prediction:\n- Apply `log(1 + x)` to volume and dollar_volume before any further normalization\n- Apply `log` to ATR and other volatility measures\n- Skip Box-Cox/Yeo-Johnson for returns — the added complexity and rolling refit cost isn't worth the marginal improvement\n- For neural nets, rank-to-gaussian normalization is simpler and more effective than Yeo-Johnson\n\n---\n\n## 11. Data Leakage Sources in Crypto ML\n\nThis section is critical. Data leakage is the number one reason crypto ML models look great in backtesting and fail in production.\n\n### Leakage Source 1: Close Price in Features\n\n**Problem:** A \"close\" price at timestamp T is the last trade price in the [T-5min, T] window. If your target is the direction of the next bar, using the close price is fine. But using features that depend on the close AND computing the target from the close creates a subtle correlation.\n\n**Real danger:** Features like \"close - open\" (candle body) of the current bar include information that was only known at bar close time. If your model is meant to predict at bar open, this is leakage.\n\n```python\ndef define_temporal_boundaries(\n    df: pd.DataFrame,\n    prediction_point: str = \"open\",  # when the model makes a prediction\n) -> pd.DataFrame:\n    \"\"\"\n    Correctly partition what's known at prediction time.\n    \n    If prediction_point = \"open\":\n      - Features can use: all data from PREVIOUS bars (t-1 and earlier)\n      - Features can use: open price of current bar (it's the first trade)\n      - Features CANNOT use: high, low, close, volume of current bar\n      - Target: direction from current open to next bar's open\n    \n    If prediction_point = \"close\":\n      - Features can use: all data from current and previous bars\n      - Target: direction from current close to next bar's close\n    \"\"\"\n    result = df.copy()\n    \n    if prediction_point == \"open\":\n        # Shift current-bar features to prevent leakage\n        current_bar_features = [\"high\", \"low\", \"close\", \"volume\"]\n        for col in current_bar_features:\n            if col in result.columns:\n                # Use PREVIOUS bar's value\n                result[f\"prev_{col}\"] = result[col].shift(1)\n        \n        # Target: next bar's open vs current bar's open\n        result[\"target_return\"] = np.log(\n            result[\"open\"].shift(-1) / result[\"open\"]\n        )\n    \n    elif prediction_point == \"close\":\n        # Target: next bar's close vs current bar's close\n        result[\"target_return\"] = np.log(\n            result[\"close\"].shift(-1) / result[\"close\"]\n        )\n    \n    return result\n```\n\n### Leakage Source 2: Volume at Close\n\n**Problem:** The total volume of a 5-min bar includes trades that happened throughout the entire bar, including the final milliseconds. If you're predicting at bar close, you \"know\" the total volume, but some of that volume (especially the burst right at close from MOC orders or liquidations) is highly predictive of the close price direction.\n\n**Solution:** Use only volume from previous bars, or use a \"volume so far\" that excludes the last N seconds.\n\n### Leakage Source 3: Order Book State\n\n**Problem:** Order book snapshots \"at\" bar close are ambiguous.
