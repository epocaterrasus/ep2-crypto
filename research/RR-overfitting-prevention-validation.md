Walk-Forward Validation Best Practices\n\n### 2.1 Expanding vs Sliding Window\n\n**Expanding window**: Train on all data from the start up to time t, test on t to t+k. Each fold gets more training data.\n\n**Sliding window**: Train on a fixed-size window ending at time t, test on t to t+k. The exact size should be tuned via nested cross-validation.\n\n```python\nimport numpy as np\nimport pandas as pd\n\nclass WalkForwardValidator:\n    \"\"\"\n    Walk-forward validation with purging and embargoing.\n    \"\"\"\n    def __init__(\n        self,\n        train_size: int,\n        test_size: int,\n        purge_size: int,\n        embargo_size: int,\n        expanding: bool = False,\n    ):\n        self.train_size = train_size   # e.g., 30 days of 5-min bars = 8640\n        self.test_size = test_size     # e.g., 7 days = 2016\n        self.purge_size = purge_size   # e.g., 12 bars = 1 hour\n        self.embargo_size = embargo_size  # e.g., 6 bars = 30 min\n        self.expanding = expanding\n\n    def split(self, n_samples: int):\n        \"\"\"Yield (train_indices, test_indices) tuples.\"\"\"\n        splits = []\n        start = 0 if self.expanding else None\n\n        current = self.train_size\n        while current + self.purge_size + self.test_size <= n_samples:\n            if self.expanding:\n                train_start = 0\n            else:\n                train_start = current - Advances in Financial Machine Learning — Key Ideas\n\n### 3.1 Triple Barrier Method for Labeling\n\nInstead of binary \"up/down\" labels, the triple barrier method defines three possible outcomes for each trade:\n\n1. **Upper barrier** (take profit): Price hits profit target within the time horizon\n2. **Lower barrier** (stop loss): Price hits stop-loss within the time horizon\n3. **Vertical barrier** (time expiry): Neither barrier is hit within the time horizon\n\nThis produces labels that are more aligned with actual trading: {+1 (profit target hit), -1 (stop-loss hit), 0 (expired)}.\n\n```python\ndef triple_barrier_labels(\n    close: pd.Series,\n    volatility: pd.Series,  # e.g., rolling std of returns\n    upper_mult: float = 2.0,  # upper barrier = upper_mult * volatility\n    lower_mult: float = 2.0,  # lower barrier = lower_mult * volatility\n    max_holding: int = 12,    # 12 bars = 60 minutes at 5-min\n) -> pd.DataFrame:\n    \"\"\"\n    Apply triple barrier labeling.\n    Returns DataFrame with columns: label, return, touch_time\n    \"\"\"\n    results = []\n\n    for i in range(len(close) - max_holding):\n        entry_price = close.iloc[i]\n        vol = volatility.iloc[i]\n        upper = entry_price * (1 + upper_mult * vol)\n        lower = entry_price * (1 - lower_mult * vol)\n\n        label = 0\n        ret = 0.0\n        touch_idx = i + max_holding\n\n        for j in range(i + 1, i + max_holding + 1):\n            if j >= len(close):\n                break\n            price = close.iloc[j]\n\n            if price >= upper:\n                label = 1\n                ret = (price - entry_price) / entry_price\n                touch_idx = j\n                break\n            elif price <= lower:\n                label = -1\n                ret = (price - entry_price) / entry_price\n                touch_idx = j\n                break\n\n        if label == 0:  # vertical barrier\n            final_price = close.iloc[min(i + max_holding, len(close) - 1)]\n            ret = (final_price - entry_price) / entry_price\n\n        results.append({\n            'entry_idx': i,\n            'label': label,\n            'return': ret,\n            'bars_held': touch_idx - i,\n        })\n\n    return pd.DataFrame(results)\n```\n\n**Applicability to 5-min prediction**: Highly applicable. Use rolling volatility (e.g., 1-hour realized vol) to set dynamic barriers. This naturally adapts to high/low volatility regimes.\n\n### 3.2 Meta-Labeling\n\nMeta-labeling is a two-stage approach:\n\n1. **Primary model** (or rule-based system) generates trade signals (direction only: buy/sell)\n2. **Secondary model** (the meta-labeler) predicts *whether the primary model's signal will be profitable* — it outputs a probability, which determines position sizing\n\nThis decouples *direction* from *confidence*. The meta-labeler's job is to filter bad trades from the primary model.\n\n**Why this helps with overfitting**: The meta-labeler is a simpler problem (binary: \"is this trade going to work?\") than the full prediction problem. Simpler problems are harder to overfit.\n\n**Applicability to 5-min prediction**: Very applicable. Use the ensemble as the primary model, and train a meta-labeler to determine \"should I act on this signal?\"\n\n```python\ndef generate_meta_labels(\n    primary_signals: pd.Series,  # +1 or -1 from primary model\n    close: pd.Series,\n    horizon: int = 1,  # bars\n) -> pd.Series:\n    \"\"\"\n    Given primary model signals, generate meta-labels:\n    1 if the primary signal was correct, 0 otherwise.\n    \"\"\"\n    future_returns = close.pct_change(horizon).shift(-horizon)\n    signal_returns = primary_signals * future_returns\n\n    # 1 if signal was profitable (or at least not a loss), 0 otherwise\n    meta_labels = (signal_returns > 0).astype(int)\n    return meta_labels\n```\n\n### 3.3 Sample Weighting by Uniqueness\n\nNot all samples are equally informative. If two training samples' labels overlap in time (their label evaluation windows overlap), they carry redundant information. Lopez de Prado proposes weighting samples by their \"uniqueness\" — the inverse of the average number of concurrent labels.\n\n```python\ndef compute_sample_uniqueness(\n    label_starts: pd.Series,  # timestamp when label evaluation begins\n    label_ends: pd.Series,    # timestamp when label evaluation ends (barrier touch)\n) -> pd.Series:\n    \"\"\"\n    Compute average uniqueness of each sample.\n    Uniqueness = 1 / avg(concurrent labels at each point in the sample's span).\n    \"\"\"\n    # Build concurrency count at each timestamp\n    all_times = sorted(set(label_starts) | set(label_ends))\n    concurrency = pd.Series(0, index=all_times)\n\n    for start, end in zip(label_starts, label_ends):\n        mask = (concurrency.index >= start) & (concurrency.index <= end)\n        concurrency.loc[mask] += 1\n\n    # Average uniqueness per sample\n    uniqueness = []\n    for start, end in zip(label_starts, label_ends):\n        mask = (concurrency.index >= start) & (concurrency.index <= end)\n        concurrent = concurrency.loc[mask]\n        avg_uniqueness = (1.0 / concurrent).mean()\n        uniqueness.append(avg_uniqueness)\n\n    return pd.Series(uniqueness, index=label_starts.index)\n```\n\n**Applicability**: Directly applicable when using triple barrier labeling with variable holding periods.\n\n### 3.4 Fractional Differentiation\n\nStandard differencing (returns = price[t] - price[t-1]) makes the series stationary but destroys memory (long-range dependencies). Fractional differentiation applies a fractional order d (e.g., 0.3-0.7) that makes the series *just barely* stationary while preserving more memory.\n\nThe fractionally differenced series is:\n\n```\nX_d[t] = sum_{k=0}^{K} w_k * X[t-k]\nwhere w_k = -w_{k-1} * (d - k + 1) / k, w_0 = 1\n```\n\nUse the ADF test to find the minimum d that achieves stationarity.\n\n```python\ndef frac_diff_weights(d: float, threshold: float = 1e-5) -> np.ndarray:\n    \"\"\"Compute fractional differentiation weights.\"\"\"\n    weights = [1.0]\n    k = 1\n    while True:\n        w = -weights[-1] * (d - k + 1) / k\n        if abs(w) < threshold:\n            break\n        weights.append(w)\n        k += 1\n    return np.array(weights)\n\n\ndef frac_diff(series: pd.Series, d: float, threshold: float = 1e-5) -> pd.Series:\n    \"\"\"Apply fractional differentiation of order d.\"\"\"\n    weights = frac_diff_weights(d, threshold)\n    width = len(weights)\n    result = pd.Series(index=series.index, dtype=float)\n\n    for i in range(width - 1, len(series)):\n        window = series.iloc[i - width + 1:i + 1].values\n        result.iloc[i] = np.dot(weights[::-1][:len(window)], window[-len(weights):])\n\n    return result.dropna()\n\n\ndef find_min_d(series: pd.Series, p_value_threshold: float = 0.05) -> float:\n    \"\"\"Find minimum d that makes series stationary (ADF test).\"\"\"\n    from statsmodels.tsa.stattools import adfuller\n\n    for d in np.arange(0.0, 1.05, 0.05):\n        diffed = frac_diff(series, d)\n        if len(diffed.dropna()) < 100:\n            continue\n        adf_result = adfuller(diffed.dropna())\n        if adf_result[1] < p_value_threshold:\n            return round(d, 2)\n    return 1.0  # full differentiation as fallback\n```\n\n**Applicability to 5-min prediction**: Apply to raw prices to create features that are stationary but preserve trend memory. Typical d for BTC: 0.3-0.5.\n\n---\n\n## 4. Deflated Sharpe Ratio\n\nThe standard Sharpe Ratio is biased upward when you test multiple strategies and report the best one. The Deflated Sharpe Ratio (DSR) corrects for this.\n\n**Formula** (Bailey & Lopez de Prado, 2014):\n\n```\nDSR = P[ SR* > 0 ] = Phi( (SR* - SR_0) / sqrt(V[SR*]) )\n\nwhere:\n  SR* = observed Sharpe Ratio of the best strategy\n  SR_0 = expected max SR under the null (all strategies have SR=0):\n    SR_0 ≈ sqrt(2 * log(N)) * (1 - gamma * euler_mascheroni) + gamma * sqrt(2 * log(N))\n    (simplified: SR_0 ≈ sqrt(V[SR]) * ((1 - gamma) * Z^{-1}(1 - 1/N) + gamma * Z^{-1}(1 - 1/(N*e))))\n\n  V[SR*] = (1 + 0.5 * SR*^2 - skew * SR* + (kurt - 3)/4 * SR*^2) / (T - 1)\n\n  N = number of strategies tested\n  T = number of observations\n  skew = skewness of returns\n  kurt = kurtosis of returns\n  gamma ≈ 0.5772 (Euler-Mascheroni constant)\n```\n\n```python\nfrom scipy.stats import norm\nimport numpy as np\n\ndef deflated_sharpe_ratio(\n    observed_sr: float,\n    n_trials: int,\n    n_observations: int,\n    skewness: float = 0.0,\n    kurtosis: float = 3.0,\n    sr_benchmark: float = 0.0,\n) -> dict:\n    \"\"\"\n    Compute the Deflated Sharpe Ratio.\n\n    Args:\n        observed_sr: Sharpe ratio of the selected strategy (annualized)\n        n_trials: Number of strategies/configurations tested\n        n_observations: Number of return observations\n        skewness: Skewness of returns\n        kurtosis: Kurtosis of returns (3 = normal)\n        sr_benchmark: Benchmark SR (default 0)\n\n    Returns:\n        dict with DSR p-value and expected max SR\n    \"\"\"\n    # Expected maximum SR under null hypothesis\n    euler_mascheroni = 0.5772156649\n    expected_max_sr = (\n        np.sqrt(2 * np.log(n_trials))\n        - (np.log(np.pi) + np.log(np.log(n_trials)) + 2 * euler_mascheroni)\n        / (2 * np.sqrt(2 * np.log(n_trials)))\n    )\n\n    # Variance of SR estimate\n    sr_var = (\n        1 + 0.5 * observed_sr**2\n        - skewness * observed_sr\n        + ((kurtosis - 3) / 4) * observed_sr**2\n    ) / (n_observations - 1)\n\n    sr_std = np.sqrt(sr_var)\n\n    # DSR: probability that observed SR exceeds expected max under null\n    dsr = norm.cdf((observed_sr - expected_max_sr) / sr_std)\n\n    return {\n        'deflated_sharpe_ratio': dsr,\n        'expected_max_sr_under_null': expected_max_sr,\n        'sr_std_error': sr_std,\n        'is_significant': dsr > 0.95,  # 95% confidence\n        'interpretation': (\n            f\"After testing {n_trials} strategies, observed SR of {observed_sr:.3f} \"\n            f\"has a {dsr:.1%} probability of being genuinely positive. \"\n            f\"Expected max SR from {n_trials} random strategies: {expected_max_sr:.3f}\"\n        )\n    }\n\n# Example usage:\n# You tested 50 feature combinations, best SR = 1.8\n# result = deflated_sharpe_ratio(\n#     observed_sr=1.8, n_trials=50, n_observations=2000,\n#     skewness=-0.3, kurtosis=5.0\n# )\n# If result['dsr'] < 0.95, your \"best\" strategy is likely data-mined.\n```\n\n**Key insight**: If you tried 50 configurations and the best has SR=1.5, the DSR will likely show it is not significant. You need SR > ~2.5 after 50 trials to have confidence.\n\n---\n\n## 5. White's Reality Check / Hansen's SPA Test\n\nThese are bootstrap-based tests for whether the best-performing strategy among many is genuinely better than a benchmark (typically buy-and-hold or zero return).\n\n### White's Reality Check (2000)\n\nTests H0: \"the best strategy has no predictive power\" against the alternative \"at least one strategy is genuinely profitable.\"\n\n### Hansen's Superior Predictive Ability (SPA) Test (2005)\n\nAn improvement over White's RC that has more power (fewer false negatives) by studentizing the test statistics and handling the \"least favorable configuration\" better.\n\n```python\ndef bootstrap_spa_test(\n    strategy_returns: np.ndarray,  # shape: (T, N) — T obs, N strategies\n    benchmark_returns: np.ndarray,  # shape: (T,) — benchmark returns\n    n_bootstrap: int = 1000,\n    block_size: int = 20,  # for stationary bootstrap\n) -> dict:\n    \"\"\"\n    Hansen's Superior Predictive Ability (SPA) test.\n    Tests if the best strategy is genuinely better than the benchmark.\n\n    Uses circular block bootstrap to preserve autocorrelation.\n    \"\"\"\n    T, N = strategy_returns.shape\n    excess = strategy_returns - benchmark_returns[:, np.newaxis]  # (T, N)\n    mean_excess = excess.mean(axis=0)  # (N,)\n\n    # Test statistic: max standardized excess return\n    std_excess = excess.std(axis=0, ddof=1) / np.sqrt(T)\n    std_excess[std_excess == 0] = 1e-10\n    test_stat = np.max(mean_excess / std_excess)\n\n    # Bootstrap distribution\n    boot_stats = []\n    for _ in range(n_bootstrap):\n        # Circular block bootstrap\n        boot_idx = _circular_block_bootstrap(T, block_size)\n        boot_excess = excess[boot_idx]\n        boot_mean = boot_excess.mean(axis=0) - mean_excess  # re-center under H0\n        boot_std = boot_excess.std(axis=0, ddof=1) / np.sqrt(T)\n        boot_std[boot_std == 0] = 1e-10\n        boot_stat = np.max(boot_mean / boot_std)\n        boot_stats.append(boot_stat)\n\n    boot_stats = np.array(boot_stats)\n    p_value = np.mean(boot_stats >= test_stat)\n\n    return {\n        'test_statistic': test_stat,\n        'p_value': p_value,\n        'is_significant': p_value < 0.05,\n        'best_strategy_idx': np.argmax(mean_excess),\n        'best_excess_return': np.max(mean_excess),\n    }\n\n\ndef _circular_block_bootstrap(T: int, block_size: int) -> np.ndarray:\n    \"\"\"Generate indices for circular block bootstrap.\"\"\"\n    n_blocks = int(np.ceil(T / block_size))\n    starts = np.random.randint(0, T, size=n_blocks)\n    indices = []\n    for s in starts:\n        block = np.arange(s, s + block_size) % T\n        indices.extend(block)\n    return np.array(indices[:T])\n```\n\n**Library reference**: `arch` package in Python has `SPA` and `RealityCheck` tests:\n```python\nfrom arch.bootstrap import SPA\nspa = SPA(benchmark_returns, strategy_returns, block_size=20)\nspa.compute()\nprint(spa.pvalue)  # consistent, upper, and lower p-values\n```\n\n---\n\n## 6. Feature Importance Stability\n\nA feature that is \"important\" in one time window but useless in the next is capturing noise, not signal.\n\n### 6.1 Stability Test via Walk-Forward Feature Importance\n\n```python\ndef feature_importance_stability(\n    X: pd.DataFrame,\n    y: pd.Series,\n    model_fn,\n    n_splits: int = 10,\n) -> pd.DataFrame:\n    \"\"\"\n    Compute feature importance in each walk-forward fold\n    and measure stability via rank correlation.\n    \"\"\"\n    from scipy.stats import spearmanr\n    import warnings\n\n    fold_size = len(X) // (n_splits + 1)\n    importances = []\n\n    for i in range(n_splits):\n        train_end = fold_size * (i + 2)\n        train_start = fold_size * i\n        if train_end > len(X):\n            break\n\n        X_train = X.iloc[train_start:train_end]\n        y_train = y.iloc[train_start:train_end]\n\n        model = model_fn()\n        model.fit(X_train, y_train)\n\n        if hasattr(model, 'feature_importances_'):\n            imp = model.feature_importances_\n        elif hasattr(model, 'coef_'):\n            imp = np.abs(model.coef_).flatten()\n        else:\n            raise ValueError(\"Model has no feature_importances_ or coef_\")\n\n        importances.append(imp)\n\n    imp_df = pd.DataFrame(\n        importances,\n        columns=X.columns,\n    )\n\n    # Stability metrics per feature\n    stability = pd.DataFrame({\n        'mean_importance': imp_df.mean(),\n        'std_importance': imp_df.std(),\n        'cv': imp_df.std() / (imp_df.mean() + 1e-10),  # coefficient of variation\n        'min_importance': imp_df.min(),\n        'max_importance': imp_df.max(),\n    })\n\n    # Rank stability: average pairwise Spearman correlation of feature rankings\n    ranks = imp_df.rank(axis=1)\n    rank_corrs = []\n    for i in range(len(ranks)):\n        for j in range(i + 1, len(ranks)):\n            with warnings.catch_warnings():\n                warnings.simplefilter(\"ignore\")\n                corr, _ = spearmanr(ranks.iloc[i], ranks.iloc[j])\n            if not np.isnan(corr):\n                rank_corrs.append(corr)\n\n    stability_score = np.mean(rank_corrs) if rank_corrs else 0\n\n    stability['is_stable'] = stability['cv'] < 1.0  # CV < 1 means relatively stable\n    stability = stability.sort_values('mean_importance', ascending=False)\n\n    return stability, stability_score\n\n\n# Interpretation:\n# stability_score > 0.7: Feature rankings are stable — good\n# stability_score 0.4-0.7: Moderate stability — concerning\n# stability_score < 0.4: Features are essentially random — severe overfitting risk\n```\n\n### 6.2 Mean Decrease Accuracy (MDA) with Permutation\n\nMore robust than model-native feature importance. Permute one feature at a time and measure the drop in out-of-sample performance:\n\n```python\nfrom sklearn.inspection import permutation_importance\n\n# Use on OUT-OF-SAMPLE data only — never on training data\nresult = permutation_importance(\n    model, X_test, y_test,\n    n_repeats=30,\n    random_state=42,\n    scoring='neg_log_loss'  # or appropriate metric\n)\n# Features where result.importances_mean > 0 are genuinely useful\n# Features where the confidence interval crosses 0 are noise\n```\n\n### 6.3 Drop-Column Importance\n\nThe gold standard (but expensive): retrain the model without each feature and measure out-of-sample performance change. If removing a feature doesn't hurt (or helps), it's noise.\n\n---\n\n## 7. Regularization Techniques for Financial Models\n\n### 7.1 L1/L2 for XGBoost — Crypto-Specific Ranges\n\n```python\n# XGBoost regularization parameters for crypto\nxgb_params = {\n    'reg_alpha': 0.1,        # L1 regularization. Range: [0.01, 10.0]\n                              # Higher for crypto (noisy features). Start at 0.1.\n    'reg_lambda': 1.0,       # L2 regularization. Range: [0.1, 10.0]\n                              # Default 1.0 is often fine. You want *much* stronger regularization than typical ML applications. Prefer underfitting slightly (low variance) over high-capacity models.\n\n### 7.2 Dropout for GRU — Rates and Spatial Dropout\n\n```python\nimport torch.nn as nn\n\nclass RegularizedGRU(nn.Module):\n    def __init__(\n        self,\n        input_size: int,\n        hidden_size: int = 64,\n        num_layers: int = 2,\n        dropout_rate: float = 0.3,      # Range: [0.2, 0.5] for crypto\n        spatial_dropout_rate: float = 0.1,  # Range: [0.05, 0.2]\n    ):\n        super().__init__()\n\n        # Spatial dropout: drops entire feature channels\n        # This forces the model to not rely on any single feature\n        self.spatial_dropout = nn.Dropout2d(spatial_dropout_rate)\n\n        self.gru = nn.GRU(\n            input_size=input_size,\n            hidden_size=hidden_size,\n            num_layers=num_layers,\n            dropout=dropout_rate if num_layers > 1 else 0,  # between GRU layers\n            batch_first=True,\n        )\n\n        # Dropout before final layer\n        self.dropout = nn.Dropout(dropout_rate)\n        self.fc = nn.Linear(hidden_size, 1)\n\n    def forward(self, x):\n        # x shape: (batch, seq_len, features)\n\n        # Spatial dropout: treat features as \"channels\"\n        # Reshape to (batch, features, seq_len) for Dropout2d\n        x = x.permute(0, 2, 1).unsqueeze(-1)  # (batch, features, seq_len, 1)\n        x = self.spatial_dropout(x)\n        x = x.squeeze(-1).permute(0, 2, 1)  # back to (batch, seq_len, features)\n\n        out, _ = self.gru(x)\n        out = out[:, -1, :]  # last time step\n        out = self.dropout(out)\n        return self.fc(out)\n```\n\n**Dropout rate recommendations for crypto:**\n- **Input dropout (spatial)**: 0.05-0.2. Drops entire features for a sample.\n- **Recurrent dropout**: 0.2-0.3. Between GRU layers.\n- **Output dropout**: 0.3-0.5. Before the final dense layer.\n- **Increase dropout** if train/val gap is large. **Decrease** if underfitting.\n\n### 7.3 Early Stopping — Optimal Patience\n\n```python\n# XGBoost early stopping\nmodel = xgb.XGBClassifier(**xgb_params)\nmodel.fit(\n    X_train, y_train,\n    eval_set=[(X_val, y_val)],\n    # For 5-min crypto: patience of 30-50 rounds with lr=0.01\n    # This means ~30-50 trees without improvement before stopping\n    early_stopping_rounds=50,\n    verbose=False,\n)\n\n# PyTorch early stopping\nclass EarlyStopping:\n    def __init__(self, patience: int = 20, min_delta: float = 1e-4):\n        \"\"\"\n        For GRU on crypto:\n        - patience: 15-30 epochs\n        - min_delta: 1e-4 to 1e-3\n        \"\"\"\n        self.patience = patience\n        self.min_delta = min_delta\n        self.counter = 0\n        self.best_loss = float('inf')\n        self.should_stop = False\n\n    def step(self, val_loss: float) -> bool:\n        if val_loss < self.best_loss - self.min_delta:\n            self.best_loss = val_loss\n            self.counter = 0\n        else:\n            self.counter += 1\n            if self.counter >= self.patience:\n                self.should_stop = True\n        return self.should_stop\n```\n\n### 7.4 Weight Decay for Adam\n\n```python\n# AdamW is preferred over Adam + L2 regularization\n# They are NOT equivalent (Loshchilov & Hutter, 2019)\noptimizer = torch.optim.AdamW(\n    model.parameters(),\n    lr=1e-3,           # Range: [1e-4, 5e-3]\n    weight_decay=1e-2,  # Range: [1e-3, 1e-1] for crypto\n                        # Higher than typical — crypto features are noisy\n    betas=(0.9, 0.999),\n)\n\n# Learning rate schedule: cosine annealing works well\nscheduler = torch.optim.lr_scheduler.CosineAnnealingWarmRestarts(\n    optimizer,\n    T_0=10,     # restart every 10 epochs\n    T_mult=2,   # double period after each restart\n)\n```\n\n---\n\n## 8. Data Augmentation for Financial Time Series\n\n**The honest answer: data augmentation for financial time series is controversial and risky.**\n\n### 8.1 Techniques That May Help (Use With Caution)\n\n**Noise injection (jittering)**:\n```python\ndef add_gaussian_noise(X: np.ndarray, sigma: float = 0.001) -> np.ndarray:\n    \"\"\"Add small Gaussian noise to features. sigma should be << feature std.\"\"\"\n    noise = np.random.normal(0, sigma, X.shape)\n    return X + noise\n```\n- Can help as *regularization* (similar to dropout) if sigma is very small\n- sigma should be 0.1-1% of each feature's standard deviation\n- **Never** apply to the target variable\n\n**Window slicing**: Use slightly offset windows of the same data for additional training samples. This is nearly free and low risk.\n\n### 8.2 Techniques That Probably Hurt\n\n- **Time warping**: Stretching/compressing time. Destroys temporal patterns that may be real signals.\n- **Magnitude warping**: Scaling features. Can create unrealistic price dynamics.\n- **SMOTE / synthetic oversampling**: Creates synthetic samples by interpolating between real ones. In time series, interpolated samples have no financial meaning.\n- **GANs for synthetic data**: Promising in theory, but in practice GANs for financial data tend to learn the noise, not the signal.\n\n### 8.3 Better Alternative: Regime-Aware Sampling\n\nInstead of augmenting, **oversample underrepresented regimes**:\n\n```python\ndef regime_balanced_sampling(\n    X: pd.DataFrame,\n    y: pd.Series,\n    regime_labels: pd.Series,  # e.g., \"bull\", \"bear\", \"range\"\n    target_samples_per_regime: int = None,\n) -> tuple:\n    \"\"\"\n    Resample so each regime is equally represented in training.\n    This prevents the model from only learning the dominant regime.\n    \"\"\"\n    if target_samples_per_regime is None:\n        target_samples_per_regime = regime_labels.value_counts().min()\n\n    indices = []\n    for regime in regime_labels.unique():\n        regime_idx = regime_labels[regime_labels == regime].index\n        if len(regime_idx) >= target_samples_per_regime:\n            sampled = np.random.choice(\n                regime_idx, target_samples_per_regime, replace=False\n            )\n        else:\n            sampled = np.random.choice(\n                regime_idx, target_samples_per_regime, replace=True\n            )\n        indices.extend(sampled)\n\n    # Maintain time ordering\n    indices = sorted(indices)\n    return X.loc[indices], y.loc[indices]\n```\n\n---\n\n## 9. Stationarity Requirements\n\n### 9.1 Which Features Need Stationarity\n\n| Feature Type | Stationary? | Treatment |\n|---|---|---|\n| Raw price | No | Use log returns or fractional diff |\n| Volume | Often non-stationary | Use log(volume) z-scored over rolling window |\n| RSI, Stochastic | Bounded [0,100] — approximately stationary | Usually fine as-is |\n| MACD | Depends on scale | Normalize by ATR or rolling std |\n| Bollinger %B | Bounded — approximately stationary | Usually fine |\n| Order book depth | Non-stationary | Z-score with rolling window |\n| Funding rate | Approximately stationary | Usually fine |\n| Open interest | Non-stationary | Use rate of change |\n| Dollar volume | Non-stationary | Log, then z-score with rolling window |\n| Volatility (ATR) | Non-stationary (trends) | Use ratio to longer-term vol |\n\n### 9.2 How to Make Features Stationary\n\n```python\ndef make_stationary(series: pd.Series, method: str = 'returns') -> pd.Series:\n    \"\"\"Transform a series to be (approximately) stationary.\"\"\"\n    if method == 'returns':\n        return series.pct_change()\n    elif method == 'log_returns':\n        return np.log(series / series.shift(1))\n    elif method == 'diff':\n        return series.diff()\n    elif method == 'z_score':\n        # Rolling z-score — always use expanding or rolling, NEVER full-sample\n        rolling_mean = series.rolling(window=288).mean()  # 24h for 5-min bars\n        rolling_std = series.rolling(window=288).std()\n        return (series - rolling_mean) / (rolling_std + 1e-10)\n    elif method == 'frac_diff':\n        d = find_min_d(series)\n        return frac_diff(series, d)\n    else:\n        raise ValueError(f\"Unknown method: {method}\")\n```\n\n### 9.3 Log Returns vs Raw Returns\n\n- **Log returns** are preferred for financial ML because:\n  - More normally distributed\n  - Additive over time (log_return[0:T] = sum of log_return[t])\n  - Symmetric: +10% and -10% have equal magnitude\n- **Raw (simple) returns** are needed when computing actual PnL for backtesting\n- **Rule**: Use log returns for *features*, simple returns for *PnL calculation*\n\n---\n\n## 10. Transaction Costs as Regularization\n\nIncluding realistic transaction costs in the loss function is one of the most effective (and underused) anti-overfitting techniques.\n\n**Why it works**: A model that captures only small, noise-driven edges will be unprofitable after costs. Only genuine signals survive.\n\n### 10.1 Realistic Crypto Transaction Costs (BTC on major exchanges, 2024)\n\n| Cost Component | Typical Range |\n|---|---|\n| Taker fee | 0.04-0.10% |\n| Maker fee | 0.00-0.02% |\n| Slippage (100K USD order) | 0.01-0.05% |\n| Funding rate (perps) | -0.01% to +0.03% per 8h |\n| **Total round-trip** | **0.05-0.15%** |\n\n### 10.2 Cost-Aware Loss Function\n\n```python\ndef cost_aware_loss(\n    predictions: torch.Tensor,  # predicted returns or probabilities\n    targets: torch.Tensor,      # actual returns\n    positions: torch.Tensor,    # model's position sizing\n    prev_positions: torch.Tensor,  # previous position\n    cost_bps: float = 10.0,    # round-trip cost in basis points\n) -> torch.Tensor:\n    \"\"\"\n    Loss function that penalizes trades proportional to transaction costs.\n    \"\"\"\n    cost_rate = cost_bps / 10000.0\n\n    # PnL from position\n    gross_pnl = positions * targets\n\n    # Transaction cost: proportional to position change\n    turnover = torch.abs(positions - prev_positions)\n    transaction_cost = turnover * cost_rate\n\n    net_pnl = gross_pnl - transaction_cost\n\n    # Negative Sharpe-like objective (we minimize)\n    loss = -net_pnl.mean() / (net_pnl.std() + 1e-8)\n    return loss\n```\n\n### 10.3 Cost Penalty in XGBoost Custom Objective\n\n```python\ndef cost_penalized_objective(\n    y_pred: np.ndarray,\n    dtrain,\n    cost_bps: float = 10.0,\n) -> tuple:\n    \"\"\"Custom XGBoost objective that includes transaction costs.\"\"\"\n    y_true = dtrain.get_label()\n    cost_rate = cost_bps / 10000.0\n\n    # Sigmoid to get position size\n    position = 1.0 / (1.0 + np.exp(-y_pred))\n\n    # Gradient of cost-penalized logloss\n    # This effectively shrinks predictions toward 0.5 (no trade)\n    # by an amount proportional to transaction costs\n    grad = position - y_true + cost_rate * np.sign(position - 0.5)\n    hess = position * (1 - position) + cost_rate\n\n    return grad, hess\n```\n\n**Key insight**: When you include 10bps round-trip costs, a signal needs to predict >10bps moves consistently to be useful. This eliminates most noise-driven signals automatically.\n\n---\n\n## 11. Out-of-Sample Testing Protocols\n\n### 11.1 Data Split Ratios\n\nFor crypto at 5-min resolution with ~2 years of data (~210K bars):\n\n```\n|--- Development (80%) ---|--- Final Test (20%) ---|\n|                          |                        |\n|  Train (70%)  |Val(10%) |   NEVER TOUCH           |\n|               |          |   UNTIL FINAL            |\n|               |          |   EVALUATION              |\n```\n\n- **Training**: 70% (~1.4 years). Used for model fitting.\n- **Validation**: 10% (~2.5 months). Used for hyperparameter tuning, feature selection, early stopping.\n- **Final test**: 20% (~5 months). Touched ONCE for final evaluation. If you look at test results and go back to change the model, the test set is contaminated.\n\n### 11.2 The \"Touched the Test Set\" Problem\n\n**Rule**: Once you look at test set performance, ANY change to the model invalidates the test set.\n\n**Solutions**:\n1. **Multiple test sets**: Hold out 3 non-overlapping test periods. Use the first for initial evaluation, second for verification, third as the absolute final check.\n2. **Sealed test set**: Have a collaborator (or automated pipeline) run the final test. You never see intermediate results.\n3. **Paper trading as the real test**: The final test set is meaningless anyway because the real question is \"does it work on future, unseen data?\" Paper trading answers this directly.\n\n### 11.3 Rolling Out-of-Sample Evaluation\n\n```python\ndef rolling_oos_evaluation(\n    X, y, model_fn,\n    train_months: int = 12,\n    test_months: int = 1,\n    step_months: int = 1,\n) -> pd.DataFrame:\n    \"\"\"\n    Evaluate model on rolling 1-month out-of-sample windows.\n    Each window is genuinely unseen at the time of training.\n    \"\"\"\n    results = []\n    bars_per_month = 30 * 24 * 12  # ~8640 bars per month at 5-min\n\n    total_months = len(X) // bars_per_month\n    for start_month in range(0, total_months - train_months - test_months + 1, step_months):\n        train_start = start_month * bars_per_month\n        train_end = (start_month + train_months) * bars_per_month\n        test_start = train_end\n        test_end = min(test_start + test_months * bars_per_month, len(X))\n\n        model = model_fn()\n        model.fit(X.iloc[train_start:train_end], y.iloc[train_start:train_end])\n        preds = model.predict(X.iloc[test_start:test_end])\n\n        month_result = evaluate_predictions(\n            y.iloc[test_start:test_end], preds\n        )\n        month_result['period_start'] = X.index[test_start]\n        month_result['period_end'] = X.index[test_end - 1]\n        results.append(month_result)\n\n    return pd.DataFrame(results)\n```\n\n---\n\n## 12. Survivorship-Free Crypto Data\n\n### 13.1 Does This Matter for BTC-Only Prediction?\n\n**Less than for altcoin portfolios, but it still matters:**\n\n- **Exchange survivorship**: BTC/USDT on FTX had different microstructure than on Binance. Training on BTC/USDT without accounting for this creates false patterns.\n\n### 13.2 Practical Mitigation\n\nFor BTC-only prediction:\n1. Use BTC index prices (average across exchanges) rather than single-exchange data\n2. Include exchange-specific basis as a feature rather than ignoring it\n3. Flag and handle stablecoin depeg periods explicitly\n4.
