Dynamic Stops Based on Model Confidence\n\n### The Core Idea\n\nNot all trades are equal. A signal with 85% model confidence deserves more room to breathe than a signal with 55% confidence. The stop distance should scale with the quality of the signal.\n\n### Confidence-Weighted Stop Distance\n\n**Formula:**\n```\nstop_distance = base_stop × (1 + alpha × (confidence - threshold))\n```\n\nWhere:\n- `base_stop` = default stop distance (e.g., 2.0 ATR)\n- `alpha` = scaling factor (how much confidence affects stop width)\n- `confidence` = model's predicted probability (0-1)\n- `threshold` = minimum confidence to trade (e.g., 0.55)\n\n**Example calibration:**\n\n| Model Confidence | Stop Multiplier (alpha=2) | Effective Stop (if base=2.0 ATR) |\n|---|---|---|\n| 0.55 (minimum) | 2.0 × (1 + 2×(0.55-0.55)) = 2.0 ATR | 2.0 ATR |\n| 0.60 | 2.0 × (1 + 2×0.05) = 2.2 ATR | 2.2 ATR |\n| 0.65 | 2.0 × (1 + 2×0.10) = 2.4 ATR | 2.4 ATR |\n| 0.70 | 2.0 × (1 + 2×0.15) = 2.6 ATR | 2.6 ATR |\n| 0.80 | 2.0 × (1 + 2×0.25) = 3.0 ATR | 3.0 ATR |\n| 0.90 | 2.0 × (1 + 2×0.35) = 3.4 ATR | 3.4 ATR |\n\n**Implementation:**\n```python\n@dataclass\nclass DynamicStop:\n    base_multiplier: float = 2.0\n    alpha: float = 2.0\n    confidence_threshold: float = 0.55\n    min_multiplier: float = 1.5\n    max_multiplier: float = 3.5\n\n    def calculate_multiplier(self, confidence: float) -> float:\n        \"\"\"Scale stop distance with model confidence.\"\"\"\n        raw = self.base_multiplier * (\n            1 + self.alpha * (confidence - self.confidence_threshold)\n        )\n        return max(self.min_multiplier, min(self.max_multiplier, raw))\n\n    def stop_price(\n        self,\n        entry_price: float,\n        atr: float,\n        confidence: float,\n        direction: str\n    ) -> float:\n        multiplier = self.calculate_multiplier(confidence)\n        distance = multiplier * atr\n        if direction == 'long':\n            return entry_price - distance\n        return entry_price + distance\n```\n\n### Also: Confidence-Weighted Position Sizing Interaction\n\nDynamic stops interact with position sizing. If you widen the stop, you should reduce position size to keep risk constant:\n\n```\nposition_size = risk_budget / stop_distance\n```\n\nA wider stop means a smaller position but the same dollar risk. This is important: confidence-based stops work best when combined with inverse position sizing.\n\n```python\ndef position_size_with_dynamic_stop(\n    account_equity: float,\n    risk_per_trade_pct: float,  # e.g., 0.01 (1%)\n    entry_price: float,\n    stop_price: float\n) -> float:\n    \"\"\"Position size that keeps dollar risk constant regardless of stop width.\"\"\"\n    risk_amount = account_equity * risk_per_trade_pct\n    stop_distance = abs(entry_price - stop_price)\n    if stop_distance == 0:\n        return 0.0\n    return risk_amount / stop_distance\n```\n\n---\n\n## 5. Volatility-Adjusted Stops\n\n### Why Volatility Adjustment Is Essential\n\nBTC's 5-minute volatility varies by 5-10x depending on:\n- **Time of day:** US market hours (13:00-21:00 UTC) are 2-3x more volatile than Asian overnight.\n- **Day of week:** Weekends historically less volatile (though this has changed).\n- **Market regime:** Post-FOMC, post-CPI, post-halving periods can see sustained high volatility.\n- **Liquidation cascades:** Sudden spikes in volatility when large positions are force-liquidated.\n\nA fixed stop ignores all of this. A 1% stop might be 8 ATR during a quiet Sunday night (way too wide) and 0.7 ATR during a Fed announcement (way too tight).\n\n### Implementation: Volatility Regime Classification\n\n```python\nfrom enum import Enum\nfrom dataclasses import dataclass\n\nclass VolRegime(Enum):\n    LOW = \"low\"\n    NORMAL = \"normal\"\n    HIGH = \"high\"\n    EXTREME = \"extreme\"\n\n@dataclass\nclass VolatilityAdjustedStop:\n    # ATR percentile thresholds (calibrated on rolling 24h of 5-min bars)\n    low_threshold: float = 25.0      # below 25th percentile\n    high_threshold: float = 75.0     # above 75th percentile\n    extreme_threshold: float = 95.0  # above 95th percentile\n\n    # Stop multipliers per regime\n    multipliers: dict = None\n\n    def __post_init__(self):\n        if self.multipliers is None:\n            self.multipliers = {\n                VolRegime.LOW: 2.5,      # wider relative to ATR (ATR is small)\n                VolRegime.NORMAL: 2.0,   # standard\n                VolRegime.HIGH: 1.5,     # tighter relative to ATR (ATR is large)\n                VolRegime.EXTREME: 1.0,  # very tight; get out fast if wrong\n            }\n\n    def classify_regime(\n        self, current_atr: float, atr_history: np.ndarray\n    ) -> VolRegime:\n        percentile = (\n            np.sum(atr_history < current_atr) / len(atr_history)\n        ) * 100\n\n        if percentile >= self.extreme_threshold:\n            return VolRegime.EXTREME\n        elif percentile >= self.high_threshold:\n            return VolRegime.HIGH\n        elif percentile <= self.low_threshold:\n            return VolRegime.LOW\n        return VolRegime.NORMAL\n\n    def stop_distance(\n        self, current_atr: float, atr_history: np.ndarray\n    ) -> float:\n        regime = self.classify_regime(current_atr, atr_history)\n        return self.multipliers[regime] * current_atr\n```\n\n**Key nuance in the multiplier table above:** In low-vol regimes, you use a *larger multiplier* (2.5 ATR) because the ATR itself is small, so the absolute stop distance is still tight. A 2.0 ATR stop with a 1:1 take-profit means you need a 2.0 ATR move in your favor — achievable but not common within 1-3 bars.\n\n### Dynamic Take-Profit Based on Quantile Regression\n\nInstead of a fixed ratio, have the model predict the distribution of future returns and set take-profit at a specific quantile.\n\n**Approach:**\n1. Train a quantile regression model (or use quantile loss in your neural network) to predict the 75th and 90th percentile of the next-N-bar return distribution.\n2. Set take-profit at the predicted 75th percentile (conservative) or 90th percentile (aggressive).\n\n```python\n@dataclass\nclass QuantileTakeProfit:\n    \"\"\"Take-profit based on predicted return distribution.\"\"\"\n    conservative_quantile: float = 0.75  # exit at predicted 75th percentile\n    aggressive_quantile: float = 0.90    # exit at predicted 90th percentile\n\n    def take_profit_price(\n        self,\n        entry_price: float,\n        predicted_return_75: float,  # from quantile regression\n        predicted_return_90: float,\n        confidence: float,\n        direction: str\n    ) -> float:\n        \"\"\"\n        High confidence → use aggressive (90th percentile) target.\n        Low confidence → use conservative (75th percentile) target.\n        \"\"\"\n        # Blend based on confidence\n        blend = min(1.0, max(0.0, (confidence - 0.55) / 0.35))\n        target_return = (\n            predicted_return_75 * (1 - blend)\n            + predicted_return_90 * blend\n        )\n\n        if direction == 'long':\n            return entry_price * (1 + target_return)\n        return entry_price * (1 - target_return)\n```\n\n### Scaling Out\n\nExit in tranches to capture both the \"sure thing\" and the \"maybe more\" scenarios.\n\n**Common approach:**\n1. Exit 50% of position at 1:1 risk-reward.\n2. Move stop to break-even on remaining 50%.\n3. Exit remaining 50% at 2:1 or on time expiry.\n\n**Does this work?** Mathematically, scaling out is identical to running two separate half-size trades with different take-profit levels. It doesn't create edge — but it has a psychological and practical benefit: it reduces the variance of outcomes and makes the equity curve smoother.\n\n**For ML systems specifically:** Scaling out has limited value because the model should provide a continuous signal. If the model says \"still bullish,\" why exit? If it says \"no longer bullish,\" why keep any position? The model's signal should drive the exit, not a mechanical partial-exit rule.\n\n**When scaling out IS useful:** When you're using the multi-level exit system (Section 9) and want the time-based exit to only close part of a winning position while letting the trailing stop handle the rest.\n\n### Does Take-Profit Help or Hurt ML Systems?\n\n**The uncomfortable truth:** For most ML-based 5-minute systems, a fixed take-profit HURTS performance.\n\n**Why:**\n1. The model's prediction is probabilistic. It says \"60% chance of going up.\" It doesn't say \"it will go up exactly 0.5%.\"\n2. A take-profit at 1:1 exits at an arbitrary level that has no relationship to the model's prediction.\n3. The best exits for ML systems are based on the model's updated prediction (signal reversal) or time expiry (signal decay), not on a price target.\n\n**Exception:** Take-profit IS useful as a **secondary exit** for very profitable trades where the model hasn't reversed but the gain is already larger than the model's typical prediction accuracy would justify. Think of it as: \"This trade has already exceeded what the model predicted; take the win.\"\n\n```python\ndef should_take_exceptional_profit(\n    current_pnl_atr: float,  # current P&L in ATR units\n    typical_win_atr: float = 1.5,  # historical avg winning trade in ATR\n    exceptional_multiplier: float = 3.0\n) -> bool:\n    \"\"\"Exit if profit exceeds 3x the typical winning trade.\"\"\"\n    return current_pnl_atr > typical_win_atr * exceptional_multiplier\n```\n\n---\n\n## 7. The No-Stop Approach\n\n### When It Works\n\nSome ML-based crypto trading systems operate without traditional stop-losses, relying entirely on the model to signal entry and exit. This can work when:\n\n1. **The model generates frequent signals.** If the model re-evaluates every bar (5 min), and the maximum adverse excursion before the model reverses is typically small (< 1%), then the model itself acts as a \"smart stop.\"\n2. **The model has been trained on adverse scenarios.** If the training data includes flash crashes and the model learned to exit during them, it may handle tail events.\n3. **Position sizing is very conservative.** If you're risking 0.1% of equity per trade instead of 1%, even a 5% adverse move only costs 0.5% of equity.\n4. **The market is liquid and continuous.** No gaps, no exchange downtime, no slippage. (This is increasingly unrealistic for crypto.)\n\n### When It Blows Up\n\n**The catastrophic failure modes:**\n\n1. **Flash crashes.** On May 19, 2021, BTC dropped ~30% in hours with a ~15% move in under 30 minutes. On multiple occasions, 5-minute candles have shown 3-5% moves. If your model can't predict these (and no model can reliably predict flash crashes), a single event can destroy a year of profits.\n\n2. **Exchange issues.** The exchange API goes down. Your model says \"exit\" but you can't execute. By the time connectivity returns, you're down 5%. This has happened on every major exchange multiple times.\n\n3. **Liquidation cascades.** In leveraged markets, a 2% move can trigger leveraged liquidations, causing a 5% move, causing more liquidations. This feedback loop is impossible to model and can happen in a single 5-minute bar.\n\n4. **Model degradation.** ML models degrade over time as market microstructure changes. The model might have been accurate last month but is now making random predictions. Without stops, the degradation manifests as steadily accumulating losses.\n\n**Historical examples of \"no-stop\" failures in crypto:**\n- The March 2020 \"COVID crash\" saw BTC drop 40% in 24 hours, with multiple 5%+ 5-minute candles.\n- FTX collapse in Nov 2022: extreme dislocations across all crypto markets.\n- Multiple exchange-specific flash crashes (Binance LUNA wicks, KuCoin hacks, etc.).\n\n### The Hybrid Recommendation\n\n**NEVER run without some form of catastrophic stop.** The expected cost of the catastrophic stop (occasionally getting stopped out unnecessarily) is far less than the expected cost of the tail event it prevents.\n\n```python\n@dataclass\nclass HybridStopPolicy:\n    \"\"\"\n    No traditional stop for normal operation.\n    Catastrophic stop for black swan protection.\n    \"\"\"\n    catastrophic_stop_pct: float = 0.03  # 3% max loss per trade\n    max_loss_per_bar_pct: float = 0.02   # if single bar moves >2% against, exit\n\n    def should_catastrophic_exit(\n        self,\n        entry_price: float,\n        current_price: float,\n        direction: str,\n        bar_low: float = None,  # for longs\n        bar_high: float = None,  # for shorts\n    ) -> tuple[bool, str]:\n        # Check total adverse excursion\n        if direction == 'long':\n            pnl_pct = (current_price - entry_price) / entry_price\n            if pnl_pct < -self.catastrophic_stop_pct:\n                return True, \"catastrophic_stop\"\n            # Check single-bar adverse move\n            if bar_low is not None:\n                bar_move = (bar_low - entry_price) / entry_price\n                if bar_move < -self.max_loss_per_bar_pct:\n                    return True, \"single_bar_catastrophic\"\n        else:\n            pnl_pct = (entry_price - current_price) / entry_price\n            if pnl_pct < -self.catastrophic_stop_pct:\n                return True, \"catastrophic_stop\"\n            if bar_high is not None:\n                bar_move = (bar_high - entry_price) / entry_price\n                if bar_move > self.max_loss_per_bar_pct:\n                    return True, \"single_bar_catastrophic\"\n\n        return False, \"no_exit\"\n```\n\n---\n\n## 8. Stop-Loss and Transaction Costs\n\n### The Cost-Turnover Tradeoff\n\nTighter stops mean more frequent stop-outs, which means more round-trip trades, which means more transaction costs.\n\n**Crypto transaction cost components:**\n| Component | Maker | Taker | Notes |\n|---|---|---|---|\n| Exchange fee | 0.01-0.02% | 0.04-0.06% | Binance/Bybit VIP tiers |\n| Spread cost | 0.01-0.03% | 0.01-0.03% | BTC/USDT on major exchanges |\n| Slippage | 0.01-0.05% | 0.03-0.10% | Depends on size and volatility |\n| **Total round-trip** | **0.04-0.10%** | **0.10-0.25%** | |\n\n**Stop-loss orders are typically taker orders** (market orders when triggered), so assume 0.10-0.25% per round-trip for stop-triggered exits.\n\n### Break-Even Analysis\n\nIf your model has a per-trade edge of E (expected profit per trade before costs), and the round-trip cost is C, then:\n\n```\nNet edge = E - C × (1 + additional_turnover_from_stops)\n```\n\n**Example:**\n- Model edge per trade: 0.15% (realistic for a good 5-min ML model)\n- Round-trip cost: 0.12% (mid-range for taker)\n- Net edge without stops: 0.15% - 0.12% = 0.03%\n- With tight stops (0.5%) that trigger 40% of the time: each stop-out adds another round-trip cost of 0.12%\n- Net edge with tight stops: 0.15% - 0.12% - (0.40 × 0.12%) = 0.15% - 0.168% = **-0.018%** (NEGATIVE)\n\n**This is the key insight:** Tight stops can make a profitable strategy unprofitable purely through increased transaction costs.\n\n### Cost-Adjusted Optimal Stop Distance\n\nThe optimal stop distance minimizes:\n```\nTotal cost = (loss on stopped-out trades) + (opportunity cost of missed winners) + (transaction cost of additional round-trips)\n```\n\n**In practice, this means:**\n1. **Don't use stops tighter than 1.5 ATR** unless your transaction costs are exceptionally low (maker orders, high VIP tier).\n2. **Every stop-out costs ~0.12-0.25% in additional round-trip costs.** If your stop triggers on 30% of trades, that's 0.04-0.08% added to your average cost per trade.\n3. **The break-even stop rate** (where cost of stops equals their benefit) is approximately:\n\n```python\ndef max_profitable_stop_rate(\n    model_edge_pct: float,      # e.g., 0.15%\n    round_trip_cost_pct: float,  # e.g., 0.12%\n    avg_stop_loss_savings_pct: float  # avg loss prevented by stop, e.g., 0.3%\n) -> float:\n    \"\"\"\n    Maximum stop-trigger rate before costs exceed savings.\n    Returns: max fraction of trades that can be stopped out profitably.\n    \"\"\"\n    if avg_stop_loss_savings_pct <= round_trip_cost_pct:\n        return 0.0  # stops never pay for themselves\n    return (model_edge_pct) / round_trip_cost_pct\n```\n\n---\n\n## 9. Multi-Level Stop System (Recommended Architecture)\n\nThis is the synthesis of everything above into a practical, production-ready system.\n\n### The Four Levels\n\n```\nPriority (highest to lowest):\n  Level 4: Catastrophic stop     → ALWAYS active, non-negotiable\n  Level 3: Volatility-ATR stop   → Price-based safety net\n  Level 2: Time stop             → Signal decay protection\n  Level 1: Model signal reversal → Primary exit mechanism\n```\n\n**Execution priority:** Higher-numbered levels override lower ones. Backtesting Stop-Loss Strategies Accurately\n\n### The Intra-Bar Problem\n\nThe single biggest pitfall in backtesting stop-loss strategies on 5-minute bars: **you don't know what happened within the bar.**\n\n**Example:**\n- Bar open: $40,000\n- Bar high: $40,150\n- Bar low: $39,850\n- Bar close: $40,100\n- Your stop: $39,900 (long)\n\n**Was your stop hit?** The low was $39,850, which is below your stop at $39,900. So yes, the stop was hit. But when? Did price drop to $39,850 first and then rally to $40,150? Or did it rally first and then drop? The order matters enormously:\n\n- **Scenario A:** Price drops to $39,850 first → stop triggers at $39,900 → you miss the rally to $40,150. You exited with a $100 loss.\n- **Scenario B:** Price rallies to $40,150 first → then drops to $39,850 → stop triggers at $39,900. You exited with a $100 loss, but you would have also been stopped out even with a trailing stop.\n\n**Without tick data or at least 1-minute sub-bars, you cannot distinguish these scenarios.**\n\n### Best Practices for Stop-Loss Backtesting\n\n**1. Why \"assume stop hit first\" is correct for backtesting:**\n- It produces conservative (pessimistic) results.\n- If the strategy is profitable under this assumption, it will perform at least as well in live trading.\n- The alternative (assuming TP hit first) produces optimistic bias that will disappoint in production.\n\n**3.
