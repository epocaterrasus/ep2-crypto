Elevated funding regimes last 2-5 days on average before mean-reverting.\n\nHalf-life of funding rate shocks: approximately 2.5 settlements (20 hours).\n\n### 2.5 Seasonality / Intraday Patterns\n\n**By settlement time (UTC)**:\n\n| Settlement | Mean Rate | Std Dev  | Notes |\n|------------|-----------|----------|-------|\n| 00:00 UTC  | +0.0062%  | 0.021%   | Slightly higher (Asia session end) |\n| 08:00 UTC  | +0.0055%  | 0.019%   | Lowest average (Europe open) |\n| 16:00 UTC  | +0.0054%  | 0.020%   | US session |\n\nThe differences are small and not statistically significant for trading purposes.\n\n**Day of week**:\n- Weekends: slightly lower absolute funding (less speculative activity)\n- Monday/Tuesday: funding tends to be more volatile (catching up to weekend moves)\n- Friday: slightly elevated funding (weekend positioning)\n\n---\n\n## 3. Funding Rate as a Cost for Our System\n\n### 3.1 Expected Exposure Per Trade\n\nOur system's average holding period is 5-30 minutes (1-6 bars at 5-min timeframe). The probability of spanning a funding settlement:\n\n```\nSettlement probability = holding_duration_minutes / (8 * 60)\n```\n\n| Holding Duration | P(spanning settlement) | Expected settlements/trade |\n|------------------|----------------------:|---------------------------:|\n| 5 min (1 bar)    | 1.04%                 | 0.0104                     |\n| 10 min (2 bars)  | 2.08%                 | 0.0208                     |\n| 15 min (3 bars)  | 3.13%                 | 0.0313                     |\n| 30 min (6 bars)  | 6.25%                 | 0.0625                     |\n| 60 min (12 bars) | 12.50%               | 0.1250                     |\n\nFor the typical trade (10 min hold): 2% chance of hitting a funding settlement.\n\n### 3.2 Expected Funding Cost Per Trade\n\n**Average case**:\n```\nExpected cost = P(spanning) * |mean_funding_rate|\n             = 0.0208 * 0.0057%\n             = 0.000119% per trade\n             = 0.012 bps per trade\n```\n\nThis is negligible compared to the 4-6 bps exchange fee.\n\n**But this is misleading** — the problem is not the average, it is the tail.\n\n### 3.3 Worst-Case Funding Cost Per Trade\n\nIf you happen to hold through a settlement during an extreme funding episode:\n\n| Scenario              | Funding Rate | Cost (long position) |\n|-----------------------|-------------|---------------------|\n| Normal                | +0.010%     | 1.0 bps             |\n| Elevated (bull)       | +0.035%     | 3.5 bps             |\n| High (euphoria)       | +0.100%     | 10.0 bps            |\n| Extreme               | +0.300%     | 30.0 bps            |\n\nAt 30 bps, a single funding payment exceeds the entire round-trip transaction cost (8-12 bps). This can destroy the edge on a single trade.\n\n### 3.4 Annual Funding Drag\n\nAssuming 15-20 trades per day, 10-minute average hold, with no avoidance logic:\n\n```python\ntrades_per_day = 18\np_span_funding = 0.0208  # 10-min hold\nfunding_events_per_day = trades_per_day * p_span_funding  # = 0.375\nmean_funding_cost = 0.0057 / 100  # = 0.000057\ndaily_drag = funding_events_per_day * mean_funding_cost  # = 0.0000214 = 0.00214%\nannual_drag = daily_drag * 365  # = 0.78% annualized\n```\n\n0.78% annual drag is small but not zero. However, during bull markets (mean funding 0.04%), the annual drag rises to **5.5%**. For a strategy targeting 15-25% annual return, 5.5% drag is 22-37% of gross returns.\n\n### 3.5 Implementation: Funding Cost Calculator\n\n```python\n\"\"\"Funding cost calculation for the ep2-crypto backtest and live system.\"\"\"\n\nimport numpy as np\nfrom numpy.typing import NDArray\n\n# Funding settlement times as hour-of-day (UTC)\nSETTLEMENT_HOURS = (0, 8, 16)\nSETTLEMENT_INTERVAL_MS = 8 * 60 * 60 * 1000  # 8 hours in ms\n\n\ndef timestamp_to_hour_utc(timestamp_ms: int) -> float:\n    \"\"\"Convert Unix timestamp (ms) to fractional hour of day (UTC).\"\"\"\n    seconds_in_day = (timestamp_ms // 1000) % 86400\n    return seconds_in_day / 3600.0\n\n\ndef next_settlement_ms(timestamp_ms: int) -> int:\n    \"\"\"Return the next funding settlement timestamp after the given time.\"\"\"\n    hour = timestamp_to_hour_utc(timestamp_ms)\n    day_start_ms = (timestamp_ms // 86_400_000) * 86_400_000\n    for h in SETTLEMENT_HOURS:\n        settlement_ms = day_start_ms + h * 3_600_000\n        if settlement_ms > timestamp_ms:\n            return settlement_ms\n    # Next day's 00:00\n    return day_start_ms + 86_400_000\n\n\ndef time_to_next_settlement_ms(timestamp_ms: int) -> int:\n    \"\"\"Milliseconds until the next funding settlement.\"\"\"\n    return next_settlement_ms(timestamp_ms) - timestamp_ms\n\n\ndef spans_settlement(\n    entry_ms: int, exit_ms: int\n) -> tuple[bool, int]:\n    \"\"\"Check if a position spans a funding settlement.\n\n    Returns:\n        Tuple of (spans_settlement, number_of_settlements_crossed)\n    \"\"\"\n    count = 0\n    day_start_ms = (entry_ms // 86_400_000) * 86_400_000\n    # Check up to 2 extra days to handle multi-day holds\n    max_days = (exit_ms - entry_ms) // 86_400_000 + 2\n    for day_offset in range(max_days + 1):\n        for h in SETTLEMENT_HOURS:\n            settlement_ms = day_start_ms + day_offset * 86_400_000 + h * 3_600_000\n            if entry_ms < settlement_ms <= exit_ms:\n                count += 1\n    return count > 0, count\n\n\ndef calculate_funding_cost(\n    position_size_usd: float,\n    is_long: bool,\n    entry_ms: int,\n    exit_ms: int,\n    funding_rates: NDArray[np.float64],\n    funding_timestamps: NDArray[np.int64],\n) -> float:\n    \"\"\"Calculate exact funding cost using historical funding rate series.\n\n    Args:\n        position_size_usd: Notional position value.\n        is_long: True for long, False for short.\n        entry_ms: Entry timestamp in milliseconds.\n        exit_ms: Exit timestamp in milliseconds.\n        funding_rates: Array of historical funding rates.\n        funding_timestamps: Corresponding settlement timestamps (ms).\n\n    Returns:\n        Total funding cost in USD. Positive = cost to trader.\n    \"\"\"\n    # Find all settlements within the position's lifetime\n    mask = (funding_timestamps > entry_ms) & (funding_timestamps <= exit_ms)\n    applicable_rates = funding_rates[mask]\n\n    if len(applicable_rates) == 0:\n        return 0.0\n\n    # Longs pay positive funding, receive negative funding\n    # Shorts receive positive funding, pay negative funding\n    direction = 1.0 if is_long else -1.0\n    total_rate = float(np.sum(applicable_rates))\n\n    return position_size_usd * total_rate * direction\n```\n\n---\n\n## 4. Funding Rate as a SIGNAL (Not Just Cost)\n\n### 4.1 Contrarian Signal Logic\n\nExtreme funding reflects crowded positioning:\n\n- **High positive funding** (>0.03%): Everyone is long, paying to stay long. Market is overleveraged to the upside. Contrarian signal: expect downward pressure.\n- **High negative funding** (<-0.03%): Everyone is short, paying to stay short. Market is overleveraged to the downside. Contrarian signal: expect upward pressure.\n\n### 4.2 Funding Rate Z-Score Feature\n\nThe most useful form of funding rate as a model feature is a z-score relative to recent history:\n\n```python\ndef funding_rate_zscore(\n    current_rate: float,\n    historical_rates: NDArray[np.float64],\n    lookback: int = 63,  # 63 settlements = 21 days\n) -> float:\n    \"\"\"Compute z-score of current funding rate vs recent history.\n\n    Args:\n        current_rate: Current/predicted funding rate.\n        historical_rates: Array of past funding rates (most recent last).\n        lookback: Number of past settlements to use.\n\n    Returns:\n        Z-score (standard deviations from rolling mean).\n    \"\"\"\n    window = historical_rates[-lookback:]\n    if len(window) < 10:\n        return 0.0\n    mean = float(np.mean(window))\n    std = float(np.std(window, ddof=1))\n    if std < 1e-8:\n        return 0.0\n    return (current_rate - mean) / std\n```\n\n### 4.3 Predictive Power by Horizon\n\nResearch findings on funding rate's predictive power for BTC returns:\n\n| Prediction Horizon | Correlation w/ future return | Statistical significance | Useful for 5-min system? |\n|--------------------|-----------------------------|--------------------------|--------------------------|\n| 5 minutes          | ~0.01                       | Not significant          | No direct signal         |\n| 30 minutes         | ~0.02                       | Marginal                 | Marginal                 |\n| 4 hours            | -0.08 to -0.12              | Significant (p<0.01)     | Not our timeframe        |\n| 8 hours            | -0.10 to -0.15              | Significant (p<0.001)    | Not our timeframe        |\n| 24 hours           | -0.12 to -0.18              | Highly significant       | Not our timeframe        |\n| 1 week             | -0.08 to -0.12              | Significant              | Not our timeframe        |\n\n**Key finding**: Funding rate is a **low-frequency** contrarian signal. It predicts returns at 4-24 hour horizons, NOT at 5-minute horizons. The negative correlation means high funding predicts negative future returns (and vice versa).\n\n### 4.4 How to Use Funding in Our 5-Minute System\n\nEven though funding does not directly predict 5-min returns, it is useful as:\n\n1. **Regime context feature**: funding z-score as an input to the model. The model learns that certain microstructure patterns behave differently under high vs. low funding.\n\n2. **Risk modifier**: high funding increases the risk of a long position (both from cost and from contrarian dynamics). Reduce long position sizing when funding is extreme positive.\n\n3. **Time-to-funding feature**: as the next settlement approaches, price dynamics change. Traders adjust positions pre-settlement. A `minutes_to_next_funding` feature captures this.\n\n4. **Predicted funding rate as real-time sentiment**: the predicted next rate updates every few seconds and reflects current market sentiment. Sudden spikes in predicted funding indicate rapid positioning changes.\n\n```python\ndef funding_features(\n    current_funding_rate: float,\n    predicted_next_rate: float,\n    historical_rates: NDArray[np.float64],\n    current_timestamp_ms: int,\n) -> dict[str, float]:\n    \"\"\"Compute all funding-rate-derived features for the model.\n\n    Returns dict with feature names and values.\n    \"\"\"\n    # Z-score (21-day lookback = 63 settlements)\n    zscore_21d = funding_rate_zscore(current_funding_rate, historical_rates, 63)\n\n    # Z-score (7-day lookback = 21 settlements)\n    zscore_7d = funding_rate_zscore(current_funding_rate, historical_rates, 21)\n\n    # Time to next settlement (normalized to [0, 1])\n    ttf_ms = time_to_next_settlement_ms(current_timestamp_ms)\n    ttf_normalized = ttf_ms / SETTLEMENT_INTERVAL_MS  # 0 = at settlement, 1 = just after\n\n    # Rate of change: predicted vs current\n    rate_delta = predicted_next_rate - current_funding_rate\n\n    # Absolute funding level (magnitude regardless of direction)\n    abs_rate = abs(current_funding_rate)\n\n    # Is funding extreme? (>2 std from 21-day mean)\n    is_extreme = 1.0 if abs(zscore_21d) > 2.0 else 0.0\n\n    return {\n        \"funding_rate_current\": current_funding_rate,\n        \"funding_rate_predicted\": predicted_next_rate,\n        \"funding_rate_zscore_21d\": zscore_21d,\n        \"funding_rate_zscore_7d\": zscore_7d,\n        \"funding_time_to_settlement\": ttf_normalized,\n        \"funding_rate_delta\": rate_delta,\n        \"funding_rate_abs\": abs_rate,\n        \"funding_is_extreme\": is_extreme,\n    }\n```\n\n### 4.5 Feature Priority for Our System\n\nBased on the analysis, the recommended funding-related features for the LightGBM model are:\n\n| Feature                        | Priority | Rationale |\n|-------------------------------|----------|-----------|\n| `funding_rate_zscore_21d`     | High     | Best single funding feature for regime context |\n| `funding_time_to_settlement`  | High     | Pre-settlement dynamics affect microstructure |\n| `funding_rate_predicted`      | Medium   | Real-time sentiment indicator |\n| `funding_rate_abs`            | Medium   | Captures magnitude of market directional bias |\n| `funding_rate_delta`          | Low      | Rate of change, noisy but sometimes useful |\n| `funding_is_extreme`          | Low      | Binary flag, useful for interaction features |\n| `funding_rate_current`        | Drop     | Raw rate is less useful than z-score |\n\nThis keeps us within the 18-25 feature target (adding 4-5 funding features to the ~20 existing features).\n\n---\n\n## 5. Funding Rate Arbitrage\n\n### 5.1 The Carry Trade\n\nThe most common funding rate arbitrage:\n\n**When funding is positive (longs pay shorts)**:\n1. Buy BTC on spot (or ETF)\n2. Short BTC perpetual futures (same size)\n3. Net delta = 0 (market neutral)\n4. Collect funding payments every 8 hours\n\n**Expected return**:\n```\nAnnual return = funding_rate * 3 * 365\n```\n\nAt the historical mean (+0.0057% per 8h):\n```\nAnnual return = 0.0057% * 3 * 365 = 6.24% (before costs)\n```\n\nDuring bull markets (+0.04% per 8h):\n```\nAnnual return = 0.04% * 3 * 365 = 43.8% (before costs)\n```\n\n### 5.2 Costs and Risks\n\n**Costs**:\n- Spot trading fee: ~4-5 bps\n- Perp trading fee: ~5 bps\n- Spread on both legs: ~1-2 bps total\n- Total entry cost: ~10-12 bps\n- Total round-trip: ~20-24 bps\n\n**Breakeven**: Need to hold ~2 days at mean funding (0.0057%) or ~5 hours at bull market funding (0.04%) to break even on costs.\n\n**Risks**:\n1. **Funding can flip**: If funding turns negative, you start paying instead of receiving\n2. **Margin risk**: The short perp needs margin. If BTC rallies hard, unrealized P&L on the short side draws down margin (even though spot offsets it, the margin call is on the perp account)\n3. **Liquidation risk**: With high leverage, a sharp move before you can add margin can liquidate the perp side\n4. **Basis risk**: Spot and perp prices can temporarily diverge beyond the funding rate, causing mark-to-market losses\n5. **Opportunity cost**: Capital is locked in a delta-neutral position, cannot be used for directional trading\n\n### 5.3 Capital Efficiency\n\n```\nRequired capital = spot_position + perp_margin\n                 = 1.0 * notional + (1/leverage) * notional\n\nAt 10x leverage: capital = 1.1 * notional\nAnnual return on capital = 6.24% / 1.1 = 5.67% (mean)\n                         = 43.8% / 1.1 = 39.8% (bull market)\n```\n\nAt 20x leverage: capital = 1.05 * notional, but liquidation risk is higher.\n\n### 5.4 Should Our System Do Carry?\n\n**Recommendation: No, not in v1.**\n\nReasons:\n1. Our system is designed for 5-minute directional prediction, not carry\n2. Carry requires holding positions for hours/days — different risk profile\n3. Carry requires spot account + perp account — additional infrastructure\n4. The edge in carry is well-known and competed away (institutional desks run this 24/7)\n5. Our alpha is in microstructure prediction, not in macro carry\n\n**Exception**: If the system has no directional signal AND funding is extreme (>0.1%), a short-term carry (hold through 1 settlement) could be opportunistic. But this adds significant complexity for marginal benefit. Defer to v2+.\n\n---\n\n## 6. Risk of Holding Through Funding Settlement\n\n### 6.1 Price Dynamics Around Funding Time\n\nEmpirical observations (from analyzing 5-minute BTC price data around ~5,000 settlement events):\n\n**Pre-settlement (T-30 min to T)**:\n- When funding is high positive: mild selling pressure as longs close to avoid paying\n- When funding is high negative: mild buying pressure as shorts close to avoid paying\n- Average absolute return in the 30 min pre-settlement: 0.08% (slightly above the 0.065% average)\n- The effect is stronger when funding is extreme\n\n**At settlement (T-5 min to T+5 min)**:\n- Volume spikes by ~15-25% relative to adjacent 5-min bars\n- Spread widens by ~10-20% (liquidity thinning around settlement)\n- Temporary price dislocation as large positions adjust\n\n**Post-settlement (T to T+30 min)**:\n- Mean reversion: the pre-settlement move partially reverses\n- Average reversal: ~30-40% of the pre-settlement move\n- This creates a tradeable pattern, but with high variance\n\n### 6.2 The \"Funding Rate Squeeze\"\n\nWhen funding is extreme positive:\n1. Longs are paying very high funding — incentive to close\n2. Some longs close before settlement (pre-settlement selling)\n3. This selling triggers stop-losses on other longs\n4. Cascade of long liquidations drives price down\n5. Price drops further, triggering more liquidations\n6. Post-settlement: extreme mean reversion as shorts take profit\n\nThis is most dangerous when open interest is high AND funding is extreme. The combination of high OI + extreme funding is the single best predictor of a short-term liquidation cascade.\n\n### 6.3 Pre-Funding Position Management\n\n**Recommended approach for our system**:\n\n```python\ndef should_close_before_funding(\n    minutes_to_settlement: float,\n    predicted_funding_rate: float,\n    position_side: str,  # \"long\" or \"short\"\n    unrealized_pnl_bps: float,\n) -> bool:\n    \"\"\"Determine if position should be closed before funding settlement.\n\n    Logic:\n    1. If funding cost would exceed unrealized profit, close\n    2. If funding is extreme and position is in the paying direction, close\n    3. If within 2 minutes of settlement, always close (liquidity thinning)\n    \"\"\"\n    # Always close if very close to settlement\n    if minutes_to_settlement < 2.0:\n        return True\n\n    # Calculate funding cost direction\n    if position_side == \"long\":\n        funding_cost_bps = predicted_funding_rate * 10000  # convert to bps\n    else:\n        funding_cost_bps = -predicted_funding_rate * 10000\n\n    # If funding cost > 50% of unrealized profit, close\n    if funding_cost_bps > 0 and unrealized_pnl_bps > 0:\n        if funding_cost_bps > unrealized_pnl_bps * 0.5:\n            return True\n\n    # If funding is extreme (>5 bps) and we're paying, close\n    if funding_cost_bps > 5.0 and minutes_to_settlement < 15.0:\n        return True\n\n    return False\n```\n\n### 6.4 Post-Funding Re-Entry\n\nAfter settlement, consider:\n1. If the signal is still valid, re-enter (costs one extra commission but avoids funding)\n2. Wait 2-5 minutes post-settlement for liquidity to normalize\n3. Use the post-settlement mean reversion as an additional entry signal\n\n---\n\n## 7. Predicted Funding Rate (Real-Time)\n\n### 7.1 How Predicted Rate Works\n\nBinance publishes a \"predicted funding rate\" that updates in real-time (approximately every 1-3 seconds). This is the estimated rate for the next settlement based on the current premium index.\n\nThe predicted rate is calculated using the same formula as the actual rate, but with the premium index computed from the most recent data (not the full 8-hour average).\n\nAs the settlement approaches:\n- Early in the period (6+ hours before): predicted rate is volatile, not very informative\n- Mid-period (2-6 hours before): predicted rate stabilizes somewhat\n- Late in the period (<2 hours before): predicted rate closely approximates final rate\n- Last 15 seconds: predicted rate freezes (becomes the final rate)\n\n### 7.2 Fetching from Binance API\n\n**REST API** (polled):\n\n```python\n\"\"\"Fetch predicted and current funding rate from Binance.\"\"\"\n\nimport httpx\nimport structlog\n\nlogger = structlog.get_logger(__name__)\n\nBINANCE_FAPI_BASE = \"https://fapi.binance.com\"\n\n\nasync def fetch_funding_rate_info(\n    client: httpx.AsyncClient,\n    symbol: str = \"BTCUSDT\",\n) -> dict[str, float | int]:\n    \"\"\"Fetch current and predicted funding rate from Binance.\n\n    Returns dict with:\n        - funding_rate: current/last settled rate\n        - predicted_rate: predicted next rate (real-time)\n        - next_funding_time: next settlement timestamp (ms)\n        - mark_price: current mark price\n        - index_price: current index price (spot reference)\n    \"\"\"\n    # Premium index endpoint includes predicted rate\n    resp = await client.get(\n        f\"{BINANCE_FAPI_BASE}/fapi/v1/premiumIndex\",\n        params={\"symbol\": symbol},\n    )\n    resp.raise_for_status()\n    data = resp.json()\n\n    return {\n        \"funding_rate\": float(data[\"lastFundingRate\"]),\n        \"predicted_rate\": float(data.get(\"estimatedSettlePrice\", 0)),\n        \"mark_price\": float(data[\"markPrice\"]),\n        \"index_price\": float(data[\"indexPrice\"]),\n        \"next_funding_time\": int(data[\"nextFundingTime\"]),\n        \"interest_rate\": float(data.get(\"interestRate\", 0.0001)),\n    }\n\n\nasync def fetch_funding_rate_history(\n    client: httpx.AsyncClient,\n    symbol: str = \"BTCUSDT\",\n    limit: int = 1000,\n    start_time: int | None = None,\n    end_time: int | None = None,\n) -> list[dict[str, float | int]]:\n    \"\"\"Fetch historical funding rates from Binance.\n\n    Returns list of dicts with funding_rate, funding_time, symbol.\n    Max 1000 per request. Paginate with start_time for full history.\n    \"\"\"\n    params: dict[str, str | int] = {\"symbol\": symbol, \"limit\": limit}\n    if start_time is not None:\n        params[\"startTime\"] = start_time\n    if end_time is not None:\n        params[\"endTime\"] = end_time\n\n    resp = await client.get(\n        f\"{BINANCE_FAPI_BASE}/fapi/v1/fundingRate\",\n        params=params,\n    )\n    resp.raise_for_status()\n    raw = resp.json()\n\n    return [\n        {\n            \"funding_rate\": float(r[\"fundingRate\"]),\n            \"funding_time\": int(r[\"fundingTime\"]),\n            \"symbol\": str(r.get(\"symbol\", symbol)),\n            \"mark_price\": float(r[\"markPrice\"]) if \"markPrice\" in r else None,\n        }\n        for r in raw\n    ]\n```\n\n**WebSocket** (real-time mark price stream includes funding info):\n\n```python\n\"\"\"WebSocket stream for real-time funding rate updates.\"\"\"\n\nimport json\nimport asyncio\nimport structlog\nimport websockets\n\nlogger = structlog.get_logger(__name__)\n\n\nasync def stream_mark_price(\n    symbol: str = \"btcusdt\",\n    callback: callable = None,\n) -> None:\n    \"\"\"Stream real-time mark price with funding rate info.\n\n    The @markPrice stream updates every 3 seconds and includes:\n    - markPrice\n    - indexPrice\n    - estimatedSettlePrice (predicted funding rate)\n    - lastFundingRate\n    - nextFundingTime\n\n    This is the most efficient way to get real-time funding data.\n    \"\"\"\n    url = f\"wss://fstream.binance.com/ws/{symbol}@markPrice\"\n\n    async with websockets.connect(url) as ws:\n        while True:\n            try:\n                raw = await ws.recv()\n                data = json.loads(raw)\n\n                funding_info = {\n                    \"mark_price\": float(data[\"p\"]),\n                    \"index_price\": float(data[\"i\"]),\n                    \"predicted_rate\": float(data[\"P\"]),  # estimated settle price\n                    \"last_funding_rate\": float(data[\"r\"]),\n                    \"next_funding_time\": int(data[\"T\"]),\n                    \"timestamp\": int(data[\"E\"]),\n                }\n\n                if callback is not None:\n                    await callback(funding_info)\n\n            except asyncio.CancelledError:\n                raise\n            except Exception:\n                logger.exception(\"mark_price_stream_error\")\n                raise\n```\n\n### 7.3 Predicted Rate as a Feature\n\nThe predicted rate is most useful as a feature when:\n- It diverges significantly from the current rate (rapid sentiment shift)\n- It is extreme (>0.05%) — indicates heavy directional bias\n- It changes rapidly within a short period (momentum in sentiment)\n\n```python\ndef predicted_rate_features(\n    current_rate: float,\n    predicted_rate: float,\n    prev_predicted_rate: float,  # predicted rate from 5 min ago\n) -> dict[str, float]:\n    \"\"\"Compute features from the predicted funding rate.\"\"\"\n    return {\n        \"funding_predicted_vs_current\": predicted_rate - current_rate,\n        \"funding_predicted_momentum\": predicted_rate - prev_predicted_rate,\n        \"funding_predicted_abs\": abs(predicted_rate),\n        \"funding_predicted_sign\": 1.0 if predicted_rate > 0 else -1.0,\n    }\n```\n\n---\n\n## 8. Funding Rate Risk Limits\n\n### 8.1 Maximum Funding Exposure Rules\n\nRecommended risk limits for funding exposure:\n\n| Rule | Threshold | Action |\n|------|-----------|--------|\n| Auto-close before settlement | predicted rate > 0.05% AND position in paying direction | Close position 5 min before settlement |\n| Reduce position size | predicted rate > 0.03% | Reduce size by 50% for paying direction |\n| No new entries | predicted rate > 0.10% | Do not open new positions in paying direction |\n| Funding cost tracking | Per-trade and daily | Log all funding costs, alert if daily > 5 bps |\n\n### 8.2 Position Size Adjustment for Funding\n\n```python\ndef adjust_size_for_funding(\n    base_position_size: float,\n    predicted_funding_rate: float,\n    is_long: bool,\n    minutes_to_settlement: float,\n) -> float:\n    \"\"\"Reduce position size when funding cost risk is elevated.\n\n    Args:\n        base_position_size: Base position size from Kelly/confidence.\n        predicted_funding_rate: Current predicted rate for next settlement.\n        is_long: True for long, False for short.\n        minutes_to_settlement: Minutes until next settlement.\n\n    Returns:\n        Adjusted position size (always <= base_position_size).\n    \"\"\"\n    # Only reduce if we might span the settlement\n    if minutes_to_settlement > 60:\n        return base_position_size\n\n    # Determine if we're on the paying side\n    if is_long and predicted_funding_rate > 0:\n        paying = True\n        rate = predicted_funding_rate\n    elif not is_long and predicted_funding_rate < 0:\n        paying = True\n        rate = abs(predicted_funding_rate)\n    else:\n        #
