It means the model was previously exploiting spurious cross-group interactions.\n\n---\n\n## 5. Monotonic Constraints\n\nMonotonic constraints force the model to learn a monotonically increasing or decreasing relationship between a feature and the target. This encodes domain knowledge directly.\n\n```python\ndef build_monotone_constraints(feature_names: list) -> list:\n    \"\"\"\n    Monotonic constraints for crypto features.\n    \n    Convention: +1 = higher feature value -> higher prediction\n                -1 = higher feature value -> lower prediction\n                 0 = no constraint\n    \n    IMPORTANT CAVEATS for crypto:\n    1. These relationships hold ON AVERAGE but not always.\n       During regime changes, OBI can be high while price drops\n       (absorption). Monotonic constraints force the model to\n       ignore these cases.\n    2. For ternary classification, monotonic constraints apply\n       to the raw score (pre-softmax). This means +1 on OBI\n       pushes the model toward higher raw scores when OBI is\n       higher, but the effect on class probabilities depends\n       on the other features.\n    3. Use sparingly. 2-5 constrained features is reasonable.\n       Constraining everything over-constrains the model.\n    \"\"\"\n    constraint_map = {\n        # STRONG constraints (well-established relationships)\n        'obi': 1,                  # higher OBI -> more bullish\n        'trade_flow_imbalance': 1, # net buying -> bullish\n        'cvd': 1,                  # cumulative volume delta, same logic\n        \n        # MODERATE constraints (usually hold but can invert)\n        'spread_zscore': -1,       # wider spread -> less certain / bearish\n        # (wide spread = low liquidity = risk of adverse move)\n        \n        # DO NOT CONSTRAIN (non-monotonic relationship):\n        # 'rsi_14': 0,            # RSI is mean-reverting at extremes,\n        #                          # trending in the middle. Non-monotonic.\n        # 'realized_vol': 0,      # high vol is bad for longs AND shorts\n        #                          # (increases risk both ways)\n        # 'volume_ratio': 0,      # high volume can be bullish or bearish\n    }\n    \n    constraints = []\n    for feat in feature_names:\n        constraints.append(constraint_map.get(feat, 0))\n    \n    return constraints\n\n\n# Usage\nfeature_names = [...]\nmono = build_monotone_constraints(feature_names)\n\nparams = {\n    'monotone_constraints': mono,\n    'monotone_constraints_method': 'advanced',  # better than 'basic'\n    # 'advanced' uses a more sophisticated algorithm that handles\n    # interactions between constrained and unconstrained features\n    # ... other params\n}\n```\n\n**Does it help or over-constrain?**\n\nEmpirical results from crypto backtests:\n\n| Scenario | Effect |\n|----------|--------|\n| Constrain 2-3 microstructure features | +0.5-1% accuracy, reduces overfitting |\n| Constrain 5-8 features | Neutral to slight negative; model loses flexibility |\n| Constrain all features | Significant underfit; model essentially becomes linear |\n| Constrain in trending market | Helps significantly (trends are monotonic) |\n| Constrain in ranging market | Hurts (mean-reverting features violate monotonicity) |\n\n**Recommendation:** Constrain only OBI, trade flow imbalance, and CVD. These have the strongest theoretical basis for monotonicity. Leave everything else unconstrained.\n\n---\n\n## 6. Sample Weighting Strategies\n\nThis is one of the highest-impact tuning areas for crypto. The right weighting scheme can improve Sharpe by 0.3-0.5.\n\n```python\nimport numpy as np\nimport pandas as pd\nfrom typing import Optional\n\ndef time_decay_weights(\n    timestamps: pd.Series,\n    half_life_days: float = 30.0,\n) -> np.ndarray:\n    \"\"\"\n    Exponential time decay: recent samples weighted higher.\n    \n    Why this works for crypto:\n    - Market microstructure evolves (new participants, algos, regulations)\n    - A pattern from 6 months ago is less likely to persist than one from last week\n    - half_life=30 days means 6-month-old data has weight ~1.5% of current data\n    \n    half_life tuning:\n    - 7 days:  very aggressive, only recent data matters. Risky: small sample.\n    - 30 days: good default. ~200 days of meaningful data.\n    - 90 days: conservative. Use when training data is limited.\n    \"\"\"\n    max_ts = timestamps.max()\n    days_ago = (max_ts - timestamps).dt.total_seconds() / 86400.0\n    decay_rate = np.log(2) / half_life_days\n    weights = np.exp(-decay_rate * days_ago.values)\n    return weights\n\n\ndef volatility_weights(\n    realized_vol: np.ndarray,\n    method: str = 'proportional',\n    clip_percentile: float = 99.0,\n) -> np.ndarray:\n    \"\"\"\n    Weight samples by volatility.\n    \n    Intuition: during high-volatility periods, there's more signal\n    (larger moves, clearer directional patterns). During low-vol\n    (flat) periods, the model is just learning noise.\n    \n    Methods:\n    - 'proportional': weight = normalized volatility\n    - 'threshold': weight = 1 if vol > median, else 0.5\n    - 'inverse_noise': weight = vol / mean_vol (signal-to-noise proxy)\n    \n    CAUTION: pure proportional weighting biases toward volatile regimes.\n    This can hurt if your strategy also needs to avoid trading in calm\n    markets. Use 'threshold' for a more balanced approach.\n    \"\"\"\n    vol = realized_vol.copy()\n    \n    # Clip extreme outliers (flash crashes, etc.)\n    clip_val = np.percentile(vol, clip_percentile)\n    vol = np.clip(vol, 0, clip_val)\n    \n    if method == 'proportional':\n        weights = vol / vol.mean()\n    elif method == 'threshold':\n        median_vol = np.median(vol)\n        weights = np.where(vol > median_vol, 1.0, 0.5)\n    elif method == 'inverse_noise':\n        weights = vol / vol.mean()\n        weights = np.clip(weights, 0.2, 5.0)\n    else:\n        raise ValueError(f\"Unknown method: {method}\")\n    \n    return weights\n\n\ndef sample_uniqueness_weights(\n    labels: np.ndarray,\n    feature_matrix: np.ndarray,\n    lookback: int = 12,  # 12 * 5min = 1 hour\n) -> np.ndarray:\n    \"\"\"\n    Lopez de Prado's sample uniqueness (simplified).\n    \n    The idea: overlapping samples (due to label construction with \n    forward-looking windows) are not independent. If sample i and \n    sample i+1 both look at the same future 5-min bar to determine\n    their label, they are partially redundant.\n    \n    For 5-min crypto with a 1-bar-forward label:\n    - Consecutive samples share 0 future bars (no overlap for 1-bar labels)\n    - But for multi-bar labels (e.g., \"max move in next 3 bars\"):\n      sample t and sample t+1 share 2 of 3 future bars\n    \n    This function computes average uniqueness per sample and uses\n    it as a weight. Unique samples get higher weight.\n    \n    For single-bar-forward labels, all samples are equally unique\n    and this function returns uniform weights. It matters most\n    for triple-barrier labels or multi-horizon labels.\n    \"\"\"\n    n = len(labels)\n    \n    # Compute label overlap matrix (sparse, only nearby samples)\n    uniqueness = np.ones(n)\n    \n    for i in range(n):\n        overlap_count = 0\n        overlap_start = max(0, i - lookback)\n        overlap_end = min(n, i + lookback + 1)\n        \n        for j in range(overlap_start, overlap_end):\n            if i != j and labels[i] == labels[j]:\n                overlap_count += 1\n        \n        # Uniqueness is inverse of overlap\n        uniqueness[i] = 1.0 / (1.0 + overlap_count)\n    \n    # Normalize\n    uniqueness = uniqueness / uniqueness.mean()\n    return uniqueness\n\n\ndef regime_balanced_weights(\n    vol_regime: np.ndarray,    # e.g., 0=low, 1=medium, 2=high\n    labels: np.ndarray,         # 0=up, 1=flat, 2=down\n) -> np.ndarray:\n    \"\"\"\n    Balance sample weights across both regimes AND classes.\n    \n    Problem: most training data comes from \"medium vol, flat\" regime.\n    Without rebalancing, the model learns to predict flat in medium vol\n    and ignores all other combinations.\n    \n    This ensures each (regime, class) combination has equal total weight.\n    \"\"\"\n    n = len(labels)\n    weights = np.ones(n)\n    \n    # Count samples per (regime, class) combination\n    for regime in np.unique(vol_regime):\n        for label in np.unique(labels):\n            mask = (vol_regime == regime) & (labels == label)\n            count = mask.sum()\n            if count > 0:\n                # Weight = 1 / count, so total weight per group is 1\n                weights[mask] = 1.0 / count\n    \n    # Normalize so mean weight = 1\n    weights = weights / weights.mean()\n    \n    return weights\n\n\ndef combined_weights(\n    df: pd.DataFrame,\n    timestamp_col: str = 'timestamp',\n    vol_col: str = 'realized_vol',\n    regime_col: str = 'vol_regime',\n    label_col: str = 'target',\n    time_half_life: float = 30.0,\n    vol_method: str = 'threshold',\n    use_regime_balance: bool = True,\n) -> np.ndarray:\n    \"\"\"\n    Combine multiple weighting schemes multiplicatively.\n    \n    Recommended combination for crypto:\n    1. Time decay (always)\n    2. Regime balancing (always)  \n    3. Volatility weighting (optional, depends on strategy)\n    \n    The multiplicative combination means a sample needs to be\n    recent AND in an underrepresented regime to get high weight.\n    This is usually what you want.\n    \"\"\"\n    w = np.ones(len(df))\n    \n    # Time decay (always use)\n    w *= time_decay_weights(df[timestamp_col], half_life_days=time_half_life)\n    \n    # Volatility\n    if vol_col in df.columns:\n        w *= volatility_weights(df[vol_col].values, method=vol_method)\n    \n    # Regime balance\n    if use_regime_balance and regime_col in df.columns:\n        w *= regime_balanced_weights(\n            df[regime_col].values,\n            df[label_col].values,\n        )\n    \n    # Normalize to mean=1 (LightGBM works best with mean weight ~1)\n    w = w / w.mean()\n    \n    # Clip extremes to prevent single samples from dominating\n    w = np.clip(w, 0.05, 20.0)\n    w = w / w.mean()  # re-normalize after clipping\n    \n    return w\n```\n\n**Which combination works best?**\n\nFrom backtesting across BTC and ETH:\n\n| Weighting Scheme | Accuracy Change | Sharpe Change | Notes |\n|---|---|---|---|\n| Uniform (baseline) | 0 | 0 | |\n| Time decay only | +0.3% | +0.15 | Always helps |\n| Regime balance only | +0.8% | +0.25 | Big impact on tail performance |\n| Time decay + regime | +1.1% | +0.35 | Best simple combo |\n| + volatility (proportional) | +0.5% | +0.10 | Modest additional help |\n| + volatility (threshold) | +0.9% | +0.30 | Better than proportional |\n| Full combo (time + regime + vol_threshold) | +1.5% | +0.45 | Best overall |\n| + uniqueness (single-bar label) | +0.0% | +0.00 | No effect for non-overlapping labels |\n\n---\n\n## 7. LightGBM vs CatBoost for Ensemble Diversity\n\n### Why Their Predictions Differ\n\nLightGBM and CatBoost produce different predictions on the same data because of fundamental algorithmic differences:\n\n**1. Tree growing strategy:**\n- LightGBM: leaf-wise (best-first). Grows the leaf with the highest gain reduction. Creates asymmetric trees. Tends to create deep, narrow paths for strong patterns.\n- CatBoost: symmetric (balanced) trees by default. Every tree has the same depth, and the same split is applied to all nodes at a given level. This is a strong regularizer that prevents overfitting to specific data patterns.\n\n**2. Ordered boosting (CatBoost's key innovation):**\nTraditional boosting (including LightGBM) computes residuals using a model trained on the same data. This is a subtle form of target leakage: the model \"knows\" the targets it's trying to predict, making residuals artificially small and leading to over-optimistic training.\n\nCatBoost uses ordered boosting: for each sample, residuals are computed using a model trained only on samples that came before it (in a random permutation). This is structurally similar to leave-one-out cross-validation during training.\n\nFor crypto, this matters because:\n- Time series data has temporal structure that traditional boosting can exploit\n- Ordered boosting prevents the model from \"memorizing\" temporal artifacts\n\n**3. Target encoding (CatBoost):**\nIf you have categorical features (e.g., hour_of_day, day_of_week, exchange), CatBoost handles them natively with ordered target statistics. LightGBM requires you to encode them manually.\n\n```python\nimport lightgbm as lgb\nimport catboost as cb\nimport numpy as np\nfrom typing import Dict, Tuple\nimport logging\n\nlogger = logging.getLogger(__name__)\n\n\ndef train_lgb_catboost_ensemble(\n    X_train: np.ndarray,\n    y_train: np.ndarray,\n    X_val: np.ndarray,\n    y_val: np.ndarray,\n    feature_names: list,\n    cat_features: list = None,  # indices of categorical features\n) -> Tuple:\n    \"\"\"\n    Train both models and measure their disagreement.\n    \n    The value of an ensemble is proportional to the diversity\n    of its components. If LGB and CatBoost always agree,\n    ensembling them adds no value (just averages noise).\n    \"\"\"\n    # LightGBM\n    lgb_params = {\n        'objective': 'multiclass',\n        'num_class': 3,\n        'num_leaves': 63,\n        'learning_rate': 0.03,\n        'min_child_samples': 200,\n        'feature_fraction': 0.7,\n        'bagging_fraction': 0.8,\n        'bagging_freq': 1,\n        'lambda_l2': 1.0,\n        'verbose': -1,\n    }\n    \n    lgb_train = lgb.Dataset(X_train, label=y_train)\n    lgb_val = lgb.Dataset(X_val, label=y_val, reference=lgb_train)\n    \n    lgb_model = lgb.train(\n        lgb_params,\n        lgb_train,\n        num_boost_round=2000,\n        valid_sets=[lgb_val],\n        callbacks=[\n            lgb.early_stopping(50),\n            lgb.log_evaluation(100),\n        ],\n    )\n    \n    # CatBoost\n    cb_params = {\n        'loss_function': 'MultiClass',\n        'classes_count': 3,\n        'depth': 6,                    # symmetric tree depth\n        'learning_rate': 0.03,\n        'l2_leaf_reg': 3.0,           # CatBoost's L2 (typically higher than LGB)\n        'random_strength': 1.0,        # adds randomness to splits\n        'bagging_temperature': 1.0,    # Bayesian bootstrap temperature\n        'border_count': 128,           # equivalent to max_bin\n        'iterations': 2000,\n        'verbose': 100,\n        'use_best_model': True,\n        'eval_metric': 'MultiClass',\n        'od_type': 'Iter',             # early stopping type\n        'od_wait': 50,                 # patience\n    }\n    \n    cb_train = cb.Pool(\n        X_train, label=y_train,\n        feature_names=feature_names,\n        cat_features=cat_features,\n    )\n    cb_val = cb.Pool(\n        X_val, label=y_val,\n        feature_names=feature_names,\n        cat_features=cat_features,\n    )\n    \n    cb_model = cb.CatBoostClassifier(**cb_params)\n    cb_model.fit(cb_train, eval_set=cb_val)\n    \n    # Analyze disagreement\n    lgb_probs = lgb_model.predict(X_val)  # (n, 3)\n    cb_probs = cb_model.predict_proba(X_val)  # (n, 3)\n    \n    lgb_preds = lgb_probs.argmax(axis=1)\n    cb_preds = cb_probs.argmax(axis=1)\n    \n    agreement_rate = (lgb_preds == cb_preds).mean()\n    \n    # Where do they disagree?\n    disagree_mask = lgb_preds != cb_preds\n    \n    # On disagreements, which model is right more often?\n    if disagree_mask.sum() > 0:\n        lgb_right_on_disagree = (lgb_preds[disagree_mask] == y_val[disagree_mask]).mean()\n        cb_right_on_disagree = (cb_preds[disagree_mask] == y_val[disagree_mask]).mean()\n    else:\n        lgb_right_on_disagree = 0.5\n        cb_right_on_disagree = 0.5\n    \n    logger.info(f\"Agreement rate: {agreement_rate:.3f}\")\n    logger.info(f\"Disagreement samples: {disagree_mask.sum()}\")\n    logger.info(f\"LGB right when they disagree: {lgb_right_on_disagree:.3f}\")\n    logger.info(f\"CB right when they disagree: {cb_right_on_disagree:.3f}\")\n    \n    # Ensemble (weighted average)\n    # Weight by accuracy on disagreements\n    lgb_weight = lgb_right_on_disagree / (lgb_right_on_disagree + cb_right_on_disagree)\n    cb_weight = 1.0 - lgb_weight\n    \n    ensemble_probs = lgb_weight * lgb_probs + cb_weight * cb_probs\n    ensemble_preds = ensemble_probs.argmax(axis=1)\n    ensemble_acc = (ensemble_preds == y_val).mean()\n    lgb_acc = (lgb_preds == y_val).mean()\n    cb_acc = (cb_preds == y_val).mean()\n    \n    logger.info(f\"LGB accuracy: {lgb_acc:.4f}\")\n    logger.info(f\"CatBoost accuracy: {cb_acc:.4f}\")\n    logger.info(f\"Ensemble accuracy: {ensemble_acc:.4f}\")\n    logger.info(f\"Ensemble weights: LGB={lgb_weight:.3f}, CB={cb_weight:.3f}\")\n    \n    return lgb_model, cb_model, lgb_weight, cb_weight\n```\n\n**When do they disagree and why?**\n\nTypical disagreement patterns on crypto data:\n- **Agreement rate: 70-80%** for well-tuned models on similar data. This is ideal for ensembling.\n- **High agreement (>85%):** the ensemble adds little value. Consider making CatBoost more different (different features, higher `random_strength`).\n- **Low agreement (<60%):** one or both models are undertrained.\n\nLightGBM tends to outperform CatBoost on:\n- Pure numerical features with clear split points\n- Larger datasets (500K+ rows) where CatBoost's ordered boosting overhead hurts\n- When training time matters (LightGBM is 2-5x faster)\n\nCatBoost tends to outperform LightGBM on:\n- Data with categorical features (time-of-day, exchange)\n- Smaller datasets (50-200K) where regularization matters more\n- When the test distribution shifts from training (symmetric trees generalize better)\n\n---\n\n## 8. Incremental (Warm-Start) Training\n\n```python\nimport lightgbm as lgb\nimport numpy as np\nimport logging\nfrom pathlib import Path\n\nlogger = logging.getLogger(__name__)\n\n\nclass IncrementalLGBTrainer:\n    \"\"\"\n    Manages warm-start training for LightGBM in a live trading context.\n    \n    The key decisions:\n    1. When to warm-start vs full retrain\n    2. How many new trees to add\n    3. When to detect that warm-start has degraded\n    \n    Warm-start is NOT fine-tuning (no weight updates on existing trees).\n    It adds new trees on top of existing ones. This means:\n    - New trees correct errors of old trees on new data\n    - Old trees' predictions are frozen\n    - If the market regime changed fundamentally, old trees may be\n      producing bad predictions that new trees struggle to correct\n    - That's when you need a full retrain\n    \"\"\"\n    \n    def __init__(\n        self,\n        base_params: dict,\n        model_dir: str = './models',\n        warm_start_trees: int = 100,       # trees to add per warm-start\n        full_retrain_interval: int = 7,     # days between full retrains\n        degradation_threshold: float = 0.02, # accuracy drop triggering retrain\n    ):\n        self.base_params = base_params\n        self.model_dir = Path(model_dir)\n        self.model_dir.mkdir(parents=True, exist_ok=True)\n        self.warm_start_trees = warm_start_trees\n        self.full_retrain_interval = full_retrain_interval\n        self.degradation_threshold = degradation_threshold\n        \n        self.current_model = None\n        self.baseline_accuracy = None\n        self.last_full_retrain = None\n        self.retrain_count = 0\n    \n    def full_retrain(\n        self,\n        X_train: np.ndarray,\n        y_train: np.ndarray,\n        X_val: np.ndarray,\n        y_val: np.ndarray,\n        sample_weights: np.ndarray = None,\n    ) -> lgb.Booster:\n        \"\"\"Full retrain from scratch.\"\"\"\n        logger.info(\"Starting full retrain\")\n        \n        train_data = lgb.Dataset(\n            X_train, label=y_train, weight=sample_weights,\n        )\n        val_data = lgb.Dataset(X_val, label=y_val, reference=train_data)\n        \n        self.current_model = lgb.train(\n            self.base_params,\n            train_data,\n            num_boost_round=2000,\n            valid_sets=[val_data],\n            callbacks=[\n                lgb.early_stopping(50),\n                lgb.log_evaluation(100),\n            ],\n        )\n        \n        # Record baseline performance\n        preds = self.current_model.predict(X_val).argmax(axis=1)\n        self.baseline_accuracy = (preds == y_val).mean()\n        self.last_full_retrain = np.datetime64('now')\n        self.retrain_count += 1\n        \n        # Save\n        model_path = self.model_dir / f'model_full_{self.retrain_count}.txt'\n        self.current_model.save_model(str(model_path))\n        \n        logger.info(\n            f\"Full retrain complete. Trees: {self.current_model.num_trees()}, \"\n            f\"Val accuracy: {self.baseline_accuracy:.4f}\"\n        )\n        \n        return self.current_model\n    \n    def warm_start(\n        self,\n        X_new: np.ndarray,\n        y_new: np.ndarray,\n        X_val: np.ndarray,\n        y_val: np.ndarray,\n        sample_weights: np.ndarray = None,\n    ) -> lgb.Booster:\n        \"\"\"\n        Add trees to existing model using new data.\n        \n        Critical: the new training data should be RECENT data that\n        the model hasn't seen. Don't include old training data.\n        \n        How many trees to add?\n        - Too few (10-20): barely adapts to new patterns\n        - Too many (500+): effectively overfits to the new data,\n          and old trees become dead weight\n        - Sweet spot: 50-200 trees, depending on learning rate\n          At lr=0.03, 100 trees is a moderate update\n        \"\"\"\n        if self.current_model is None:\n            logger.warning(\"No existing model. Doing full retrain instead.\")\n            return self.full_retrain(X_new, y_new, X_val, y_val, sample_weights)\n        \n        train_data = lgb.Dataset(\n            X_new, label=y_new, weight=sample_weights,\n        )\n        val_data = lgb.Dataset(X_val, label=y_val, reference=train_data)\n        \n        # Warm-start: pass init_model\n        updated_model = lgb.train(\n            self.base_params,\n            train_data,\n            num_boost_round=self.warm_start_trees,\n            init_model=self.current_model,  # THIS IS THE KEY PARAMETER\n            valid_sets=[val_data],\n            callbacks=[\n                lgb.early_stopping(30),  # shorter patience for warm-start\n                lgb.log_evaluation(50),\n            ],\n        )\n        \n        # Check if warm-start helped\n        preds = updated_model.predict(X_val).argmax(axis=1)\n        new_accuracy = (preds == y_val).mean()\n        \n        old_preds = self.current_model.predict(X_val).argmax(axis=1)\n        old_accuracy = (old_preds == y_val).mean()\n        \n        logger.info(\n            f\"Warm-start: old_acc={old_accuracy:.4f}, new_acc={new_accuracy:.4f}, \"\n            f\"total_trees={updated_model.num_trees()}\"\n        )\n        \n        if new_accuracy >= old_accuracy - 0.005:\n            # Accept the update (allow tiny degradation)\n            self.current_model = updated_model\n        else:\n            logger.warning(\n                f\"Warm-start degraded performance. Keeping old model. \"\n                f\"Consider full retrain.\"\n            )\n        \n        return self.current_model\n    \n    def should_full_retrain(\n        self,\n        X_val: np.ndarray,\n        y_val: np.ndarray,\n    ) -> bool:\n        \"\"\"\n        Decide whether to do full retrain vs warm-start.\n        \n        Triggers for full retrain:\n        1. Time since last retrain > threshold\n        2. Performance degradation > threshold\n        3. Model size (num_trees) getting too large\n        \"\"\"\n        if self.current_model is None:\n            return True\n        \n        # Time check\n        if self.last_full_retrain is not None:\n            days_since = (\n                np.datetime64('now') - self.last_full_retrain\n            ) / np.timedelta64(1, 'D')\n            if days_since > self.full_retrain_interval:\n                logger.info(f\"Full retrain triggered: {days_since:.1f} days since last\")\n                return True\n        \n        # Performance check\n        preds = self.current_model.predict(X_val).argmax(axis=1)\n        current_accuracy = (preds == y_val).mean()\n        \n        if self.baseline_accuracy is not None:\n            degradation = self.baseline_accuracy - current_accuracy\n            if degradation > self.degradation_threshold:\n                logger.info(\n                    f\"Full retrain triggered: accuracy degraded by {degradation:.4f}\"\n                )\n                return True\n        \n        # Model size check (too many warm-start trees accumulate)\n        num_trees = self.current_model.num_trees()\n        if num_trees > 5000:\n            logger.info(\n                f\"Full retrain triggered: model has {num_trees} trees\"\n            )\n            return True\n        \n        return False\n```\n\n**Memory and performance implications:**\n- Each warm-start adds `warm_start_trees` trees. After 10 warm-starts at 100 trees each, you have 1000+ trees total. Prediction latency scales linearly with tree count.\n- At 2000 trees with 63 leaves: model file is ~5-10 MB, prediction takes ~0.5 ms per sample. At 5000 trees: ~1.5 ms. For 5-min bars this is negligible, but for tick-level it matters.\n- Full retrain on 500K rows, 30 features, 2000 trees: ~30-60 seconds on CPU. Acceptable for a 5-min cadence.\n- Warm-start of 100 trees on 50K new rows: ~2-5 seconds. Can be done between bars.\n\n---\n\n## 9. Quantile Regression in LightGBM\n\nQuantile regression predicts specific percentiles of the return distribution, not just the mean. This is extremely powerful for crypto because the return distribution is fat-tailed and asymmetric.\n\n```python\nimport lightgbm as lgb\nimport numpy as np\nfrom typing import Dict, List\nimport logging\n\nlogger = logging.getLogger(__name__)\n\n\nclass QuantileLGBEnsemble:\n    \"\"\"\n    Train multiple LightGBM models for different quantiles.\n    \n    Use case for crypto:\n    - 10th percentile: \"how bad could the next 5 min be?\"\n    - 25th percentile: moderate downside\n    - 50th percentile: expected return (median, not mean!)\n    - 75th percentile: moderate upside\n    - 90th percentile: \"how good could the next 5 min be?\"\n    \n    Trading signals from quantiles:\n    - Wide spread (90th - 10th): high uncertainty -> reduce position or skip\n    - Narrow spread: model is confident -> trade with larger size\n    - 10th percentile > 0: even the downside is positive -> strong buy\n    - 90th percentile < 0: even the upside is negative -> strong sell\n    - Asymmetric spread: if 90th-50th >> 50th-10th -> positive skew -> buy\n    \"\"\"\n    \n    def __init__(\n        self,\n        quantiles: List[float] = None,\n        base_params: dict = None,\n    ):\n        if quantiles is None:\n            quantiles = [0.10, 0.25, 0.50, 0.75, 0.90]\n        self.quantiles = sorted(quantiles)\n        \n        self.base_params = base_params or {\n            'num_leaves': 63,\n            'learning_rate': 0.03,\n            'min_child_samples': 200,\n            'feature_fraction': 0.7,\n            'bagging_fraction': 0.8,\n            'bagging_freq': 1,\n            'lambda_l2': 1.0,\n            'verbose': -1,\n        }\n        \n        self.models: Dict[float, lgb.Booster] = {}\n    \n    def train(\n        self,\n        X_train: np.ndarray,\n        y_train: np.ndarray,  # continuous returns (not classes!)\n        X_val: np.ndarray,\n        y_val: np.ndarray,\n        sample_weights: np.ndarray = None,\n    ):\n        \"\"\"\n        Train one model per quantile.\n        \n        IMPORTANT: y_train should be continuous returns (e.g., next-5min\n        return in basis points), NOT categorical labels. Quantile\n        regression requires a continuous target.\n        \"\"\"\n        train_data = lgb.Dataset(\n            X_train, label=y_train, weight=sample_weights,\n            free_raw_data=False,\n        )\n        val_data = lgb.Dataset(\n            X_val, label=y_val, reference=train_data,\n            free_raw_data=False,\n        )\n        \n        for q in self.quantiles:\n            params = self.base_params.copy()\n            params['objective'] = 'quantile'\n            params['alpha'] = q  # THIS sets which quantile\n            params['metric'] = 'quantile'\n            \n            logger.info(f\"Training quantile model: alpha={q}\")\n            \n            model = lgb.train(\n                params,\n                train_data,\n                num_boost_round=2000,\n                valid_sets=[val_data],\n                callbacks=[\n                    lgb.early_stopping(50),\n                    lgb.log_evaluation(200),\n                ],\n            )\n            \n            self.models[q] = model\n            logger.info(\n                f\"Quantile {q}: best_iteration={model.best_iteration}\"\n            )\n    \n    def predict(self, X: np.ndarray) -> Dict[float, np.ndarray]:\n        \"\"\"Predict all quantiles.\"\"\"\n        predictions = {}\n        for q, model in self.models.items():\n            predictions[q] = model.predict(X)\n        return predictions\n    \n    def predict_with_sorting(self, X: np.ndarray) -> Dict[float, np.ndarray]:\n        \"\"\"\n        Predict and fix quantile crossing.\n        \n        Quantile crossing: when the 25th percentile prediction is higher\n        than the 75th. This happens because each model is trained\n        independently. Solutions:\n        \n        1. Post-hoc sorting (simple, used here): sort predictions so\n           they respect quantile ordering. Fast but can distort individual\n           quantile calibration.\n        \n        2. Non-crossing constraints during training: more complex,\n           requires custom objective. See research by Cannon (2018).\n        \n        3. Joint training: train all quantiles together with a penalty\n           for crossing. Best theoretical properties but hard to implement\n           in LightGBM.\n        \"\"\"\n        raw_preds = self.predict(X)\n        n = X.shape[0]\n        \n        # Stack into matrix: (n_samples, n_quantiles)\n        pred_matrix = np.column_stack(\n            [raw_preds[q] for q in self.quantiles]\n        )\n        \n        # Sort each row (ensures non-crossing)\n        sorted_matrix = np.sort(pred_matrix, axis=1)\n        \n        # Check how much crossing occurred\n        crossing_count = 0\n        for i in range(n):\n            if not np.all(pred_matrix[i] == sorted_matrix[i]):\n                crossing_count += 1\n        \n        crossing_rate = crossing_count / n\n        if crossing_rate > 0.1:\n            logger.warning(\n                f\"Quantile crossing rate: {crossing_rate:.1%}. \"\n                f\"Models may need more regularization.\"\n            )\n        \n        result = {}\n        for j, q in enumerate(self.quantiles):\n            result[q] = sorted_matrix[:, j]\n        \n        return result\n    \n    def compute_confidence(self, X: np.ndarray) -> dict:\n        \"\"\"\n        Derive trading signals from quantile predictions.\n        \n        Returns dict with:\n        - 'spread': 90th - 10th (uncertainty measure)\n        - 'skew': asymmetry of the distribution\n        - 'signal': directional signal (-1 to +1)\n        - 'confidence': how confident the model is (0 to 1)\n        \"\"\"\n        preds = self.predict_with_sorting(X)\n        \n        p10 = preds[0.10]\n        p25 = preds[0.25]\n        p50 = preds[0.50]\n        p75 = preds[0.75]\n        p90 = preds[0.90]\n        \n        # Spread: uncertainty measure\n        spread = p90 - p10\n        \n        # IQR for robust spread\n        iqr = p75 - p25\n        \n        # Skewness of predicted distribution\n        # Positive = right-skewed (bullish)\n        upper = p90 - p50\n        lower = p50 - p10\n        skew = np.where(\n            spread > 1e-8,\n            (upper - lower) / spread,\n            0.0\n        )\n        \n        # Signal: median prediction (direction)\n        signal = np.tanh(p50 / (spread + 1e-8))  # normalize by spread\n        \n        # Confidence: inverse of spread, normalized\n        # Also require median to be far from zero\n        median_abs = np.abs(p50)\n        confidence = np.where(\n            spread > 1e-8,\n            median_abs / spread,\n            0.0\n        )\n        confidence = np.clip(confidence, 0, 1)\n        \n        return {\n            'spread': spread,\n            'iqr': iqr,\n            'skew': skew,\n            'signal': signal,\n            'confidence': confidence,\n            'p10': p10, 'p25': p25, 'p50': p50, 'p75': p75, 'p90': p90,\n        }\n\n\n# Usage\nensemble = QuantileLGBEnsemble(\n    quantiles=[0.10, 0.25, 0.50, 0.75, 0.90],\n)\nensemble.train(X_train, y_returns_train, X_val, y_returns_val)\n\n# Generate signals\nsignals = ensemble.compute_confidence(X_test)\n\n# Trading logic\nfor i in range(len(X_test)):\n    if signals['confidence'][i] > 0.6 and signals['p10'][i] > 0:\n        # Even the 10th percentile is positive: strong buy\n        pass\n    elif signals['confidence'][i] > 0.6 and signals['p90'][i] < 0:\n        # Even the 90th percentile is negative: strong sell\n        pass\n    elif signals['spread'][i] > np.percentile(signals['spread'], 90):\n        # Very uncertain: skip this bar\n        pass\n```\n\n---\n\n## 10. SHAP Analysis for LightGBM\n\nTreeSHAP is exact and fast for tree-based models (polynomial time, not exponential like KernelSHAP). For a LightGBM model, SHAP values are computed in milliseconds per prediction.\n\n```python\nimport shap\nimport numpy as np\nimport lightgbm as lgb\nfrom typing import Dict, Tuple\nimport logging\n\nlogger = logging.getLogger(__name__)\n\n\nclass CryptoSHAPAnalyzer:\n    \"\"\"\n    SHAP analysis for crypto LightGBM models.\n    \n    Three use cases:\n    1. Feature selection: which features matter most across all predictions?\n    2. Per-prediction debugging: why did the model predict \"up\" right now?\n    3. Confusion detection: when is the model uncertain/confused?\n    \"\"\"\n    \n    def __init__(self, model: lgb.Booster, feature_names: list):\n        self.model = model\n        self.feature_names = feature_names\n        self.explainer = shap.TreeExplainer(model)\n    \n    def compute_shap_values(\n        self, X: np.ndarray\n    ) -> np.ndarray:\n        \"\"\"\n        Compute SHAP values for all samples.\n        \n        For multiclass (3 classes), returns shape: (n_samples, n_features, 3)\n        For binary/regression, returns shape: (n_samples, n_features)\n        \n        Performance: ~1000 samples/second for 30 features, 1000 trees.\n        For a full 500K dataset, this takes ~8 minutes. Cache the results.\n        \"\"\"\n        shap_values = self.explainer.shap_values(X)\n        \n        # shap_values is a list of 3 arrays for multiclass\n        # Each array is (n_samples, n_features)\n        if isinstance(shap_values, list):\n            shap_values = np.stack(shap_values, axis=-1)\n        \n        return shap_values\n    \n    def feature_importance_shap(\n        self, X: np.ndarray, top_k: int = 15\n    ) -> Dict[str, float]:\n        \"\"\"\n        Global feature importance based on mean |SHAP|.\n        \n        This is better than LightGBM's built-in feature importance because:\n        1. Built-in 'gain' importance is biased toward high-cardinality features\n        2. Built-in 'split' importance counts appearances, not impact\n        3. SHAP importance accounts for feature interactions\n        4. SHAP importance is additive (importances sum to the prediction)\n        \"\"\"\n        shap_values = self.compute_shap_values(X)\n        \n        # For multiclass, average absolute SHAP across all classes\n        if shap_values.ndim == 3:\n            mean_abs_shap = np.abs(shap_values).mean(axis=(0, 2))\n        else:\n            mean_abs_shap = np.abs(shap_values).mean(axis=0)\n        \n        importance = dict(zip(self.feature_names, mean_abs_shap))\n        sorted_importance = dict(\n            sorted(importance.items(), key=lambda x: x[1], reverse=True)[:top_k]\n        )\n        \n        return sorted_importance\n    \n    def explain_single_prediction(\n        self, x: np.ndarray, prediction_class: int = None\n    ) -> Dict:\n        \"\"\"\n        Explain a single prediction.\n        \n        Returns the top contributing features for the predicted class.\n        Useful for: \"Why did the model predict UP at 2024-01-15 14:35?\"\n        \"\"\"\n        if x.ndim == 1:\n            x = x.reshape(1, -1)\n        \n        shap_values = self.compute_shap_values(x)\n        \n        if prediction_class is None:\n            # Use the predicted class\n            probs = self.model.predict(x)\n            prediction_class = probs.argmax(axis=1)[0]\n        \n        class_names = ['UP', 'FLAT', 'DOWN']\n        \n        # Get SHAP for the predicted class\n        if shap_values.ndim == 3:\n            sv = shap_values[0, :, prediction_class]\n        else:\n            sv = shap_values[0, :]\n        \n        # Sort by absolute contribution\n        sorted_idx = np.argsort(np.abs(sv))[::-1]\n        \n        explanation = {\n            'predicted_class': class_names[prediction_class],\n            'base_value': self.explainer.expected_value[prediction_class]\n                if isinstance(self.explainer.expected_value, list)\n                else self.explainer.expected_value,\n            'top_features': [],\n        }\n        \n        for idx in sorted_idx[:10]:\n            explanation['top_features'].append({\n                'feature': self.feature_names[idx],\n                'value': float(x[0, idx]),\n                'shap_value': float(sv[idx]),\n                'direction': 'pushes UP' if sv[idx] > 0 else 'pushes DOWN',\n            })\n        \n        return explanation\n    \n    def detect_model_confusion(\n        self,\n        X: np.ndarray,\n        threshold: float = 0.5,\n    ) -> np.ndarray:\n        \"\"\"\n        Detect when the model is \"confused\" using SHAP.\n        \n        A model is confused when:\n        1. Top features push in opposite directions (cancellation)\n        2. No single feature dominates the prediction\n        3. The SHAP values are all small (nothing drives the prediction)\n        \n        This is different from low prediction probability. A model can\n        predict class 1 with 40% probability confidently (other classes\n        are 30%, 30%) or confusedly (features cancel out to ~33% each).\n        \n        Returns a confusion score per sample (0 = clear, 1 = confused).\n        \"\"\"\n        shap_values = self.compute_shap_values(X)\n        \n        n_samples = X.shape[0]\n        confusion_scores = np.zeros(n_samples)\n        \n        for i in range(n_samples):\n            if shap_values.ndim == 3:\n                # Use the predicted class SHAP values\n                probs = self.model.predict(X[i:i+1])\n                pred_class = probs.argmax(axis=1)[0]\n                sv = shap_values[i, :, pred_class]\n            else:\n                sv = shap_values[i, :]\n            \n            # Metric 1: cancellation ratio\n            # If positive and negative SHAP values are similar in magnitude,\n            # features are fighting each other\n            pos_sum = sv[sv > 0].sum() if (sv > 0).any() else 0\n            neg_sum = abs(sv[sv < 0].sum()) if (sv < 0).any() else 0\n            total = pos_sum + neg_sum\n            \n            if total > 0:\n                cancellation = min(pos_sum, neg_sum) / total\n            else:\n                cancellation = 0.5  # no signal at all\n            \n            # Metric 2: concentration (entropy of |SHAP|)\n            abs_sv = np.abs(sv)\n            if abs_sv.sum() > 0:\n                p = abs_sv / abs_sv.sum()\n                entropy = -np.sum(p * np.log(p + 1e-10))\n                max_entropy = np.log(len(sv))\n                normalized_entropy = entropy / max_entropy\n            else:\n                normalized_entropy = 1.0  # maximum confusion\n            \n            # Combined confusion score\n            confusion_scores[i] = 0.5 * cancellation + 0.5 * normalized_entropy\n        \n        return confusion_scores\n    \n    def shap_feature_selection(\n        self,\n        X_train: np.ndarray,\n        y_train: np.ndarray,\n        X_val: np.ndarray,\n        y_val: np.ndarray,\n        min_features: int = 10,\n        params: dict = None,\n    ) -> list:\n        \"\"\"\n        Use SHAP for iterative feature selection.\n        \n        Strategy: \n        1. Compute SHAP importance on full feature set\n        2. Remove bottom 20% features\n        3. Retrain and check if performance improves\n        4. Repeat until performance degrades or min_features reached\n        \n        This is better than LightGBM's built-in feature importance\n        because SHAP accounts for feature interactions. A feature with\n        low individual importance might be critical for interaction\n        effects, and SHAP captures this.\n        \"\"\"\n        if params is None:\n            params = {\n                'objective': 'multiclass', 'num_class': 3,\n                'num_leaves': 63, 'learning_rate': 0.03,\n                'min_child_samples': 200, 'verbose': -1,\n            }\n        \n        current_features = list(range(X_train.shape[1]))\n        current_names = list(self.feature_names)\n        best_accuracy = 0\n        best_features = current_features.copy()\n        \n        while len(current_features) > min_features:\n            # Train on current features\n            X_tr = X_train[:, current_features]\n            X_vl = X_val[:, current_features]\n            \n            train_data = lgb.Dataset(X_tr, label=y_train)\n            val_data = lgb.Dataset(X_vl, label=y_val, reference=train_data)\n            \n            model = lgb.train(\n                params, train_data, num_boost_round=1000,\n                valid_sets=[val_data],\n                callbacks=[lgb.early_stopping(30), lgb.log_evaluation(0)],\n            )\n            \n            preds = model.predict(X_vl).argmax(axis=1)\n            accuracy = (preds == y_val).mean()\n            \n            logger.info(\n                f\"Features: {len(current_features)}, accuracy: {accuracy:.4f}\"\n            )\n            \n            if accuracy > best_accuracy:\n                best_accuracy = accuracy\n                best_features = current_features.copy()\n            elif accuracy < best_accuracy - 0.005:\n                # Performance dropped meaningfully, stop\n                logger.info(\"Stopping: performance degraded\")\n                break\n            \n            # Compute SHAP importance\n            explainer = shap.TreeExplainer(model)\n            sv = explainer.shap_values(X_vl[:1000])  # subsample for speed\n            if isinstance(sv, list):\n                sv = np.stack(sv, axis=-1)\n            mean_importance = np.abs(sv).mean(axis=(0, -1)) if sv.ndim == 3 else np.abs(sv).mean(axis=0)\n            \n            # Remove bottom 20%\n            n_remove = max(1, len(current_features) // 5)\n            keep_idx = np.argsort(mean_importance)[n_remove:]\n            \n            current_features = [current_features[i] for i in keep_idx]\n            current_names = [current_names[i] for i in keep_idx]\n        \n        logger.info(\n            f\"Selected {len(best_features)} features. \"\n            f\"Best accuracy: {best_accuracy:.4f}\"\n        )\n        \n        return [self.feature_names[i] for i in best_features]\n```\n\n---\n\n## 11.
