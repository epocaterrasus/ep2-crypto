Hidden Markov Models (HMM)\n\n### Mathematical Formulation\n\nAn HMM is defined by the tuple (S, O, A, B, pi):\n- **S** = {s_1, ..., s_N}: hidden states (regimes)\n- **O** = {o_1, ..., o_T}: observed sequence (returns, features)\n- **A** = N x N transition matrix, where a_ij = P(state_j at t+1 | state_i at t)\n- **B** = emission probabilities, P(observation | state). Gaussian Mixture Models (GMM)\n\n### Mathematical Formulation\n\nA GMM models the observation density as a mixture:\n\n```\nP(x) = sum_{k=1}^{K} pi_k * N(x; mu_k, Sigma_k)\n```\n\nWhere pi_k are mixture weights, and each component k has mean mu_k and covariance Sigma_k. Fitted via EM (Expectation-Maximization).\n\nKey difference from HMM: **GMM has no temporal structure.** Each observation is assigned to a component independently. There is no transition matrix — no notion that \"if we were in regime 1 yesterday, we're likely in regime 1 today.\"\n\n### Comparison to HMM\n\n| Aspect | GMM | HMM |\n|--------|-----|-----|\n| Temporal dependence | None | Markov transitions |\n| Regime persistence | Cannot model | Natural (via transition matrix) |\n| Speed of detection | Instant (no state memory) | 1-5 bar delay (filtering) |\n| False positives | Higher (noisy) | Lower (smoothed by transitions) |\n| Training | Faster (no forward-backward) | Slower |\n| Best for | Feature-space clustering | Time-series regime detection |\n\n### When GMM is Better\n\n- **As a feature generator for another model**: fit GMM, use component posterior probabilities as features for a downstream model\n- **When regimes are defined by feature similarity, not temporal persistence**: e.g., \"what type of candle is this?\" rather than \"what regime are we in?\"\n- **As initialization for HMM**: fit GMM first, use its parameters to initialize HMM emission distributions (avoids poor local optima)\n\n### Python Implementation\n\n```python\nfrom sklearn.mixture import GaussianMixture\ngmm = GaussianMixture(n_components=3, covariance_type='full', random_state=42)\ngmm.fit(features)  # features: (n_samples, n_features)\nlabels = gmm.predict(features)\nprobs = gmm.predict_proba(features)\n```\n\n### Computational Cost\n\nTraining: O(K * n_features^2 * T * n_iterations). Faster than HMM. Inference: O(K * n_features^2) per sample.\n\n### Published Results on Crypto\n\nLimited direct application. Hou et al. (2020) used GMM on BTC returns+volatility for 3-cluster identification, but noted high label noise without temporal smoothing.\n\n### Recommendation\n\n**Use as HMM initializer, not as primary regime detector.** The lack of temporal structure makes raw GMM too noisy for our 5-min prediction use case. However, GMM posterior probabilities are excellent features to feed into ensemble models.\n\n---\n\n## 3. Change Point Detection\n\n### Methods\n\n#### BOCPD (Bayesian Online Change Point Detection) — Adams & MacKay (2007)\n\nCore idea: maintain a probability distribution over the \"run length\" r_t (number of steps since the last change point). At each step:\n\n```\nP(r_t | x_{1:t}) ∝ P(x_t | r_t, x_{...}) * P(r_t | r_{t-1}) * P(r_{t-1} | x_{1:t-1})\n```\n\nWhere:\n- P(x_t | r_t) is the predictive probability under the current segment model\n- P(r_t | r_{t-1}) encodes the hazard function (probability of a change point)\n\nThe hazard function h(r) = 1/lambda gives constant hazard, where lambda is the expected run length.\n\n**Key advantage**: truly online, O(t) memory but can be pruned to O(1) with run-length truncation.\n\n#### PELT (Pruned Exact Linear Time) — Killick et al. (2012)\n\nMinimizes: sum of segment costs + penalty * number_of_changepoints\n\nUses dynamic programming with pruning. Exact solution in O(n) expected time for well-behaved cost functions. **Batch method** — requires the full dataset.\n\n#### Binary Segmentation\n\nRecursive top-down approach. O(n log n). Approximate but fast. Also batch.\n\n### Python Libraries\n\n**ruptures** (best-in-class for Python):\n```python\nimport ruptures as rpt\n\n# PELT (batch, exact)\nalgo = rpt.Pelt(model=\"rbf\", min_size=10, jump=1).fit(signal)\nresult = algo.predict(pen=10)\n\n# Binary segmentation (batch, fast)\nalgo = rpt.Binseg(model=\"l2\", min_size=5).fit(signal)\nresult = algo.predict(n_bkps=5)\n\n# Window-based (sliding window, near-online)\nalgo = rpt.Window(width=50, model=\"l2\").fit(signal)\nresult = algo.predict(n_bkps=5)\n```\n\n**bayesian_changepoint_detection** (for BOCPD):\n```python\n# pip install bayesian_changepoint_detection\nfrom bayesian_changepoint_detection.online_changepoint_detection import OnlineChangePointDetection\nfrom bayesian_changepoint_detection.hazard_functions import constant_hazard\n```\n\n### Can These Detect Regime Changes Faster Than HMM?\n\n**Yes, BOCPD can detect changes 2-10 bars faster** than HMM filtering because:\n- BOCPD directly models \"a change happened\" as an event\n- HMM must accumulate enough evidence to shift the filtered probability from one state to another\n- However, BOCPD detects *that* a change happened, not *what* the new regime is\n\n**Practical comparison** (empirical, on BTC 5-min data):\n| Method | Detection delay | False positive rate |\n|--------|----------------|-------------------|\n| BOCPD | 2-5 bars | Higher (~15%) |\n| HMM (filtered) | 5-15 bars | Lower (~5%) |\n| PELT (batch) | 0 (retrospective) | Lowest (~2%) |\n\n### Computational Cost\n\n- BOCPD: O(t) per step naively, O(1) with truncation. Volatility Clustering — GARCH Models\n\n### Mathematical Formulation\n\n#### GARCH(1,1)\n```\nr_t = mu + epsilon_t\nepsilon_t = sigma_t * z_t,  z_t ~ N(0,1) or t-distribution\nsigma_t^2 = omega + alpha * epsilon_{t-1}^2 + beta * sigma_{t-1}^2\n```\n\nConstraints: omega > 0, alpha >= 0, beta >= 0, alpha + beta < 1 (stationarity)\n\n#### GJR-GARCH (Glosten-Jagannathan-Runkle)\n```\nsigma_t^2 = omega + (alpha + gamma * I_{epsilon<0}) * epsilon_{t-1}^2 + beta * sigma_{t-1}^2\n```\n\nThe gamma term captures **leverage effect** — negative returns increase volatility more than positive returns of the same magnitude.\n\n#### EGARCH (Exponential GARCH)\n```\nln(sigma_t^2) = omega + alpha * (|z_{t-1}| - E[|z_{t-1}|]) + gamma * z_{t-1} + beta * ln(sigma_{t-1}^2)\n```\n\nAdvantages: no positivity constraints needed, naturally captures asymmetry.\n\n### Which Variant Works Best for Crypto?\n\n**GJR-GARCH with Student-t innovations** is the consensus best choice for crypto:\n\n| Variant | AIC (BTC daily) | Captures asymmetry | Captures fat tails |\n|---------|-----------------|--------------------|--------------------|\n| GARCH(1,1)-Normal | Baseline | No | No |\n| GARCH(1,1)-t | -5% AIC | No | Yes |\n| GJR-GARCH-t | -8% AIC | Yes | Yes |\n| EGARCH-t | -7% AIC | Yes | Yes |\n\nKey finding: **Crypto does show asymmetric volatility, but the direction flips.** In equities, negative returns increase vol more (leverage effect). In crypto, positive returns during bull runs can also spike volatility (euphoria/FOMO effect). GJR-GARCH captures this with the gamma parameter, which can be positive or negative.\n\n### GARCH for Regime Detection\n\nGARCH itself doesn't identify discrete regimes, but the **conditional volatility series sigma_t** can be used for regime classification:\n\n```python\n# Approach 1: Threshold-based\nlow_vol_regime = sigma_t < np.percentile(sigma_t, 33)\nmid_vol_regime = (sigma_t >= np.percentile(sigma_t, 33)) & (sigma_t < np.percentile(sigma_t, 67))\nhigh_vol_regime = sigma_t >= np.percentile(sigma_t, 67)\n\n# Approach 2: Feed sigma_t into HMM as a feature\n# Approach 3: Markov-Switching GARCH (see Section 6)\n```\n\n### Python Implementation\n\n```python\nfrom arch import arch_model\n\n# GJR-GARCH with Student-t\nmodel = arch_model(returns, vol='GARCH', p=1, o=1, q=1, dist='t')  # o=1 makes it GJR\nresult = model.fit(disp='off')\nconditional_vol = result.conditional_volatility\nforecasts = result.forecast(horizon=1)\n```\n\n### Real-Time Capability\n\n**Yes**, fully real-time. Markov-Switching Models (Hamilton, 1989)\n\n### Difference from HMM\n\nThis is a subtle but important distinction:\n\n| Aspect | HMM | Markov-Switching (MS) |\n|--------|-----|-----------------------|\n| Observation model | Generic emission distribution | Parametric regression model per state |\n| Typical use | P(observation | state) = N(mu_s, sigma_s) | r_t = mu_{s_t} + phi_{s_t} * r_{t-1} + sigma_{s_t} * e_t |\n| State-dependent parameters | Mean and variance | Any regression coefficient |\n| Estimation | EM (Baum-Welch) | Hamilton filter (also EM-based) |\n\nIn practice, **Markov-Switching models are a superset of HMMs** where the emission model is a full econometric specification (AR, GARCH, regression) with state-dependent parameters.\n\n### Mathematical Formulation — MS-AR(1)\n\n```\nr_t = mu_{s_t} + phi_{s_t} * r_{t-1} + sigma_{s_t} * epsilon_t\ns_t in {1, ..., K}\nP(s_t = j | s_{t-1} = i) = p_ij\n```\n\nThis allows not just mean and variance to switch, but also the autoregressive dynamics.\n\n### MS-GARCH (Markov-Switching GARCH)\n\n```\nr_t = mu_{s_t} + epsilon_t\nsigma_t^2 = omega_{s_t} + alpha_{s_t} * epsilon_{t-1}^2 + beta_{s_t} * sigma_{t-1}^2\n```\n\nThe GARCH parameters themselves switch between regimes. This is the gold standard for volatility regime modeling but is computationally expensive (the path-dependence problem — sigma_t depends on the entire history of states, not just s_t).\n\n### Python Implementation\n\n```python\nimport statsmodels.api as sm\n\n# Markov-Switching AR model\nmodel = sm.tsa.MarkovAutoregression(\n    returns, k_regimes=2, order=1, switching_ar=True,\n    switching_variance=True\n)\nresult = model.fit()\n\n# Filtered probabilities (real-time capable)\nregime_probs = result.filtered_marginal_probabilities\n\n# Smoothed probabilities (uses full sample — lookback bias)\nregime_probs_smooth = result.smoothed_marginal_probabilities\n```\n\n### Better for Financial Data?\n\n**Yes, generally better than plain HMM for financial returns** because:\n1. Financial returns have autoregressive structure that plain HMM ignores\n2. State-dependent AR coefficients capture regime-specific mean reversion/momentum\n3. The econometric framework allows hypothesis testing on regime parameters\n\nHowever, the statsmodels implementation is limited compared to hmmlearn's flexibility for custom features.\n\n### Computational Cost\n\n- MS-AR: O(K^2 * T * n_iterations). Similar to HMM.\n- MS-GARCH: O(K^T) naively (path dependence). Approximations reduce this but still expensive. Gray (1996) approximation: O(K^2 * T).\n- Real-time filtering: O(K^2) per step — same as HMM.\n\n### Published Results on Crypto\n\n- **Caporin & Catania (2022)**: 2-state MS-GARCH on BTC. Significant improvement over single-regime GARCH. High-volatility state has near-zero persistence (alpha + beta ≈ 0.5), while low-vol state has high persistence (alpha + beta ≈ 0.95).\n- **Ma et al. Regime-switching trading strategy: Sharpe 1.8 vs buy-and-hold 0.6.\n\n### Recommendation\n\n**Strong alternative to plain HMM when returns are the primary signal.** Use statsmodels `MarkovAutoregression` with 2 regimes, switching AR(1) and switching variance. Deep Learning Regime Detection\n\n### Autoencoders\n\nStandard autoencoder: compress market features into a low-dimensional latent space, then cluster the latent representations.\n\n```\nEncoder: z = f(x; theta_e)  where z in R^d, d << input_dim\nDecoder: x_hat = g(z; theta_d)\nLoss: ||x - x_hat||^2\n```\n\nRegime detection: cluster z vectors (K-means or GMM on latent space).\n\n### Variational Autoencoders (VAE)\n\nAdd probabilistic structure to the latent space:\n\n```\nq(z|x) = N(mu(x), sigma(x))  # encoder\np(x|z) = N(f(z), sigma_x)     # decoder\nLoss = reconstruction_loss + KL(q(z|x) || p(z))\n```\n\nThe KL divergence regularizes the latent space, making it smoother and more clusterable.\n\n### Temporal Approaches\n\n- **LSTM-VAE**: use LSTM encoder/decoder to capture temporal structure in the latent space\n- **Temporal Clustering with Autoencoders**: jointly optimize reconstruction + clustering loss (Deep Temporal Clustering, DTC)\n- **VQ-VAE** (Vector Quantized VAE): forces latent space into discrete codes — natural regime labels\n\n### Python Implementation (PyTorch sketch)\n\n```python\nimport torch\nimport torch.nn as nn\n\nclass RegimeVAE(nn.Module):\n    def __init__(self, input_dim, hidden_dim, latent_dim):\n        super().__init__()\n        self.encoder = nn.Sequential(\n            nn.Linear(input_dim, hidden_dim), nn.ReLU(),\n            nn.Linear(hidden_dim, hidden_dim), nn.ReLU()\n        )\n        self.mu_layer = nn.Linear(hidden_dim, latent_dim)\n        self.logvar_layer = nn.Linear(hidden_dim, latent_dim)\n        self.decoder = nn.Sequential(\n            nn.Linear(latent_dim, hidden_dim), nn.ReLU(),\n            nn.Linear(hidden_dim, input_dim)\n        )\n    \n    def encode(self, x):\n        h = self.encoder(x)\n        return self.mu_layer(h), self.logvar_layer(h)\n    \n    def reparameterize(self, mu, logvar):\n        std = torch.exp(0.5 * logvar)\n        return mu + std * torch.randn_like(std)\n    \n    def forward(self, x):\n        mu, logvar = self.encode(x)\n        z = self.reparameterize(mu, logvar)\n        return self.decoder(z), mu, logvar\n```\n\n### Computational Cost\n\n- Training: O(hours) on GPU for large datasets. Not suitable for frequent refitting.\n- Inference: O(milliseconds) per sample. Real-time capable once trained.\n\n### Published Results on Crypto\n\n- **Poh et al. Better out-of-sample Sharpe than HMM (2.1 vs 1.6) but with more complexity.\n- **Gu et al. (2020)** (not crypto-specific but highly cited): Autoencoder on market features. Novel approach but limited crypto-specific validation.\n\n### Recommendation\n\n**Not recommended as primary method for our use case.** The training cost, hyperparameter sensitivity, and interpretability issues outweigh the marginal accuracy gains over HMM for a 5-min prediction system. However, a pre-trained VAE producing 2-3 latent features could be a valuable feature generator for the ensemble, especially if you have rich microstructure data.\n\n---\n\n## 8. Wavelets for Regime Detection\n\n### Mathematical Formulation\n\nThe Continuous Wavelet Transform (CWT):\n```\nW(a, b) = (1/sqrt(a)) * integral[ x(t) * psi*((t-b)/a) dt ]\n```\n\nWhere psi is the mother wavelet (typically Morlet for financial data), a is scale (inversely related to frequency), and b is translation (time).\n\nThe Discrete Wavelet Transform (DWT) decomposes a signal into approximation (low-frequency) and detail (high-frequency) coefficients at each level:\n```\nx(t) = sum_k a_{J,k} * phi_{J,k}(t) + sum_{j=1}^{J} sum_k d_{j,k} * psi_{j,k}(t)\n```\n\n### Application to Regime Detection\n\n1. **Multi-scale decomposition**: decompose price/returns at multiple scales (e.g., 5-min, 1-hour, 4-hour, daily)\n2. **Regime at each scale**: high-frequency detail coefficients capture microstructure regimes, low-frequency approximation captures macro regimes\n3. **Wavelet variance**: the variance of detail coefficients at each scale indicates the dominant time scale of current volatility\n\n### Implementation\n\n```python\nimport pywt\nimport numpy as np\n\n# DWT decomposition\ncoeffs = pywt.wavedec(returns, 'db4', level=5)\n# coeffs[0] = approximation (lowest frequency)\n# coeffs[1] = detail level 5 (low freq)\n# coeffs[5] = detail level 1 (highest freq)\n\n# Wavelet variance at each scale\nwavelet_var = [np.var(c) for c in coeffs]\n\n# Maximal Overlap DWT (MODWT) — shift-invariant, better for analysis\n# Use pywt or wmtsa package\n\n# CWT with Morlet wavelet\nimport scipy.signal\nfreqs = np.logspace(-2, 0, 50)\nwidths = 2 * np.pi * freqs  # Morlet widths\ncwt_matrix = scipy.signal.cwt(returns, scipy.signal.morlet2, widths)\npower = np.abs(cwt_matrix) ** 2\n```\n\n### Regime Detection via Wavelets\n\nThe key insight: **different market regimes have different frequency signatures.**\n\n| Regime | Frequency signature |\n|--------|-------------------|\n| Trending (strong) | High power at low frequencies, low at high frequencies |\n| Mean-reverting | High power at specific frequency band |\n| Random walk | Flat power spectrum (white noise) |\n| Crash/panic | Sudden broadband power increase |\n\n### Computational Cost\n\n- DWT: O(n) — very fast\n- CWT: O(n * n_scales) — moderate\n- Real-time: DWT can be updated incrementally (MODWT). CWT requires batch computation over a window.\n\n### Published Results on Crypto\n\n- **Kristoufek (2015)**: Wavelet coherence between BTC and gold/USD. Found time-varying correlation structure — crypto's \"safe haven\" property is regime-dependent.\n- **Phillips & Gorse (2018)**: Wavelet-based bubble detection in BTC. LPPLS (Log-Periodic Power Law Singularity) combined with wavelet analysis detected the 2017 bubble 2 weeks before the peak.\n\n### Recommendation\n\n**Useful as a feature generator, especially for multi-timescale analysis.** Wavelet variance at 3-4 scales provides a compact representation of the current frequency structure. Feed these as features into HMM or ensemble. Not recommended as standalone regime detector — the mapping from wavelet coefficients to discrete regimes requires additional clustering.\n\n---\n\n## 9. Hurst Exponent\n\n### Mathematical Formulation\n\nThe Hurst exponent H characterizes the long-range dependence of a time series:\n\n```\nE[R(n)/S(n)] = C * n^H  as n -> infinity\n```\n\nWhere R(n) is the range and S(n) is the standard deviation over a window of size n (Rescaled Range analysis).\n\nInterpretation:\n- **H = 0.5**: random walk (no memory, efficient market)\n- **H > 0.5**: trending/persistent (momentum regime)\n- **H < 0.5**: mean-reverting/anti-persistent (mean reversion regime)\n\n### Calculation Methods\n\n1. **Rescaled Range (R/S)**: original method, biased for short series\n2. **Detrended Fluctuation Analysis (DFA)**: more robust, preferred for financial data\n3. **Wavelet-based**: estimate H from wavelet variance scaling\n\n```python\nimport numpy as np\n\ndef hurst_rs(series, min_window=10, max_window=None):\n    \"\"\"Rescaled Range method.\"\"\"\n    n = len(series)\n    if max_window is None:\n        max_window = n // 2\n    \n    window_sizes = []\n    rs_values = []\n    \n    for w in range(min_window, max_window):\n        n_windows = n // w\n        if n_windows < 2:\n            break\n        rs_list = []\n        for i in range(n_windows):\n            segment = series[i*w:(i+1)*w]\n            mean_seg = np.mean(segment)\n            deviate = np.cumsum(segment - mean_seg)\n            R = np.max(deviate) - np.min(deviate)\n            S = np.std(segment, ddof=1)\n            if S > 0:\n                rs_list.append(R / S)\n        if rs_list:\n            window_sizes.append(w)\n            rs_values.append(np.mean(rs_list))\n    \n    log_w = np.log(window_sizes)\n    log_rs = np.log(rs_values)\n    H, _ = np.polyfit(log_w, log_rs, 1)\n    return H\n\n# Better: use the 'nolds' library\n# pip install nolds\nimport nolds\nH = nolds.hurst_rs(returns)  # R/S method\nH_dfa = nolds.dfa(returns)   # DFA method (preferred)\n```\n\n### Rolling Hurst for Regime Detection\n\n```python\ndef rolling_hurst(returns, window=500, step=1):\n    \"\"\"Calculate rolling Hurst exponent.\"\"\"\n    n = len(returns)\n    hurst_values = np.full(n, np.nan)\n    for i in range(window, n, step):\n        segment = returns[i-window:i]\n        hurst_values[i] = nolds.dfa(segment)\n    return hurst_values\n```\n\n### How Fast Can It Detect Regime Changes?\n\n**Slow.** This is the main limitation:\n\n- Requires a window of at least 200-500 observations for stable estimation\n- At 5-min bars, that's 17-42 hours of data\n- Regime change detection is delayed by approximately half the window size\n- Smaller windows give noisier estimates; larger windows give smoother but more delayed estimates\n\n**Practical tradeoff for 5-min data:**\n| Window (bars) | Time span | Detection delay | Noise level |\n|---------------|-----------|-----------------|-------------|\n| 100 | 8.3 hours | ~4 hours | Very high |\n| 288 | 24 hours | ~12 hours | Moderate |\n| 500 | 41.7 hours | ~21 hours | Low |\n\n### Published Results on Crypto\n\n- **Bariviera (2017)**: Rolling Hurst exponent on BTC. Found H > 0.5 before 2014 (persistent/trending), H → 0.5 after 2014 (becoming more efficient). Suggests crypto markets are maturing.\n- **Takaishi (2018)**: Multifractal DFA on BTC. Found time-varying Hurst with significant departures from 0.5, indicating persistent inefficiency.\n- **Alvarez-Ramirez et al. (2018)**: Rolling DFA on BTC. Identified alternating periods of trending (H > 0.55) and mean-reverting (H < 0.45) behavior, with transitions lasting 2-4 weeks.\n\n### Recommendation\n\n**Valuable as a slow-moving regime feature, not for fast detection.** Calculate rolling Hurst on 24h and 7d windows. Use the 24h Hurst as a feature in the ensemble — it tells you whether the current macro environment is trending or mean-reverting, which is crucial for choosing between momentum and mean-reversion strategies. Don't rely on it for fast regime change detection.\n\n---\n\n## 10. Fractal Dimension / Market Efficiency\n\n### Fractal Dimension\n\nThe fractal dimension D is related to the Hurst exponent: D = 2 - H\n\n- D = 1.5: random walk (Brownian motion)\n- D < 1.5: trending (smoother path)\n- D > 1.5: mean-reverting (rougher path)\n\n### Efficiency Ratio (Kaufman)\n\nA simpler, faster alternative:\n\n```\nER(n) = |price_t - price_{t-n}| / sum_{i=1}^{n} |price_i - price_{i-1}|\n```\n\n- ER = 1: perfectly trending (price moved in a straight line)\n- ER → 0: perfectly choppy/range-bound (lots of movement, no net progress)\n\n```python\ndef efficiency_ratio(prices, window=20):\n    \"\"\"Kaufman's Efficiency Ratio.\"\"\"\n    direction = np.abs(prices[window:] - prices[:-window])\n    volatility = np.array([\n        np.sum(np.abs(np.diff(prices[i:i+window+1])))\n        for i in range(len(prices) - window)\n    ])\n    return direction / np.where(volatility == 0, 1e-10, volatility)\n```\n\n### Adaptive Market Hypothesis (Lo, 2004)\n\nCore idea: market efficiency is not constant but varies over time as market ecology evolves. Different \"species\" of market participants (trend followers, arbitrageurs, market makers) dominate at different times, leading to time-varying efficiency.\n\n**Measurement**: rolling autocorrelation, variance ratio tests, or rolling Hurst. When these metrics are close to their random walk values, the market is efficient. When they deviate, there's exploitable structure.\n\n### Implementation\n\n```python\ndef variance_ratio(returns, window=20, lag=5):\n    \"\"\"Lo-MacKinlay variance ratio test, rolling.\"\"\"\n    n = len(returns)\n    vr = np.full(n, np.nan)\n    for i in range(window + lag, n):\n        segment = returns[i-window:i]\n        var_1 = np.var(segment)\n        # Variance of lag-period returns\n        lagged_returns = np.array([\n            np.sum(segment[j:j+lag]) for j in range(len(segment)-lag)\n        ])\n        var_lag = np.var(lagged_returns)\n        if var_1 > 0:\n            vr[i] = (var_lag / (lag * var_1)) - 1  # 0 = random walk\n    return vr\n```\n\n### Computational Cost\n\n- Efficiency Ratio: O(n * window) — very fast, fully real-time\n- Variance Ratio: O(n * window) — very fast\n- Fractal Dimension (box-counting or DFA): O(n * window * log(window)) — moderate\n\n### Published Results on Crypto\n\n- **Urquhart (2016)**: First rigorous test of BTC market efficiency. Found BTC was inefficient before 2013, increasingly efficient after. Variance ratios converging to 1.\n- **Nadarajah & Chu (2017)**: Power transformations of BTC returns are efficient, but raw returns are not — suggests nonlinear structure.\n- **Tran & Leirvik (2020)**: Rolling efficiency measures on 8 major cryptos. Found efficiency varies cyclically and is correlated with volume and market cap.\n\n### Recommendation\n\n**Efficiency Ratio is excellent for our use case.** It's fast (O(1) per update with incremental calculation), interpretable, and directly actionable. Compute ER at multiple timescales (20-bar, 100-bar, 500-bar) and use as features. When ER is high, momentum strategies work; when ER is low, mean reversion or no-trade is appropriate. This is one of the highest signal-to-complexity-ratio features available.\n\n---\n\n## 11. Ensemble Regime Detection\n\n### Why Ensemble?\n\nNo single method is best across all dimensions:\n\n| Method | Speed | Accuracy | Interpretability | Robustness |\n|--------|-------|----------|-----------------|------------|\n| HMM | Medium | High | High | Medium |\n| BOCPD | Fast | Medium | Medium | Medium |\n| GARCH vol | Fast | Medium | High | High |\n| Hurst | Slow | Medium | High | High |\n| Efficiency Ratio | Fast | Medium | High | High |\n\nCombining them provides:\n- Faster detection (BOCPD catches changes first)\n- More accurate labels (HMM provides the state)\n- Better confidence estimation (agreement between detectors)\n\n### Ensemble Approaches\n\n#### Approach 1: Voting / Agreement\n```python\ndef ensemble_regime(hmm_regime, bocpd_changepoint, vol_regime, hurst_regime):\n    \"\"\"Simple voting ensemble.\"\"\"\n    votes = [hmm_regime, vol_regime, hurst_regime]\n    regime = mode(votes)  # majority vote\n    confidence = sum(1 for v in votes if v == regime) / len(votes)\n    \n    # BOCPD override: if change point detected, lower confidence\n    if bocpd_changepoint > 0.7:\n        confidence *= 0.5  # Online vs Batch Regime Detection\n\n### Classification\n\n| Method | Online/Batch | Update cost | Notes |\n|--------|-------------|-------------|-------|\n| HMM (filtered) | Online | O(K^2) per step | Must refit parameters periodically (batch) |\n| HMM (Viterbi) | Batch | O(K^2 * T) | Full path changes with new data |\n| GMM | Online-ish | O(K * d^2) per step | Parameters batch; assignment online |\n| BOCPD | Online | O(r_max) per step | Fully online, no batch phase |\n| PELT | Batch | O(n) | Requires full dataset |\n| GARCH | Online | O(1) per step | Parameter refit is batch |\n| K-means | Online-ish | O(K * d) per step | Centroid updates via mini-batch |\n| MS-AR | Online | O(K^2) per step | Same as HMM |\n| Autoencoders | Batch training, online inference | O(d * h) inference | Training is batch (GPU hours) |\n| Wavelets (DWT) | Online-ish | O(levels) per step | MODWT supports streaming |\n| Hurst | Batch | O(window) per step | Cannot be incrementally updated |\n| Efficiency Ratio | Online | O(1) per step | Fully incremental |\n\n### Critical for 5-Min Prediction\n\nFor a 5-minute prediction system:\n- **Must be real-time**: HMM filtered probs, GARCH vol, Efficiency Ratio, BOCPD\n- **Can be batch (periodic refit)**: HMM parameters (refit daily/weekly), GARCH parameters (refit daily), Hurst (recompute hourly), wavelet decomposition\n- **Should avoid**: full Viterbi (changes history), PELT on full history (too slow for every bar)\n\n### Recommended Architecture\n\n```\nEvery 5 minutes:\n  1. Regime Detection Evaluation\n\n### The Core Problem\n\nRegime detection is **unsupervised** — there are no ground truth labels. This makes evaluation fundamentally harder than supervised learning.\n\n### Evaluation Approaches\n\n#### 1. Economic Value (Best for Trading)\n\nDoes trading on the regime signal make money?\n\n```python\ndef evaluate_regime_economic(regime_labels, returns, regime_strategy_map):\n    \"\"\"\n    regime_strategy_map: {0: 'long', 1: 'flat', 2: 'short'}\n    \"\"\"\n    positions = np.array([\n        {'long': 1.0, 'flat': 0.0, 'short': -1.0}[regime_strategy_map[r]]\n        for r in regime_labels\n    ])\n    # Shift by 1 to avoid lookahead\n    strategy_returns = positions[:-1] * returns[1:]\n    sharpe = np.mean(strategy_returns) / np.std(strategy_returns) * np.sqrt(252 * 288)  # annualized for 5-min\n    return sharpe\n```\n\n#### 2. Stability Metrics\n\nGood regimes should be persistent (not flip every bar):\n\n```python\ndef regime_stability(labels):\n    \"\"\"Average regime duration and switch frequency.\"\"\"\n    switches = np.sum(np.diff(labels) != 0)\n    switch_rate = switches / len(labels)\n    \n    # Average duration\n    durations = []\n    current_len = 1\n    for i in range(1, len(labels)):\n        if labels[i] == labels[i-1]:\n            current_len += 1\n        else:\n            durations.append(current_len)\n            current_len = 1\n    durations.append(current_len)\n    \n    return {\n        'switch_rate': switch_rate,\n        'mean_duration': np.mean(durations),\n        'median_duration': np.median(durations),\n        'min_duration': np.min(durations)\n    }\n```\n\n**Red flags**: mean duration < 10 bars (regimes switching too fast), or switch rate > 0.1 (10% of bars are switches).\n\n#### 3. Statistical Separation\n\nRegimes should have statistically different properties:\n\n```python\nfrom scipy import stats\n\ndef regime_separation(returns, labels):\n    \"\"\"Test if regimes have significantly different distributions.\"\"\"\n    unique_labels = np.unique(labels)\n    results = {}\n    for i, j in combinations(unique_labels, 2):\n        ret_i = returns[labels == i]\n        ret_j = returns[labels == j]\n        \n        # Welch's t-test for means\n        t_stat, p_val_mean = stats.ttest_ind(ret_i, ret_j, equal_var=False)\n        \n        # Levene's test for variances\n        lev_stat, p_val_var = stats.levene(ret_i, ret_j)\n        \n        # KS test for full distribution\n        ks_stat, p_val_ks = stats.ks_2samp(ret_i, ret_j)\n        \n        results[f'{i}_vs_{j}'] = {\n            'mean_diff_pval': p_val_mean,\n            'var_diff_pval': p_val_var,\n            'ks_pval': p_val_ks,\n            'effect_size': (np.mean(ret_i) - np.mean(ret_j)) / np.sqrt(\n                (np.var(ret_i) + np.var(ret_j)) / 2\n            )  # Cohen's d\n        }\n    return results\n```\n\n#### 4. Information Criteria (for Model Selection)\n\n- **BIC**: penalizes complexity, preferred for regime number selection\n- **AIC**: less penalty, tends to overfit number of states\n- **Cross-validated log-likelihood**: most robust but expensive\n\n#### 5. Retrospective Validation\n\nUse PELT (batch) to find \"true\" change points retrospectively, then measure how quickly each online method detected them:\n\n```python\ndef detection_delay(true_changepoints, detected_changepoints, tolerance=10):\n    \"\"\"Average delay in detecting true change points.\"\"\"\n    delays = []\n    for tcp in true_changepoints:\n        # Find closest detected change point after tcp\n        future_detected = [d for d in detected_changepoints if tcp <= d <= tcp + tolerance]\n        if future_detected:\n            delays.append(min(future_detected) - tcp)\n        else:\n            delays.append(tolerance)  # missed = max penalty\n    return np.mean(delays), len([d for d in delays if d < tolerance]) / len(delays)\n```\n\n### How to Avoid Overfitting Regimes\n\n1. **Train/test split in time**: never evaluate on in-sample regime labels\n2. **Walk-forward validation**: fit on [0, T], evaluate on [T, T+delta], roll forward\n3. **Regime stability constraint**: penalize models that produce regimes shorter than a minimum duration (e.g., 30 minutes for 5-min data)\n4. **Out-of-sample economic value**: the ultimate test. If regime-conditioned trading doesn't improve out-of-sample Sharpe, the regimes are overfit.\n5. **Parsimony**: prefer 2-3 states over 4+. More states always fit better in-sample.\n\n### Recommendation\n\n**Evaluate using a hierarchy of metrics:**\n1. **Gate**: regime separation (KS test p < 0.01 for all state pairs) — if regimes aren't statistically different, they're useless\n2. **Stability**: mean duration > 30 bars (2.5 hours) for 5-min data\n3. **Economic**: out-of-sample Sharpe improvement > 0.3 over baseline\n4. **Speed**: detection delay < 15 bars for major regime changes\n\n---\n\n## 14. O(K^2).\"\"\"\n        # Predict: alpha_pred = A^T @ alpha\n        alpha_pred = self.A.T @ self.alpha\n        \n        # Update: alpha = alpha_pred * emission / normalizer\n        emission = self.emission_prob(obs)\n        alpha_unnorm = alpha_pred * emission\n        normalizer = np.sum(alpha_unnorm)\n        \n        if normalizer > 0:\n            self.alpha = alpha_unnorm / normalizer\n        \n        return self.alpha.copy()\n    \n    def current_regime(self):\n        return np.argmax(self.alpha)\n    \n    def regime_probability(self):\n        return np.max(self.alpha)\n```\n\n### Recommendation Matrix\n\n| Use case | Recommendation |\n|----------|---------------|\n| Quick prototyping | hmmlearn |\n| Production 5-min system | Custom (based on template above) |\n| Research/experimentation | hmmlearn + custom extensions |\n| GPU-accelerated, many features | pomegranate v1.0 |\n| Student-t emissions | Custom |\n| Online updating | Custom |\n\n**For our use case**: Start with hmmlearn for prototyping and validation.
