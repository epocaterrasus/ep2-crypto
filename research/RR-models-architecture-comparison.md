Ensemble Methods Beyond Simple Averaging\n\nThis is where the real gains are for production systems.\n\n### Method Comparison\n\n#### a) Stacking (Meta-Learner)\n\nThe base models (LightGBM, CatBoost, GRU, TCN) each produce predictions. Random splits leak future information.\n\n#### b) Bayesian Model Averaging (BMA)\n\nWeights models by their posterior probability given the data. As a model performs better on recent data, its weight increases.\n\n**Evidence:**\n- Kolm & Ritter (2020): BMA outperformed equal-weight and best-single-model on equity prediction.\n- Theoretical advantage for regime changes in crypto: when a regime shifts, BMA naturally down-weights models that no longer perform.\n- **Practical limitation:** Computing exact posteriors is intractable; approximations (BIC-based) lose the theoretical elegance.\n\n#### c) Dynamic Ensemble Selection (DES)\n\nFor each new prediction, select which subset of models to use based on the characteristics of the input.\n\n**Evidence:**\n- Cruz et al. (2018), survey of DES: DES outperformed static ensembles on non-stationary data.\n- Most relevant variant for crypto: **META-DES** (Meta-learning for Dynamic Ensemble Selection). Uses a meta-classifier trained to predict which base classifiers will be correct for a given input.\n- **KNORA** (K-Nearest Oracles): For each test sample, find similar samples in a validation set and use models that performed best on those neighbors.\n- No crypto-specific DES publications, but the non-stationarity argument is strong.\n\n#### d) My Ranking for 5-Min Crypto\n\n| Method | Expected Gain | Complexity | Recommendation |\n|--------|--------------|------------|----------------|\n| Stacking | +2-5% over best single | Low | **Yes, implement first** |\n| Dynamic ensemble selection | +1-3% over static stacking | Medium | **Yes, implement second** |\n| Bayesian model averaging | +0.5-2% over equal weight | Medium | Worth trying |\n| Simple averaging | Baseline | Trivial | Starting point only |\n\n**Recommendation:** Implement stacking with time-series CV as the primary ensemble method. Online Learning Architectures\n\n### The Core Question: Online vs. Batch Retraining\n\n| Approach | How It Works | Latency of Adaptation |\n|----------|-------------|----------------------|\n| Batch retrain (daily) | Full retrain on sliding window | 24 hours |\n| Batch retrain (hourly) | Full retrain on sliding window | 1 hour |\n| Online (incremental) | Update on each new sample | 5 minutes |\n| Warm-start batch | Retrain from previous weights | Varies |\n\n### River Library (Online ML)\n\nRiver provides online versions of: logistic regression, Hoeffding trees, online bagging/boosting, k-NN, naive Bayes, and more.\n\n**For crypto:**\n- Online logistic regression with feature hashing can adapt to regime changes within minutes.\n- Hoeffding Adaptive Tree (HAT) detects concept drift and rebuilds subtrees.\n- **Limitation:** Online models are typically less accurate than batch models at any point in time. Risk: catastrophic forgetting.\n- **Elastic Weight Consolidation (EWC):** Penalizes changes to weights that were important for previous data. Reinforcement Learning for Trading\n\n### The Approaches\n\n| Algorithm | Type | Key Property |\n|-----------|------|-------------|\n| DQN | Value-based | Discrete actions (buy/sell/hold) |\n| PPO | Policy gradient | Continuous actions, stable training |\n| SAC | Actor-critic | Maximum entropy, good exploration |\n| A2C/A3C | Actor-critic | Parallel training |\n| TD3 | Actor-critic | Twin critics reduce overestimation |\n\n### RL vs Supervised Prediction + Rule-Based Execution\n\n**This is the core question.** Published evidence:\n\n- Sattarov et al. Here is why:**\n\n1. **Sample efficiency:** RL needs 10-100x more data than supervised learning to converge. At 5-min bars, that means years of data, during which crypto regimes change completely.\n2. **Reward sparsity:** Most 5-min bars have near-zero return after fees. RL struggles with sparse rewards.\n3. **Non-stationarity:** RL assumes a somewhat stationary environment. Crypto at 5-min is violently non-stationary.\n4. **Partial observability:** RL assumes the state captures relevant information. We cannot observe all market participants' intentions.\n5. **Sim-to-real gap:** RL trained on historical data assumes perfect execution at mid-price. Real execution has slippage, partial fills, and market impact. Model Calibration\n\n### Why This Matters\n\nFor a confidence-gated system, the predicted probability must reflect the true probability. If the model says \"75% chance of up move,\" it should be correct 75% of the time. Most ML models are poorly calibrated out of the box.\n\n### Calibration Methods\n\n#### a) Platt Scaling\n\nFit a logistic regression on the model's raw output scores against true labels. Transforms uncalibrated scores into calibrated probabilities.\n\n```\nP(y=1 | f(x)) = 1 / (1 + exp(A * f(x) + B))\n```\n\n- **Pros:** Simple, fast, works well for neural networks.\n- **Cons:** Assumes a sigmoid relationship. May underfit if the calibration curve is non-monotonic.\n- **For LightGBM:** Works well because LightGBM outputs are approximately normally distributed.\n\n#### b) Isotonic Regression\n\nNon-parametric calibration. Fits a step-wise non-decreasing function to map raw scores to calibrated probabilities.\n\n- **Pros:** No functional form assumption. Can handle any calibration curve shape.\n- **Cons:** Needs more calibration data (1000+ samples minimum). Can overfit with small calibration sets.\n- **For LightGBM:** Often better than Platt scaling because tree model output distributions are irregular.\n\n#### c) Temperature Scaling\n\nDivide neural network logits by a learned temperature T before softmax.\n\n```\nP(y=k | x) = softmax(z_k / T)\n```\n\n- **Pros:** Single parameter to tune. Preserves accuracy (does not change argmax).\n- **Cons:** Only works for neural networks with logit outputs.\n- **Best for:** GRU/TCN outputs in our ensemble.\n\n#### d) Venn-ABERS Prediction\n\nA conformal prediction framework that provides guaranteed calibration under exchangeability assumption.\n\n- **Pros:** Theoretical guarantees. Outputs a probability interval rather than a point.\n- **Cons:** More computationally expensive. Exchangeability assumption is questionable for time series.\n\n### Evidence for Financial Data\n\n- Niculescu-Mizil & Caruana (2005): Platt scaling works well for SVMs and neural nets; isotonic regression works better for boosted trees and random forests.\n- Guo et al. (2017): Modern neural networks are more miscalibrated than older ones. Temperature scaling is the most effective single-parameter fix.\n- **For LightGBM specifically:** The `is_unbalanced` or `scale_pos_weight` parameters affect calibration. Even with these, post-hoc calibration improves Brier score by 5-15%.\n\n### Practical Calibration Pipeline\n\n| Step | Method | Details |\n|------|--------|---------|\n| 1 | Hold out calibration set | Last 20% of time-ordered validation data |\n| 2 | Calibrate LightGBM/CatBoost | Isotonic regression (needs 2000+ samples) |\n| 3 | Calibrate GRU/TCN | Temperature scaling on logits |\n| 4 | Calibrate meta-learner | Platt scaling (already logistic regression, so naturally calibrated) |\n| 5 | Monitor calibration drift | Track reliability diagram weekly, recalibrate if ECE > 0.05 |\n\n**Recommendation:** Isotonic regression for tree models, temperature scaling for neural networks. Quantile Regression\n\n### Why Quantile Regression for Crypto\n\nInstead of predicting \"BTC will go up,\" predict \"the 10th percentile return is -0.3%, median return is +0.1%, 90th percentile return is +0.8%.\" This gives you:\n\n- **Risk quantification:** Know the downside before entering.\n- **Confidence width:** Narrow quantile spread = high confidence. Wide spread = stay out.\n- **Asymmetric risk:** Maybe the median is positive but the 10th percentile is catastrophic. Skip that trade.\n\n### Implementation Options\n\n#### a) Quantile Regression in LightGBM\n\nLightGBM natively supports quantile regression via `objective='quantile'` with `alpha` parameter.\n\n```python\n# Train separate models for each quantile\nmodel_10 = lgb.train(params={'objective': 'quantile', 'alpha': 0.1}, ...)\nmodel_50 = lgb.train(params={'objective': 'quantile', 'alpha': 0.5}, ...)\nmodel_90 = lgb.train(params={'objective': 'quantile', 'alpha': 0.9}, ...)\n```\n\n- **Pros:** Fast, uses same features as classification model.\n- **Cons:** Need separate model per quantile. Loss function is pinball loss for each quantile.\n\n- **Pros:** Single model, parameters shared across quantiles, no crossing if architecture enforces monotonicity.\n- **Cons:** Harder to train than classification.\n\n#### c) Conformalized Quantile Regression\n\nCombines quantile regression with conformal prediction to provide coverage guarantees.\n\n- Romano et al. (2019): CQR guarantees that the prediction interval contains the true value with probability >= 1-alpha, even if the quantile regression model is misspecified.\n- **For crypto:** The non-stationarity violates the exchangeability assumption, but in practice, CQR with a sliding calibration window still provides useful intervals.\n\n### For Our Confidence-Gated System\n\n**The quantile spread is a natural confidence measure:**\n\n```\nconfidence = 1 / (quantile_90 - quantile_10)\n```\n\nWhen the model is uncertain, the spread widens, confidence drops, and the system abstains. This is more principled than using classification probability alone because it directly measures uncertainty about the magnitude of the move.\n\n**Recommendation:** Implement LightGBM quantile regression (10th, 25th, 50th, 75th, 90th percentiles) alongside the classification model.
