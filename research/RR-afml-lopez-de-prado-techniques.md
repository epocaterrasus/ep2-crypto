During Asian session, 5 minutes might have 200 trades. During a US CPI release, 5 minutes might have 50,000 trades. The same bar duration contains vastly different information.\n\nDollar bars normalize for this. They also exhibit returns distributions closer to Gaussian (better for ML assumptions) compared to time bars.\n\n**The 2025 Springer study (Financial Innovation)** specifically tested these on BTC and ETH with tick-level data (2018-2023) and confirmed:\n- Information-driven bars + triple barrier labeling + deep learning outperformed time-bar approaches\n- Volume bars and dollar bars showed the most consistent improvement\n\n**Implementation:**\n\n```python\ndef dollar_bars(trades: pd.DataFrame, dollar_threshold: float) -> pd.DataFrame:\n    \"\"\"\n    trades has columns: timestamp, price, volume\n    \"\"\"\n    bars = []\n    cum_dollars = 0.0\n    open_price = trades.iloc[0][\"price\"]\n    high, low = open_price, open_price\n    cum_volume = 0.0\n    start_time = trades.iloc[0][\"timestamp\"]\n    \n    for _, trade in trades.iterrows():\n        dollars = trade[\"price\"] * trade[\"volume\"]\n        cum_dollars += dollars\n        cum_volume += trade[\"volume\"]\n        high = max(high, trade[\"price\"])\n        low = min(low, trade[\"price\"])\n        \n        if cum_dollars >= dollar_threshold:\n            bars.append({\n                \"timestamp\": trade[\"timestamp\"],\n                \"open\": open_price,\n                \"high\": high,\n                \"low\": low,\n                \"close\": trade[\"price\"],\n                \"volume\": cum_volume,\n                \"dollar_volume\": cum_dollars,\n            })\n            cum_dollars = 0.0\n            cum_volume = 0.0\n            open_price = trade[\"price\"]\n            high, low = open_price, open_price\n    \n    return pd.DataFrame(bars)\n```\n\n**Worth it for your project?** MIXED -- this is a fundamental architecture decision. Your entire pipeline is built around 5-min candles. Switching to dollar bars means:\n- Pros: Better statistical properties, more bars during active periods, fewer during dead periods\n- Cons: Variable bar frequency makes time-based features (cyclical encoding, session indicators) harder; your GRU sequence length becomes variable; all your existing feature engineering assumes fixed intervals\n\n**Recommendation:** Keep 5-min bars as your primary framework. Add dollar-bar features as supplementary signals (e.g., \"how many dollar bars formed in the last 5 minutes\" as a feature, which is effectively a trade intensity measure). You're already capturing similar information through trade flow features in Phase 4.\n\n**Effort:** 3-5 days to fully switch to dollar bars (major refactor). 1 day to add dollar-bar-derived features alongside existing 5-min bars.\n\n---\n\n## 7. Entropy Features\n\n**Types Lopez de Prado discusses:**\n\n1. **Shannon Entropy (plug-in estimator):**\n   ```\n   H = -sum(p_i * log2(p_i))\n   ```\n   where p_i is the probability of symbol i in a discretized return sequence.\n\n2. **Lempel-Ziv Complexity:** Measures how compressible a sequence is. A random sequence has high LZ complexity; a patterned sequence has low LZ complexity.\n\n3. **Kontoyiannis Entropy:** Based on the longest match length -- how far back you need to look to find a repeated pattern.\n\n**Implementation:**\n\n```python\nimport numpy as np\nfrom collections import Counter\n\ndef shannon_entropy(returns: pd.Series, bins: int = 5) -> float:\n    \"\"\"Shannon entropy of discretized returns.\"\"\"\n    # Quantile-based discretization\n    discretized = pd.qcut(returns, q=bins, labels=False, duplicates=\"drop\")\n    counts = Counter(discretized)\n    total = sum(counts.values())\n    probs = [c / total for c in counts.values()]\n    return -sum(p * np.log2(p) for p in probs if p > 0)\n\ndef lempel_ziv_complexity(binary_seq: str) -> int:\n    \"\"\"Lempel-Ziv complexity of a binary string.\"\"\"\n    n = len(binary_seq)\n    if n == 0:\n        return 0\n    complexity = 1\n    l = 1  # current prefix length\n    k = 1  # current component length\n    k_max = 1\n    \n    while l + k <= n:\n        if binary_seq[l:l+k] in binary_seq[:l+k-1]:  # simplified; use suffix tree for O(n)\n            k += 1\n        else:\n            complexity += 1\n            k_max = max(k_max, k)\n            l += k\n            k = 1\n    \n    return complexity\n\n# Rolling entropy as a feature\ndef rolling_entropy(returns: pd.Series, window: int = 60, bins: int = 5) -> pd.Series:\n    return returns.rolling(window).apply(\n        lambda x: shannon_entropy(x, bins), raw=False\n    )\n```\n\n**Predictive power:** Entropy features capture market regime information from a different angle than volatility:\n- Low entropy = predictable patterns = momentum strategies work\n- High entropy = random walk = mean reversion or no trade\n- Entropy spikes can precede regime changes\n\n**Worth it for your project?** LOW-MEDIUM priority. Entropy features are interesting but computationally expensive to compute in rolling windows and the signal is weak at 5-min resolution. Your HMM regime detector already captures much of the same information (regime = proxy for market predictability). If you implement it, use Shannon entropy on 60-bar rolling windows as a single feature -- don't over-invest here.\n\n**Effort:** 1 day.\n\n---\n\n## 8. Bet Sizing\n\n**Lopez de Prado's formula:** Convert model probability p into a bet size using the normal CDF:\n\n```\nz = (p - 0.5) / sqrt(p * (1 - p))\nbet_size = 2 * Phi(z) - 1\n```\n\nwhere Phi is the standard normal CDF. This maps:\n- p = 0.5 (no edge) -> bet_size = 0\n- p = 0.6 -> bet_size ~ 0.38\n- p = 0.7 -> bet_size ~ 0.63\n- p = 0.8 -> bet_size ~ 0.82\n- p = 0.95 -> bet_size ~ 0.97\n\n**Implementation:**\n\n```python\nfrom scipy.stats import norm\n\ndef bet_size_from_probability(prob: float, max_position: float = 1.0) -> float:\n    \"\"\"\n    Lopez de Prado bet sizing: probability -> position size.\n    prob: model's predicted probability of correct direction (0.5 to 1.0)\n    \"\"\"\n    if prob <= 0.5:\n        return 0.0\n    z = (prob - 0.5) / np.sqrt(prob * (1.0 - prob))\n    size = 2.0 * norm.cdf(z) - 1.0\n    return size * max_position\n\n# With meta-labeling: use secondary model's probability directly\ndirection = primary_model.predict(X)  # +1 or -1\nconfidence = meta_model.predict_proba(X_meta)[:, 1]  # P(primary is correct)\nposition = direction * bet_size_from_probability(confidence) * max_leverage\n```\n\n**Integration with your architecture:** Your plan already has confidence gating at 0.6. This replaces the binary gate with a continuous sizing function. When XGBoost says 0.61 confidence, you take a small position. When it says 0.85, you take a large one.\n\n**Worth it for your project?** YES -- medium priority. Direct upgrade to your ensemble output. Instead of binary \"trade / don't trade,\" you get continuous position sizing. Pairs naturally with meta-labeling.\n\n**Effort:** 0.5 days. Trivial to implement.\n\n---\n\n## 9. Combinatorial Purged Cross-Validation (CPCV)\n\n**What it does:** Standard time-series CV generates one backtest path. CPCV generates many paths by combinatorially selecting test groups from N non-overlapping time segments.\n\n**Algorithm:**\n1. Divide time series into N sequential groups\n2. For each combination of k groups as test sets (C(N, k) combinations):\n   - Use remaining N-k groups for training\n   - **Purge:** Remove training samples whose labels overlap with test period boundaries\n   - **Embargo:** Further remove a buffer of samples after each purge boundary\n3. This generates C(N,k) / C(N-k, k) unique backtest paths\n4. Evaluate strategy on each path to get a distribution of performance metrics\n\n**Key parameters:**\n- N = number of groups (typically 6-12)\n- k = number of test groups per split (typically 2)\n- purge_size = label horizon (for triple barrier: max holding period)\n- embargo_pct = additional buffer (typically 1-2% of dataset)\n\n**Implementation:** Available in `skfolio.model_selection.CombinatorialPurgedCV` (well-maintained, scikit-learn compatible).\n\n```python\nfrom skfolio.model_selection import CombinatorialPurgedCV\n\ncpcv = CombinatorialPurgedCV(\n    n_folds=10,\n    n_test_folds=2,\n    purge_size=12,    # 12 bars = max holding period\n    embargo_size=2,   # 2 extra bars buffer\n)\n\nfor train_idx, test_idx in cpcv.split(X):\n    model.fit(X[train_idx], y[train_idx])\n    score = model.score(X[test_idx], y[test_idx])\n```\n\n**vs. standard time-series CV:** CPCV is strictly better -- it generates a distribution of backtest paths instead of a single path, which lets you estimate the probability of overfitting. Standard TimeSeriesSplit with a single walk-forward window is a special case.\n\n**Worth it for your project?** LOW priority for now. Your walk-forward validation (Phase 9) with sliding windows is the right approach for live trading validation. CPCV is more valuable for strategy selection (choosing between multiple strategies) than for training a single model. Add it later when you're comparing strategies.\n\n**Effort:** 1-2 days with skfolio. More effort to integrate with your walk-forward pipeline.\n\n---\n\n## 10. Feature Importance Methods\n\n**MDA (Mean Decrease Accuracy):**\n- Permute feature j, measure drop in OOS accuracy\n- Pro: Works with any model, uses OOS data\n- Con: Suffers from substitution effect (correlated features split importance)\n- Formula: `MDA(j) = score_original - score_permuted_j`\n\n**MDI (Mean Decrease Impurity):**\n- From tree-based models only: how much each feature reduces impurity (Gini/entropy) across all splits\n- Pro: Fast, free from XGBoost via `get_score(importance_type='gain')`\n- Con: Biased toward high-cardinality features; uses in-sample data\n\n**SFI (Single Feature Importance):**\n- Train separate model on each feature alone, measure OOS performance\n- Pro: No substitution effect\n- Con: Misses interaction effects; very expensive (N models to train)\n\n**Which is most reliable for financial features?** Lopez de Prado recommends using **MDA with clustering** (Clustered MDA). Group correlated features into clusters first, then compute MDA on cluster representatives. This eliminates the substitution effect while preserving interaction detection.\n\nFor your use case, **MDI from XGBoost** (`model.get_score(importance_type='gain')`) is the practical choice for ongoing monitoring. Use **MDA** during offline analysis to validate that MDI rankings are reasonable. Skip SFI unless you suspect a specific feature is only valuable in isolation.\n\n**Worth it for your project?** Your plan already includes feature importance tracking in Phase 7. MDI is built into XGBoost. Add MDA as a periodic validation check.\n\n**Effort:** MDI = 0 (already available). MDA = 0.5 days. SFI = 1-2 days (probably skip).\n\n---\n\n## 11. Backtest Overfitting (PBO)\n\n**The problem:** If you try N strategy configurations and pick the best backtest, the probability that this \"best\" strategy is actually overfitting to historical noise grows with N. After only 7 trials, you're expected to find a 2-year backtest with Sharpe > 1.0 even when the true expected Sharpe is 0.\n\n**PBO computation (CSCV method):**\n1. Divide backtest returns into S equal time segments\n2. For each combinatorial split into in-sample (IS) and out-of-sample (OOS) halves:\n   - For each of N strategy configs, compute IS performance\n   - Select the strategy with best IS performance\n   - Check its OOS performance rank\n3. PBO = fraction of combinations where the IS-optimal strategy has below-median OOS performance\n\n**Minimum Backtest Length (MinBTL):**\n```\nMinBTL = (N * E[max_SR_under_null]^2) / (SR_target^2)\n```\nWhere N = number of trials, and E[max_SR_under_null] grows with sqrt(2 * ln(N)).\n\n**Library:** `pypbo` (pip install pypbo) -- implements the full CSCV framework.\n\n**Worth it for your project?** LOW priority. PBO is designed for strategy selection over many candidates. You're building one strategy with Optuna hyperparameter tuning. For your use case, the walk-forward validation with realistic transaction costs is a more direct test of overfitting. However, keep the MinBTL formula in mind: if your Optuna study tries 100 configurations, you need a much longer backtest to be confident. With 5-min bars, 30 days gives you 8,640 samples -- likely sufficient for <100 trials.\n\n**Effort:** 1 day with pypbo. Low ROI for your specific setup.\n\n---\n\n## 12. Structural Breaks (SADF / GSADF)\n\n**SADF (Sup Augmented Dickey-Fuller):** Runs ADF tests with a fixed start point and expanding end point. The supremum of the test statistics detects explosive behavior (bubbles).\n\n**GSADF:** Extends SADF by also varying the start point. This detects multiple bubble episodes and date-stamps them.\n\n**Formula:**\n```\nADF regression: delta(y_t) = alpha + beta*y_{t-1} + sum(gamma_i * delta(y_{t-i})) + epsilon_t\n\nSADF = sup{ADF(r2)} for r2 in [r0, 1]  (r0 = minimum window fraction)\nGSADF = sup{ADF(r1, r2)} for r1 in [0, r2-r0], r2 in [r0, 1]\n```\n\n**Crypto application:** The 2025 MDPI study confirmed GSADF detects structural changes in BTC, ETH, SOL that align with macro events (monetary policy, ETF approvals). Works well on daily data.\n\n**Real-time feasibility:** GSADF requires substantial lookback. On 5-min bars, you need thousands of observations for statistical power. The test itself is O(n^3) in the naive implementation. Not suitable for real-time 5-min prediction.\n\n**Worth it for your project?** LOW priority for real-time prediction. Potentially useful as an offline feature: compute GSADF on daily data, use the \"are we in a bubble\" state as a daily-resolution feature fed to your model. But your HMM regime detector already captures this concept more flexibly.\n\n**Effort:** 2-3 days. Libraries exist but integration with real-time pipeline is non-trivial.\n\n---\n\n## 13.
